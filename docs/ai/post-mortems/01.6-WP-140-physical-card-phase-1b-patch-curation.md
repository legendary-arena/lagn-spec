## 3. Mid-Execution Amendments

### 3.1 `applyPatch` underscore-prefix exclusion (DURING idempotency spot-check)

**Trigger:** Initial idempotency spot-check failed — removing `_skipPair` from `3dtc.patch.json` produced a non-byte-identical `data/cards/3dtc.json` (the diff showed the `_skipPair` field itself appearing in the output JSON).

**Root cause:** `applyPatch`'s field-copy loop excluded only `slug` / `_slug` / `cards`. Every other patch field — including `_skipPair` — was being copied to the target hero in `result.heroes[i]` and ending up in the output JSON. `buildPhysicalCards` reads `_skipPair` for validation but doesn't overwrite the `target._skipPair` value, so the leaked field persisted.

**Fix:** Extended the exclusion in `applyPatch` to skip any underscore-prefixed key, with a justification comment explaining that the convention covers `_op` / `_slug` / `_skipPair` / `_abilityTokenRewrite` / any future patch annotation grammar uniformly. Real card-data fields never use the underscore-prefix convention so the rule is safe.

**Re-verification:** Idempotency spot-check passed on the second attempt. Strict-mode convert exits 0. All registry + engine tests pass.

This amendment was the **right kind** of mid-execution discovery — caught by the WP's own idempotency invariant before commit. The fix is a one-line refinement to a 5-year-old patch-merge loop that nobody had reason to question until now.

### 3.2 Worklist-line parser bug (DURING per-set generator development)

**Trigger:** First run of the patch generator script (temporary, not committed — used for bulk `_skipPair` / `physicalCards` block authoring) produced a `Cluster coverage failure for set "chmp" hero "gwenpool"` error during the post-generation convert run.

**Root cause:** The generator's regex parsed cluster member names from the worklist line's parenthesized list using a naive `split(', ')`. `chmp/gwenpool` has a card named `Come On, Nobody Reads Card Names` (with embedded commas); the naive split incorrectly produced 3 names from a 2-cluster.

**Fix:** Switched the generator's cluster-member derivation from worklist-line parsing to direct lookup against `hero-card-counts.json`. Names with count matching the cluster's count are unambiguously the cluster members. Cross-checked against the worklist's `has N cards` size assertion to catch any discrepancy.

**Re-verification:** Re-ran generator from clean state (after `git checkout HEAD -- scripts/convert-cards/inputs/patches/`), then convert pipeline exited 0 under strict mode.

This bug was internal to the temporary generator (not committed); the post-mortem captures it for the Phase 2 / WP-141 author who may build similar tooling.

### 3.3 Convert script `// why:` softening (PRE Commit A)

## 5. Interesting Cluster Cases

### 5.1 Cloak/Dagger (cvwr) — 4-cluster + 2-cluster, all splits

`cvwr/cloak-dagger` is the most-split hero in the registry: 3 split pairs across 7 cards (Above/Below, Flee/Fight, Darkness/Light, plus Penumbra solo). The 4-cluster at count=5 contains all four of (Above, Below, Flee, Fight) sharing the same `cardCounts` value. Without the `divided:` field, this cluster would be ambiguous between (a) two splits (matching the actual physical printing) and (b) four independent solos. The `divided:` field unambiguously paired Above+Below and Flee+Fight, giving the 2 splits + correct deck size.

### 5.2 Storm/Black Panther (cvwr) — Falcon-WS-pattern hero

Same 7-card pattern as `bkwd/falcon-winter-soldier`: 3 splits + 1 solo rare. Subtitle alternation Storm/Black Panther across all 6 split sides; King & Queen of Wakanda is the unsplit rare. Confirms the pattern generalizes — Falcon/WS isn't the only Falcon-WS-pattern hero in the registry.

### 5.3 Wanda/Vision (msis) — 9-card hero with 3 splits

`msis/wanda-vision` has 9 cards in `cards[]`: 3 split pairs + 2 unsplit commons (We Have to Destroy It, Witchcraft) + 1 unsplit rare (Odd Couple). Three clusters across counts 3 and 2:
- count=3 cluster: 5 cards (We Have to Destroy It, Witchcraft, Hold On, Let Go, plus a third unsplit at count=3 in cardCounts) — wait, actually rechecking cardCounts: `{We Have to Destroy It:3, Witchcraft:3, Hold On:3, Let Go:3, Magic:2, Science:2, Rage:2, Grief:2, Odd Couple:1}`. The count=3 cluster has 4 cards: WHTDI, Witchcraft, Hold On, Let Go. Hold On+Let Go is one split pair; WHTDI and Witchcraft are unsplit solos.
- count=2 cluster: 4 cards Magic, Science, Rage, Grief. Two split pairs (Magic/Science and Rage/Grief).

Mixed-resolution 4-cluster: 1 split (Hold On+Let Go) + 2 solos (WHTDI, Witchcraft) → `physicalCards: [{count:3, sides:[hold-on,let-go]}, {count:3, sides:[we-have-to-destroy-it]}, {count:3, sides:[witchcraft]}]` for the count=3 cluster. Auto-fill handles the rare.

### 5.4 Rocket/Groot (mgtg) — Subtitle drift

Three subtitle pairs in `mgtg/rocket-groot` are all `Rocket / Baby Groot`. The third pair has a curious back-side subtitle of `Groot` (no "Baby"). Could be intentional thematic drift or a typo in the npm source. Authored the patch using `divided:` pairing (which matches the structural intent regardless of subtitle text) and noted the discrepancy in the per-set patch authoring summary. Phase 2 / WP-141 should verify against the actual card art if image-display behavior depends on the subtitle text.

### 5.5 wonder-man (antm) — 3-cluster of all-solos in a non-`divided:` set

`antm/wonder-man` has cardCounts `{One-Hit Wonder:5, Ionic Energy:5, Absorb Ambient Power:5, Steady Hand:3, Untouchable:1}`. Three cards at count=5 form a 3-cluster. `antm.js` has zero `divided:` entries → the 3-cluster is all-solo coincidence. Resolution: `physicalCards: [{count:5, sides:[absorb-ambient-power]}, {count:5, sides:[ionic-energy]}, {count:5, sides:[one-hit-wonder]}]`. Auto-fill handles the uncommon and rare. This is the canonical pattern for size-3+ all-solo clusters where `_skipPair` cannot cover atomically.

### 5.6 legion (xmen) — 7-card hero with 3 splits, no subtitles

`xmen/legion` has 3 split pairs marked with `divided: 1` / `divided: 2` but ZERO subtitle entries on either face. The structural `divided:` signal correctly identified all five physicalCard groupings (3 splits + 1 solo at count=3 ... actually wait, let me recount: cardCounts says `{Bend Steel:5, Bend Light:5, Split Personality:5, Split Eardrums:5, Channel Time:3, Channel Fire:3, Maelstrom of Clashing Powers:1}` — 4 cards at count 5 (2 splits) + 2 at count 3 (1 split) + 1 rare. ## 7. Forward Signal for Phase 2 / WP-141

Phase 2 / WP-141 (engine + viewer consumer migration) will need to handle:

### 7.1 Heroes with auto-fill physicalCards entries

Many cvwr / mgtg / msis / xmen heroes have a mix of declared splits + auto-filled solos. The auto-fill IDs continue from `p${declaredCount + 1}` so declared IDs (`p1`, `p2`, ... for splits) are stable. Phase 2 consumers reading `physicalCards[].id` should not assume a particular ID-to-side mapping; they should resolve via the runtime `sideToPhysicalCard` index (D-13806).

### 7.2 Mixed-cluster heroes (split + solo at the same count)

cvwr's `cap-secret-avenger`, `goliath`, `hulkling`, `falcon`, `hercules`, `tigra` and similar heroes have 3-clusters where 2 cards are a split pair and the third is a solo. Their `physicalCards` arrays have both 2-side entries (the split) and 1-side entries (the solo at the same count). Deck-reservoir math (Phase 2 §B) reads `physicalCards[].count` directly — the split contributes one entry of count=N, the solo contributes a separate entry of count=N. Total deck contribution at that count = N (split) + N (solo) = 2N, vs the cardCounts sum 2N + N = 3N. Phase 2's `buildHeroDeck` migration must read physicalCards.count, not summed cardCounts.

### 7.3 wwhk's many 1-side declarations

`wwhk` is in the changed-data-files list despite zero `divided:` entries because every wwhk worklist hero has a 3+-cluster forced into `physicalCards[]` 1-side declarations. wwhk's transform-mechanic cards (a separate game mechanic from physical-card splits) all carry `transformed: true` in the npm source. Phase 2's viewer consumer migration may want to surface the transform-vs-split distinction; for now, both cases collapse to 1-side `physicalCards` entries (D-13803 uniform model).

### 7.4 Non-hero subtitle entries

dkcy, fear, mdns, rvlt, ssw1, ssw2 have subtitle entries on villain groups / masterminds / schemes / henchmen. None are split-side cards in the registry sense. Phase 2 consumers only iterate `result.heroes[i].physicalCards[]`, so non-hero subtitles don't surface.

### 7.5 Patch-driven authoring as the only valid source

Phase 2 inherits D-13805 (patches are the sole source of split declarations) extended by D-13901 (patches are the sole source of false-positive declarations). Engine code never auto-detects splits from cardCounts heuristics; consumers always read the registry's curated `physicalCards[]` array.

## 9. Conventions Established by This WP

- **`_skipPair` annotation grammar (D-13901)** is the false-positive escape hatch. Future paired-equal audit-warning surfaces (e.g., if a future schema adds a new derived-validation surface on registry data) follow the same pattern: declarative patch annotation, mutual exclusion with `physicalCards[].sides`, cluster coverage rule, idempotency invariant, deterministic per-hero execution order.
- **Underscore-prefix convention for patch-only fields.** `_op`, `_slug`, `_skipPair`, `_abilityTokenRewrite` are all build-time annotations; `applyPatch` excludes them from output JSON uniformly. Future patch annotation grammar should follow the convention (e.g., `_skipTriple` if D-13802's 2-side ceiling is ever raised).
- **Worklist freeze artifact.** Capturing the audit-warning surface to a session-context file before any patch authoring lets the executor verify byte-identity over the working session and provides a forensic record for retrospective review. Future high-touch curation WPs should follow the pattern.
