## 2. New Patterns Discovered

### 2.1 Rarity-driven flat-array walk as a sibling-snapshot variant

The WP-014B / WP-018 / WP-111 sibling-snapshot pattern resolves registry data once at `Game.setup()` into a flat `Record<CardExtId, X>` map. `buildHeroDeck` extends that pattern with a **rarity-driven flat-array walk**: each hero card (`heroes[i].cards[j]`) contributes a fixed integer number of copies into a flat `CardExtId[]` array, keyed by the per-card `rarityLabel`. The walk is ordering-stable — heroes per `config.heroDeckIds` order; cards per registry `cards[]` order; copies appended in rarity-map iteration order (Common 1 → Common 2 → Uncommon → Rare). Distinct from the four sibling-snapshot maps, but uses the same registry-walk-once-at-setup primitive.

Pattern is reusable for any future "shared pool of card instances built from a registry-side rule" (e.g., a future Master Strike pile, a Scheme Twist deck-of-twists, or a treasure deck per scenario).

### 2.2 Option A loud-fail throw at `Game.setup()` as the canonical surface for unknown enumerated values

Per `.claude/skills/legendary-game-engine/SKILL.md §Throwing Convention`, only `Game.setup()` may throw. WP-135 uses this surface for **D-13501 Option A loud-fail** — when `buildHeroDeckCards` encounters a hero card with a `rarityLabel` outside the four-label set, it throws a full-sentence `Error` naming the offending hero ext_id, the unrecognized label, and the supported four-label set. This pattern is the canonical answer to "what if registry data drifts out of the locked enum set?" — surface the drift the moment it is observed, at the only legitimate throw site, with a structured error message that names the offending entity, the offending value, and the locked permitted set.

Reusable for any future setup-time builder that enumerates registry data against a closed lock (e.g., `boardKeyword` against `BOARD_KEYWORDS`, `scenario tier` against `AI_POLICY_TIERS`).

### 2.3 Explicit-replace `G.messages` line as a precedent for future log-format migrations

The pre-WP-135 `recruitHero` `G.messages` push was `Player {playerId} recruited "{cardId}" from HQ slot {hqIndex}.` (WP-016 format). WP-135 **replaced** that push (not augmented) with the new byte-locked WP-135 format `Player {playerId} recruited {heroExtId}; HQ slot {hqIndex} refilled from heroDeck (heroDeck.length: {N})` (or the empty-deck variant). The replace-not-augment lock matters because:

- Two pushes per recruit would double-count message events in tests, change `computeStateHash` inputs in a non-obvious way, and introduce an "old format coexists with new format" liability.
- Tests that asserted the old format need a value-only update under the 01.5 wiring allowance.
- Replay-determinism tests catch the change immediately via the regression-guard `PRE_WP080_HASH` literal cascade (which fired exactly as expected).

This is the locked precedent for any future `G.messages` line-format migration: **replace, don't augment, and let the regression cascade catch the input-shape change.**

## 3. Anti-Patterns Discovered (and Resisted)

### 3.1 Silent-skip on unknown rarityLabel

Tempting because it would produce a "smaller-than-expected" hero deck rather than a hard failure. Rejected at D-13501 Option A. Silent-skip would produce undersized decks at runtime with no diagnostic, would fail the §A acceptance criterion that the deck contains exactly `(rarity-map-sum) × N` cards, and would let data drift propagate invisibly.

### 3.2 Augmenting (not replacing) the WP-016 `G.messages` push

Tempting because it would preserve the old test assertions verbatim. Rejected at PS-3. Two pushes per successful recruit would double-count log events, violate the determinism envelope, and require double-bookkeeping in every consumer (replay diff, projection, test fixture).

### 3.3 Extending the rarity map mid-execution to cover AMWP-class sets

Tempting because the cross-set audit revealed 76/307 heroes use `'Common 3'` / `'Uncommon 2'` outside the locked set. Rejected — extending the map mid-execution requires:

- Sourcing canonical Marvel Legendary tabletop copy counts for each new label with rule citations (none yet sourced).
- Test fixtures across the new labels.
- Updating D-13501 with the extension and the per-set hero-card count (likely 18 for AMWP-class).

The pre-flight authorized the deferred follow-up WP placeholder in `WORK_INDEX.md`, triggered the first time any `MatchSetupConfig.heroDeckIds` selection includes an AMWP-class hero. WP-135's scope held.

### 3.4 Auto-reshuffling discarded recruited cards back into `G.heroDeck`

Tempting because it would keep the HQ populated indefinitely. Rejected at D-13503. Auto-reshuffling would conflate the per-player owned-card layer (a player's discard) with the shared hero-supply layer (`G.heroDeck`), violate Marvel Legendary's tabletop rule, and break the §3 Trust & Fairness ownership boundary. Once a card is in a player's discard, it is owned by that player; the hero deck stays empty when exhausted, and the slot stays `null`.

### 3.5 Inline `splice` / `shift` / `push` on `G.heroDeck` and `G.hq`

Tempting because Immer would handle the mutation transparently. Rejected per the aliasing-defense rule at WP §K. The pure helpers `fillHqFromDeck` and `refillHqSlot` build new arrays and the orchestrator / move body rebinds `G.hq = result.hq; G.heroDeck = result.heroDeck` to apply the change. This keeps the helper contract pure and grep-able — every mutation site is visible at the rebinding line, never buried inside an inline `.shift()`.

## 4. Aliasing Audit

Every helper return is a freshly allocated array. The orchestrator and move body rebind `G` fields to those new arrays. No `G` reference escapes:

- `buildHeroDeckCards(heroDeckIds, registry)` — returns a fresh `cards: CardExtId[]` built by `cards.push(extId)` inside the rarity-map loop.
- `shuffleHeroDeck(cards, context)` — calls `context.random.Shuffle([...cards])`. The defensive copy `[...cards]` ensures the caller's array is never aliased.
- `buildHeroDeck(heroDeckIds, registry, context)` — composes the two; the returned array is the result of `shuffleHeroDeck` (a new array).
- `fillHqFromDeck(heroDeck, slotCount)` — returns `{ hq: HqZone, remainingDeck: CardExtId[] }`, both freshly allocated via `newHq.push(...)` and `remainingDeck.push(...)`. Input `heroDeck` is read-only.
- `refillHqSlot(hq, hqIndex, heroDeck)` — returns `{ hq: HqZone, heroDeck: CardExtId[] }`, both freshly allocated. Input arrays are read-only.

Tests (`city.logic.test.ts`, `buildHeroDeck.test.ts`) assert input-array immutability via deep-equal-against-pre-call-snapshot.

In `recruitHero.ts`:

```ts
const refillResult = refillHqSlot(G.hq, hqIndex, G.heroDeck);
G.hq = refillResult.hq;
G.heroDeck = refillResult.heroDeck;
```

The two `G` rebinding lines are the only mutation sites. No `G.heroDeck.shift()` / `G.hq[i] = ...` / `G.hq.splice(...)` anywhere in the new code.

In `buildInitialGameState.ts`, the orchestrator computes `filledHqResult` from `fillHqFromDeck(shuffledHeroDeck, HQ_SLOT_COUNT)` and assigns `hq: filledHqResult.hq` + `heroDeck: filledHqResult.remainingDeck` into the `LegendaryGameState` literal. No further mutation.

## 6. Lessons for Future Sibling-Snapshot WPs

### 6.1 Single-shuffle determinism envelope

The WP §7.8 lock is non-negotiable: exactly ONE `ctx.random.Shuffle` call at setup; no per-turn reshuffle, no batched front-pop, no deck reorder. Future contributors who "preserve test pass" by widening the envelope (e.g., adding a per-turn reshuffle to "fix" empty-deck behavior) silently break replay determinism. The replay-hash regression guard is the canary.

### 6.2 Ordering-stable registry walks

`buildHeroDeckCards` iterates heroes per `config.heroDeckIds` order, cards per registry `cards[]` order, copies per rarity-map iteration order. Any reorder (sorting heroes alphabetically, sorting cards by rarity rank, sorting copies by ext_id) would produce a different unshuffled flat array → different shuffled output → different `G.heroDeck` content → different `computeStateHash` → cascading replay-hash drift. Lock the iteration order at the helper site; never reorder downstream.

### 6.3 Pure helper boundaries

`buildHeroDeck.ts` and `city.logic.ts` (the WP-135 additions) have no `boardgame.io` import, no `Math.random`, no I/O, no `.reduce()`. The orchestrator passes `context: ShuffleProvider` through the function chain; the helpers never see `FnContext` or any boardgame.io type. This keeps the engine layer's pure-helper rule intact (`.claude/rules/architecture.md §Layer Boundary`).

### 6.4 Loud-fail at `Game.setup()` as the locked surface for enumerated-value drift

The pattern is reusable: when registry data is expected to satisfy a locked enum set, the setup-time builder throws a full-sentence Error naming the offending entity, the offending value, and the locked permitted set. The throw lands inside `Game.setup()` (the only legitimate throw site in the engine). No other call path may swallow the error.
