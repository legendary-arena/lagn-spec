## 9. Lessons Learned

**CRLF + a `$`-anchored row regex is a silent-empty trap.** The first dry-run produced an all-`0/0` table because `parseWorkIndex` split on `\n`, leaving a trailing `\r` on every line; `.` does not match `\r`, so the `$`-anchored checkbox regex matched nothing and the ledger parsed empty (Foundation/placeholder rows still rendered because they trim). The failure was data-silent, not a crash — caught only because the counts were obviously wrong. Fix: parsers split on `/\r?\n/`, and the write path detects and preserves the file's own EOL so the byte-unchanged-outside-markers contract holds on a CRLF repo. Future text-parsing tooling on these governance docs must assume CRLF.

**Dogfooding immediately paid for itself.** Running the generator against its own WP surfaced exactly the drift the WP predicted: two orphans (WP-236, WP-240) and the Dashboard `12 → 13` undercount (WP-238 was noded but uncounted). The total landed at `WP ✅ 235/237` where `237` equals the raw WORK_INDEX checkbox count — a strong proof that every WP is counted in exactly one cluster, no orphan, no double-count.

**The grep-gate comment self-trip recurred (4th+ time).** The cron-step explanatory comment quoted the forbidden `|| true` idiom that verification step 6 greps for — identical class to WP-226/233/236/239 (`feedback_grep_gate_comment_self_trip`). Resolved by paraphrasing the comment to "an always-true OR-suffix appended to the command" so the meaning survives without the literal token. The `architecture-inventory.yml` template carries the same literal in its comment; mirroring it verbatim is what reintroduced the trip.
