## 9. Forward-Safety Questions

- [x] **Can this code survive future refactors without touching
      unrelated layers?** Yes. The admin profile surface composes
      `requireAdminSession` through its public `AdminSessionResult`
      interface only — a future RBAC migration (per WP-159 / D-15902)
      that replaces the `is_admin` column with a `legendary.player_roles`
      JOIN leaves WP-107 byte-identical. The `requireUnsuspendedAccount`
      helper composes `legendary.players.is_suspended` through a single
      SELECT; a future schema migration (e.g., a `legendary.player_suspension_events`
      append-only table replacing the boolean) requires updating the
      helper's SELECT but no caller change
- [x] **Can replay/debugging reconstruct behavior from stored data?**
      Yes for admin actions: every successful suspend/unsuspend writes
      one row to `legendary.admin_actions` with the acting admin's
      `AccountId`, the target's `AccountId`, the action type, the
      trimmed reason, and the timestamp. The audit log IS the
      reconstruction primitive for moderation decisions. Replays do not
      capture or replay admin gameplay impact — historical scores
      remain on leaderboards per Q3 LOCK = STAY (suspension is
      forward-only)
- [x] **Would removing upstream data still fail safely?** Yes. If
      `legendary.players` were emptied, every handle resolution returns
      `null` → `404 { code: 'not_found' }`; every
      `requireUnsuspendedAccount` call returns
      `'lookup_failed'` with the zero-row canonical reason. If WP-159
      were unwired, every admin request resolves to `'unauthorized'`
      via the upstream `'session_verifier_not_configured'` collapse
      locked in WP-159 §A
- [x] **Is it impossible for this WP's output to influence gameplay
      unintentionally?** Yes. The surface has zero engine surface — no
      `G` mutation, no engine import, no move / phase / hook touch.
      The downstream gameplay impact of suspension (blocking new score
      submissions) is mediated by `requireUnsuspendedAccount`, which is
      consumed by a future write-path handler that does NOT exist at
      HEAD; gameplay impact is forward-only and explicit, not implicit
- [x] **Is the contract stable enough to be referenced by future WPs?**
      Yes. `AdminProfileResponse`, `AuditLogEntry`, `AdminActionRequest`,
      `AdminActionResponse`, `AdminProfileErrorCode`, `AdminPlayerActionType`,
      `RequireUnsuspendedAccountResult`, and
      `RequireUnsuspendedAccountErrorCode` are all locked under WP-107
      §Locked contract values. The next moderation WP extends
      `AdminPlayerActionType` additively (e.g., `'void_score'`,
      `'hide_replay'`) by updating both the union AND the DB CHECK
      constraint in migration 015 (replace-whole-row per D-11804 catalog
      semantics, lockstep update per the drift-detection test). The
      first caller of `requireUnsuspendedAccount` (the future
      score-submission request-handler WP) inherits the locked HTTP
      error mapping verbatim

## Closing Notes

The **transaction-ownership lock in the logic layer** is the
load-bearing invariant of WP-107. The grep gate that enforces
`adminProfile.routes.ts` as zero-transaction-control is what makes the
admin-mutation surface auditable as a single atomic unit at every
commit: a future moderator-edit feature that wants to "just add a
quick INSERT after the route handler returns" is structurally
prevented because the route handler has no transaction surface to
extend. Every future admin-action WP that extends `legendary.admin_actions`
MUST own its transaction in its own logic file; the routes layer
remains a thin Koa adapter.

The **append-only audit table** is the other piece worth flagging.
The DB-level CHECK + FK constraints in migration 015 are
defense-in-depth, not the primary gate — but they ARE what makes the
table forward-compatible to future moderation actions without a
schema migration. A future WP that adds `'void_score'` to the
`AdminPlayerActionType` union extends the CHECK constraint via a
small ALTER TABLE; the table shape, the index, the FK constraints,
and the application-layer transaction envelope all stay byte-identical.
The seam was designed for this; the next moderation WP gets to ship
its per-record logic without re-deriving any of WP-107's primitives.

The **handle-in-URL choice** (D-10703) is the third design point
worth calling out for future admin-surface WPs. Every admin-facing
URL in the codebase from this WP forward should follow the same
pattern: handle in URL, server-side resolution to AccountId before
any authorization decision. The AccountId-bearing path remains
available as a future fallback (a CLI / internal-tooling WP could
ship `:accountId` routes if a handle-less surface ever becomes
necessary), but today every admin workflow is handle-anchored and
the URL shape reflects that.

The Option A scope-out (RS-1, 2026-05-23) — removing the
score-submission intake-hook from this WP and shipping the helper
`Library-only` — was the right call. The execution session would
have had to invent a score-submission HTTP route just to wire the
helper, which would have inflated scope and added a future-rework
burden when the actual score-submission request-handler WP ships.
The helper's HTTP error mapping is locked here as a caller-contract;
the future WP wires the import + early-return guard in ≤ 6 lines
and inherits the contract verbatim. The deferral pattern (ship the
library, defer the wiring) mirrors WP-053 / D-10202 and WP-112 /
D-11204 precedents.
