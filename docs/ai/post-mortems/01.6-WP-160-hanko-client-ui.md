### Mandatory session-prompt questions

**(a) What was harder than expected?**

The Hanko SDK API drift surfaced in pre-execution inspection. The WP body §A point 3 / §Scope (In) §A spec'd `hanko.user.logout()`, but the `@teamhanko/hanko-frontend-sdk` `Hanko` class declares `private readonly user` — the public sign-out method is `hanko.logout()` directly. This was true in both 2.4.0 (the WP's floor) and 2.6.0 (the latest satisfying `^2.4.0`). Verified by inspecting the dist `.d.ts` from npm.

Folded inline per `01.0b §Common deviations` (single-correction drift, well under the ~5-amendment budget): the wrapper now calls `handle.hanko.logout()` directly; the external `signOutCurrentSession` surface is unchanged. Documented in the wrapper's `// why:` block (citing the WP-vs-reality drift) and in the test description ("invokes hanko.logout() exactly once").

A second small drift: the EC's grep gates on auth.ts (`grep -n "hanko\|@teamhanko"` must return zero) and on LoginPage.vue (`grep -n "<hanko-auth"` must return exactly 1) initially failed because JSDoc commentary referenced the wrapper file path (`hankoClient.ts`) and the custom element tag (`<hanko-auth>`) respectively. Both JSDoc blocks were paraphrased during execution to clear the gates without changing load-bearing behavior. This is an example of the EC's grep discipline catching prose that would otherwise look harmless.

**(b) What could silently regress in future WPs?**

Three classes of silent regression to watch:

1. **A future WP imports `@teamhanko/*` into another file for convenience.** F-2 violation. The repo-wide grep `from '@teamhanko/` should fire in CI. Today no CI gate enforces this; the EC's verification grep is operator-run. **Mitigation:** a future WP that touches `apps/arena-client/src/auth/` should add a pre-commit / CI grep gate that asserts only `auth/hankoClient.ts` matches.

2. **A future WP adds an admin-only client surface and forgets to extend the route-guard closed set.** App.vue's guard is keyed on `route === 'me' || route === 'admin-billing'` exactly — if a future admin route (e.g., `?route=admin-team-audit`) lands without extending this check, the route would be silently un-guarded. **Mitigation:** the upcoming WP-107 admin-client surface (cited in WP-160 §Non-Dependencies) MUST extend the guard set; a future closed-set drift test would catch this earlier.

3. **A future broker SDK upgrade renames `getSessionToken`, `onSessionCreated`, `onSessionExpired`, or `onUserLoggedOut`.** The wrapper's tests stub a fake `HankoLike` with these names — a real-SDK rename would not fail the tests (the fake satisfies the wrapper's call sites) but would crash at production runtime on the first sign-in attempt. **Mitigation:** the `package.json` pins `^2.4.0` (major-2 lock); a future broker upgrade across a major version MUST re-run the API inspection that surfaced the `hanko.user.logout()` drift, with the new types as the source of truth.

A fourth concern (smaller): the `isAuthBootstrapping` gate in App.vue holds render until the broker init resolves. If the broker tenant is slow (>1s), users on guarded routes see "Preparing sign-in…" for that duration. Not a correctness regression, but a UX one. A future hardening WP could surface a timeout/retry surface.

**(c) Did the broker abstraction hold perfectly?**

**YES, with one caveat.**

The four-function wrapper surface (`initializeHankoClient`, `getCurrentTokenFromHandle`, `signOutCurrentSession`, `subscribeToSessionEvents`) plus the two interfaces (`HankoClientHandle`, `HankoSessionListeners`) plus the typed error (`HankoInitializationFailed`) covered every consumer need with zero leakage of broker-specific types or strings to the auth store, the LoginPage, App.vue, or MyProfilePage. The F-1 and F-2 grep gates all passed. The Pinia auth store stores `string | null` and `boolean` only. A future broker swap is a single-file replacement plus a custom-element tag rename plus a dep swap.

**Caveat:** the wrapper's `HankoLike` interface is exported (a 9th export beyond the spec's 7) so the test fake can implement the shape without importing `@teamhanko/*`. This is a small interface — `getSessionToken`, `logout`, three on-event registrars — and it does NOT leak broker-specific types (the on-event callbacks are typed `(detail: unknown) => void`). But it IS a structural mirror of the broker's API surface; a future broker whose API shape diverges sharply (e.g., promise-based event subscription) would require adjusting both `HankoLike` and the production factory in lockstep. This is acceptable for a structural test seam and matches the WP-126 server-side `HankoLike` precedent.

### Open question for the next router-related WP

Per the session prompt §Open Question for the Post-Mortem: the LoginPage uses `window.location.assign(...)` for a full reload (D-16008). This is correct for D-16006 (the post-reload App.vue bootstrap re-reads the cached cookie and instantiates the auth store fresh), but it loses in-memory Vue app state. A future WP that introduces `vue-router` may want to revisit this with SPA-style routing once auth is stable — at that point, `useAuthStore().setSession(token, null)` + a `router.push({ name: returnTo })` would replace the full reload. **Not in scope for WP-160.**
