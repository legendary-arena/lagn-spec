## New Patterns Discovered

### Defense-in-depth fallback for physicalCards migration

Each of the four consumer sites follows an identical structure:

```
const physicalCards = parsePhysicalCards(entry.physicalCards);
if (physicalCards.length > 0) {
  // primary path — iterate physicalCards
} else {
  // fallback — old per-side logic
}
```

The parser (`parsePhysicalCards` / `parseDisplayDataPhysicalCards` / `parseEconomyPhysicalCards`) validates shape defensively — each variant checks only the fields it needs (engine setup: count + sides; display data: count + sides + imageUrl; economy: count + sides). This avoids coupling between consumer sites while keeping the validation pattern consistent.

The fallback path is exercised by any hero data that pre-dates Phase 1a patching or by test fixtures without `physicalCards`. The replay hash confirmed this: the deterministic replay test uses legacy fixture data and hits the fallback path, producing the same hash.

### Viewer operates on raw SetData, not registry instances

The WP spec assumed `flattenSet` could call `registry.getPhysicalCardForSide()`. In practice, `flattenSet` receives `SetData` directly (a Zod-parsed JSON blob), not a `CardRegistry` instance. The solution was to build the lookup inline:

```typescript
const sideToImageUrl = new Map<string, string>();
for (const physicalCard of hero.physicalCards) {
  for (const sideSlug of physicalCard.sides) {
    sideToImageUrl.set(sideSlug, physicalCard.imageUrl);
  }
}
```

This achieves identical semantics to `getPhysicalCardForSide()` scoped to one hero, without threading a registry instance through `flattenSet`.

### Per-domain parsePhysicalCards variants

Rather than a shared generic parser, each domain defines its own typed variant. This avoids cross-package imports and lets each consumer validate exactly the fields it reads:

| Consumer | Interface | Validates `imageUrl`? |
|---|---|---|
| `buildHeroDeck.ts` | `PhysicalCardEntry` | No |
| `buildCardDisplayData.ts` | `DisplayDataPhysicalCardEntry` | Yes |
| `economy.logic.ts` | `EconomyPhysicalCardEntry` | No |
| `shared.ts` (viewer) | Inline (raw Zod-parsed data) | Yes (via Zod schema) |

Intentional duplication. The three engine variants are structurally similar but live in different files with different import contexts. The viewer uses Zod-parsed data directly, so no separate interface was needed.

## Risks / Open Items

1. **Phase 3 scope** — `HeroCardSchema.imageUrl` remains in the registry schema. Phase 3 (future WP) removes it and drops the fallback paths from all four consumer sites. Until then, both paths coexist.
2. **Non-hero imageUrl reads** — 8 card-type blocks in `shared.ts` still read `card.imageUrl` directly. These are out of scope per D-14103 (hero-only migration). If physicalCards extends to non-hero types, those blocks will need the same treatment.
3. **Test fixtures lack physicalCards** — the replay-hash test and several existing unit tests exercise only the fallback path. Phase 3 should add physicalCards-bearing fixtures to the deterministic replay suite so the primary path is hash-verified.
