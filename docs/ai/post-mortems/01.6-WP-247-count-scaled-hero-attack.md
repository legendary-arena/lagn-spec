### 4. Representation & Determinism Audit

- [x] New representation is data-only + JSON-serializable: `countSource?` is a
  string-union value on the already-serializable `HeroEffectDescriptor`. No functions/
  Maps/Sets/Dates/classes added to `G`. `JSON.stringify(G)` after a grant is tested.
- [x] Execution is deterministic: grant = `magnitude × resolveCountSource(...)`, a pure
  function of `G` at play time. No RNG, no clock, no caching/memoization, no module-level
  mutable state.
- [x] Unknown/future values fail safe: an unrecognized `HeroCountSource` resolves to `0`
  (no throw); a missing `countSource` on the descriptor is a skipped no-op; an unrecognized
  count-source token emits no effect (and so does not suppress the printed attack icon).

**Determinism note:** sentinel / `PRE_WP080_HASH` untouched — no recorded fixture plays
Covert Operation, so no replay divergence (WP-236 discipline; no re-pin needed).

### 5. Mutation, Aliasing, & Reference Safety (High-Risk)

- [x] All mutation occurs only in the executor (an authorized lifecycle context dispatched
  from `playCard`). The setup parser mutates only local accumulators.
- [x] The resolver mutates nothing — it reads `G.playerZones[playerID].victory` and counts.
- [x] No mutation during the parse/resolve paths.
- [x] Pure helpers are pure (confirmed by inspection + the resolver determinism/exclusion tests).
- [x] **No aliasing bugs.** `resolveCountSource` returns a primitive `number`; it never
  returns a reference into `G`. The executor reassigns `G.turnEconomy` via the existing
  `addResources` (which returns a fresh object), never aliasing a `G` array.
