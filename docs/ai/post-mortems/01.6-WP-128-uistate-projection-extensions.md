## Section 1 — Overview

WP-128 extends the engine UI projection contract along the board-layout wireframe (`docs/ai/DESIGN-BOARD-LAYOUT.md §4`) without expanding `G`. The packet adds three new top-level `UIState` fields (`decks: UIDecksState`, `piles: UISharedPilesState`, `koPile: UIKoPileState`), five new optional per-player fields (`inPlayCards?` / `inPlayDisplay?` / `discardTopCard?` / `victoryCards?` / `victoryVP?`), six new required fields on existing types (`mastermind.attachedBystanders`, `mastermind.strikePile`, `scheme.twistPile`, `city.escapedPile`, `economy.piercing`, `economy.woundsDrawn`), and a shared `UIDisplayEntry = { extId; display: UICardDisplay }` alias used by every face-up pile / array projection. The arena-client UI binding (consuming `victoryCards[]` for the victory-pile drill-down, `inPlayCards[]` for the in-play row, `discardTopCard` for the discard preview, `koPile.cards[]` for the KO-pile drill-down, `decks.*` / `piles.*` for the shared-board counters, `mastermind.{attachedBystanders, strikePile}` / `scheme.twistPile` / `city.escapedPile` for the destination piles) is explicitly out of scope and deferred to WP-129 (board-layout components). WP-128 is the contract that makes WP-129 unblockable.

## Section 3 — New Patterns Discovered

### 3.1 Safe-skip-with-marker pattern at projection time

WP-023 / WP-025 / WP-026 / WP-030 established the safe-skip pattern at **evaluator time** (rule-execution helpers that gracefully skip unsupported effect types instead of throwing). WP-128 applies the same pattern at **projection time**: when a projection field's `G` source doesn't exist, the projection assigns a typed-stable default (`[]` / `0`) plus a CI-greppable marker (`// SAFE-SKIP-WP128`) plus a 3-clause `// why:` comment.

The pattern's value over a STOP-and-defer alternative:

- The contract ships in full so downstream consumers (WP-129) can bind to a stable shape immediately. Without the contract, the entire board-layout chain would cascade-block.
- The CI marker makes the safe-skips auditable and revertible. When the future WP adds the underlying `G` state, CI grep finds every site that needs to flip from constant to derivation. The marker count drops as sites are resolved — a built-in audit signal.
- Required-field contract preserved: each safe-skip projects a typed-stable default — never `undefined`. Consumers don't proliferate null-checks.
- Field name and shape stay identical when the future WP lands. Only the assignment value flips (constant → derivation). No consumer churn.

**When to apply this pattern in future WPs:**

- The projection contract has a clear stable shape that downstream consumers depend on.
- The `G`-side data is missing because the engine WP that produces it hasn't landed (not because the design is uncertain).
- The safe-skip count is bounded and enumerable (here: exactly 8, listed verbatim in D-12806).
- Each safe-skip site can be marked with a CI-greppable token tied to the WP that owns the deferral.

**When NOT to apply this pattern:**

- Design uncertainty (the right shape isn't known yet) — STOP-and-design instead.
- Unbounded safe-skip surface ("most fields might land as safe-skips") — the WP is too speculative; defer to a later phase.
- The safe-skip default would be semantically wrong (e.g., projecting `null` when the consumer expects a value) — escalate; pick a different shape.

### 3.2 Shared `UIDisplayEntry` type alias for repeated `{ extId, display }` shape

The pre-WP-128 codebase had `UIHQCard = { extId; display: UICardDisplay }` declared once for HQ slots. WP-128 introduces 7 more fields with the same shape (`victoryCards`, `strikePile`, `twistPile`, `escapedPile`, `koPile.cards`, `koPile.topCard`, `discardTopCard`, `attachedBystanders`). Repeating the inline literal `{ extId: string; display: UICardDisplay }` at every consumer site would have been a DRY violation; defining `UIDisplayEntry` once and reusing it keeps the projection contract uniform.

The pattern applies to any future WP that introduces N ≥ 3 fields with the same nested shape — define a named type alias once and reuse. The drift test then pins the alias's shape independently (one type-pin guards N consumer sites).

### 3.3 Per-entry shallow copy via shared helper (`buildDisplayEntries` / `deepCopyDisplayEntries`)

WP-111 established the per-entry shallow-copy aliasing-defense pattern (`resolveDisplay` returns a fresh `{...G.cardDisplayData[extId]}` per call). WP-128 generalizes this with two helpers:

- **Projection time:** `buildDisplayEntries(extIds, gameState)` — iterates `extIds`, returns `UIDisplayEntry[]` with each entry built via `{ extId, display: resolveDisplay(extId, gameState) }`. Used by `victoryCards`, `koPile.cards`.
- **Filter time:** `deepCopyDisplayEntries(entries)` — iterates input `UIDisplayEntry[]`, returns fresh `{ extId, display: { ...entry.display } }` per entry. Used by `victoryCards`, `attachedBystanders`, `strikePile`, `twistPile`, `escapedPile`, `koPile.cards`.

The forbidden patterns (per WP-111 D-11105) are still anti-patterns and are documented inline in `uiState.build.ts`:

- Returning a `G`-owned array directly (no copy at all).
- Identity-mapping (`zone.map(e => e)`) — looks like a copy but aliases the entries.
- Array spread without per-entry clone (`[...zone]`) — fresh array, aliased entries.

### 3.4 Optional + nullable field for "redacted vs visible-but-empty"

The `discardTopCard?: UIDisplayEntry | null` shape encodes two distinct states:

- **`undefined`** (optional unset): redacted by audience filter (the field was omitted entirely).
- **`null`** (optional set to null): visible, but the discard pile is empty (`discardCount === 0`).

Without this combo, `?: T` would conflate the two states; `?: T | undefined` would be ambiguous; a non-optional `T | null` couldn't be redacted. The `?: T | null` + a 3-clause `// why:` comment at the declaration site keeps the semantics legible.

This pattern applies any time a field has both a redaction posture AND a "visible but empty" rest state. Document it explicitly so future readers don't ask "why both?"

## Section 4 — Anti-Patterns Discovered

### 4.1 Tempted to flatten `G.attachedBystanders` (city-villain captures) into mastermind tile

Pre-flight Risk R-4 surfaced two interpretations for `mastermind.attachedBystanders`. Interpretation A (flatten city-villain captures from top-level `G.attachedBystanders` into the mastermind tile) was the user-recommended path initially. The pre-flight executor (Claude) flagged the change of mind in the same response that applied PS-4 — Interpretation B locked instead.

**Why Interpretation A was wrong:**

- `G.attachedBystanders` is keyed by **city villain CardExtId** — semantically "bystanders attached to specific city villains," not "bystanders attached to the mastermind."
- Flattening misrepresents city-row data as mastermind-row data.
- Bystanders attached to city villains are already rendered on those villains in the city row; flattening into the mastermind tile would double-render.
- Locks in a derivation that future engine work (Master Strike–driven mastermind capture) would have to undo.

**Guardrail at three levels** (preserves the lock against future contributors):

1. **WP-128 §Non-Negotiable Constraints bullet** — explicit "Do NOT flatten `G.attachedBystanders` (city-villain captures, top-level on `LegendaryGameState`) into this projection."
2. **D-12805 body** — Interpretation B locked with rejected-alternative block citing all three reasons.
3. **EC-131 §2 + §Common Failure Smells** — failure-smell entry warns against flattening; verification gate runs grep against the safe-skip site comment.

The lesson generalizes: when a projection's wireframe placement (mastermind tile) and the closest engine source (`G.attachedBystanders` keyed by city villain) are semantically misaligned, the safe-skip + future-WP path is more honest than a forced flatten. The wireframe location is correct; the data type is empty until a future engine WP introduces the matching engine surface.

### 4.2 Tempted to project `koPile` from `G.piles.ko`

The original WP-128 draft specified `koPile` projected from `G.piles.ko`. Pre-flight verified `LegendaryGameState` (`types.ts:408-559`) and found:

- `G.ko: CardExtId[]` exists at top level, line 481.
- `G.piles` exists but contains only `bystanders / wounds / officers / sidekicks` (no `ko` key).
- `GlobalPiles` interface (`zones.types.ts:68-77`) has no `ko` field.

PS-1 corrected the path to top-level `G.ko`. The verification gate (EC-131 §5) added a grep guard: `Select-String -Path "packages\game-engine\src\ui\uiState.build.ts" -Pattern "G\.piles\.ko\|gameState\.piles\.ko"` returns no output at session close. Even the explanatory comment that previously cited "NOT `G.piles.ko`" was reworded to avoid the literal substring so the gate stays clean.

The lesson: pre-flight read of canonical types is non-negotiable. WP drafts can drift from actual `G` shapes; pre-flight catches those drifts before execution.

### 4.3 Tempted to assign `undefined` to required safe-skip fields

The required-field contract (`mastermind.attachedBystanders: UIDisplayEntry[]` not `?:`) demands a typed-stable default. The instinct under "no source available today" is to project `undefined`, but that breaks the required-field contract: every consumer (now and in the future) would need null-checks.

**Resolved at D-12806 lock-in:** all 8 safe-skip sites use typed-stable defaults (`[]` for arrays, `0` for counts) — never `undefined`. The drift test pins each safe-skip default explicitly so a future contributor adding the real source value sees the test fail before drift propagates.

### 4.4 Tempted to widen `UIPlayerState` further

The five new optional per-player fields (`inPlayCards?` / `inPlayDisplay?` / `discardTopCard?` / `victoryCards?` / `victoryVP?`) are the minimum needed for the wireframe. Mid-execution there was a faint pull toward also adding `discardCards?: string[]` (full discard array beyond just the top) and `discardDisplay?: UICardDisplay[]`. Resisted because:

- The wireframe shows top-of-discard only. Full-discard drill-down is a future affordance.
- Adding fields beyond what the wireframe needs is scope creep.
- Each new optional adds a redaction rule the filter must respect — three more sites at the filter layer.
- Future WPs can extend the contract additively. WP-128 doesn't need to pre-empt future surface.

Held to scope. The five optional fields shipped exactly as enumerated in EC-131 §2.

## Section 7 — Lessons for Future Projection WPs

1. **Safe-skip with CI marker is the right answer when contract shape is known but `G` source isn't.** The marker pattern (`// SAFE-SKIP-WP128`) makes deferred work auditable. Each future WP that adds underlying `G` state can grep for the marker, find every site to flip, and verify the marker count drops as sites land. A WP without this auditable seam silently accumulates technical debt.

2. **Per-entry shallow copy is non-negotiable for projection arrays.** Standard tests (deep equality, JSON roundtrip) cannot detect aliasing. The aliasing test pattern (mutate a returned entry, assert next `buildUIState` returns un-corrupted shape) is the only contract that catches it. Apply this test to every WP that adds `UIDisplayEntry[]` projection.

3. **Optional + nullable encodes redaction-vs-empty distinction unambiguously.** When a field has both an audience-filter redaction posture AND a "visible but empty" rest state (`discardTopCard?: T | null`), document the distinction at the declaration site. Without the comment, future readers ask "why both?"

4. **Pre-flight verification of canonical types catches WP-draft drift.** The PS-1 `G.piles.ko` correction caught a path that the WP draft cited without verification. Three patterns make this routinely catchable: (a) read `LegendaryGameState` end-to-end before authoring projection paths; (b) verify any non-trivial nested path against `*.types.ts` source files; (c) run a pre-flight grep against `gameState.X` references vs the actual interface shape.

5. **Field-name canonical casing matters.** PS-2 corrected `victoryVp` → `victoryVP` to match `PlayerScoreBreakdown.totalVP`. The `00.6` Rule 14 lock ("canonical field names per `00.2-data-requirements.md` exactly") catches drift at lint time, but only if pre-flight reads the canonical source. WP drafts that paraphrase field names without source verification miss this.

6. **D-DEC-N → D-NNNNN mapping is non-trivial when interpretations are involved.** D-DEC-5 is "Mastermind `attachedBystanders` shape" but the WP-128 lock combines shape + data semantics (Interpretation B). The DECISIONS.md entry (D-12805) records both; the WP §Decision Points block is the source.

7. **Test-count budget [12, 20] requires aggressive consolidation.** Initial test count landed at 23 tests + 4 suites = 27. Trimmed to 17 tests + 3 suites = 20 by (a) removing redundant tests (where drift type-pins overlap with safe-skip value-pins), (b) merging conceptually-related tests into single `it` blocks with multiple assertions, (c) collapsing the WP-128 drift describes into a single block. The lesson: count test-count from the start, plan consolidation paths before writing.
