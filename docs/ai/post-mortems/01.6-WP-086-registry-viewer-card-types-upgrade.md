The EC's §Files to Produce list (lines 47-55) however contained exactly 8 files; `devLog.ts` was not among them.

### Surprise 2 — `debugMode.ts` Vite-vs-node:test runtime gotcha

**Discovery point:** End of Phase 3 verification, when running `pnpm --filter registry-viewer test` (the new runner from PS-2 Option B) for the first time. Output:

```
TypeError: Cannot read properties of undefined (reading 'DEV')
    at <anonymous> (apps/registry-viewer/src/lib/debugMode.ts:19:19)
```

The full chain: `cardTypesClient.test.ts` imports `cardTypesClient.ts`, which imports `devLog`, which imports `DEBUG_VIEWER` from `debugMode.ts`, which evaluates `import.meta.env.DEV` at module load — and `import.meta.env` is `undefined` under node:test because Vite is not in the loader chain.

**Severity:** HIGH. Test file fails at module load (not in a specific test); 1 of 5 tests reported `fail`. Without resolution, the EC §67 verification gate ("`pnpm --filter registry-viewer test` emits TAP output for ≥ 8 tests / 2 suites with `fail 0`") cannot pass.

**Resolution path:** IIFE + try/catch wrap around the existing 2-line expression preserves Vite DCE behavior in prod (Vite still substitutes `import.meta.env.DEV` to literal `false`, so the inner expression is statically false and DCE strips the URLSearchParams reference) while making the module load gracefully under node:test. Operator approved inclusion as the 10th production file with a D-86NN audit-trail entry (folded into D-8608 alongside `devLog.ts`).

**Root cause:** PS-2 Option B's pre-flight evaluation focused on the package.json + tsx devDep additions (the visible mechanics of wiring `node:test` to a Vite app). It did not enumerate the `import.meta.env.*` consumers in the consumed-by-tests transitive closure.

**Specifically:** `debugMode.ts:19` is the only `import.meta.env.DEV` consumer outside Vite's own substitution surface, but it was not flagged because the EC's pre-flight Q-table for "first-time test runner introduces" did not include that question.

## Section 9 — Carry-Forward Lessons

### 9.1 — EC §Files list reconciliation against WP body anticipation

**Lesson:** The EC's §Files to Produce list is the binding scope-lock per EC §7. The WP body and pre-flight may anticipate file changes that don't appear in the EC list. When the WP body says "this packet adds X to file Y" but file Y is not in the EC's §Files list, that's an EC-vs-WP-body contradiction that the executor will hit at compile time.

**Proposed pre-flight check:** At EC-finalize time (final pre-flight), grep the WP body for verbs like "add to" / "modify" / "extends" / "joins" / "becomes" + per-file references; reconcile against the EC §Files list. Any anticipated modification not in the EC list should either (a) get added to §Files at finalize time, or (b) be explicitly called out as "anticipated edit beyond §Files; allowed because additive/trivial; no escalation needed."

**Existing precedent surfaced by this lesson:** WP-086's preflight had THIS SAME ISSUE. preflight-wp086.md line 200 reads *"No modification of `apps/registry-viewer/src/lib/{devLog,...}.ts`"* — internally contradicting lines 59 + 76 + the WP body. A second-pass reconciliation gate (or even a simple grep-self-consistency check on the preflight) would have caught the contradiction before execution.

### 9.2 — First-time test runner introduces a new runtime context (Vite vs node:test)

**Lesson:** Adding a `node:test` runner to a Vite-only app introduces a new runtime context that lacks Vite's compile-time injections. Modules that work fine in the Vite path may load at runtime in the node:test path and throw at module load.

**Specific Vite injections to audit:**
- `import.meta.env.DEV` / `.PROD` / `.MODE` / `.SSR` — substituted at compile time by Vite; `undefined` at runtime under node:test
- `import.meta.url` — works in both contexts (ESM standard) but may resolve differently
- `process.env.VITE_*` variables — set by Vite's dev server; absent from node:test
- DOM globals (`window`, `document`, `location`, `navigator`) — present in browser; absent from node:test
- `import.meta.glob` (Vite-only build-time helper) — absent from node:test

**Proposed pre-flight check:** For any "first-time test runner introduces" change, the pre-flight Q-table should include:
- (a) Inventory of `import.meta.env.*` consumers in the consumed-by-tests transitive closure (grep `import\.meta\.env`).
- (b) Inventory of DOM-global consumers in the same closure (grep `\b(window|document|location|navigator)\b`).
- (c) Explicit decision on whether each gets a runtime guard (try/catch, `typeof` check), a build-time substitution (Vite's `define`), or a module-replacement (e.g., a `*.test.ts`-only mock).

In WP-086's case, item (a) would have flagged `debugMode.ts:19` at pre-flight time and surfaced the IIFE+try/catch fix as PS-N rather than a mid-execution surprise.

### 9.3 — Deletion-then-readd at the same path requires doc-staleness sweep (Option-C precedent)

**Lesson:** When a WP reintroduces a previously-deleted artifact at the same path with a different schema, doc staleness propagates in non-obvious ways. The WP/EC's §Files list typically focuses on production code; rules files (`.claude/rules/*`), data-requirements docs (`00.2-*.md`), educational comments in code (`httpRegistry.ts:51-53`), and CLAUDE.md tables tend to escape the file-list radar.

**Proposed pre-flight check:** Any WP that satisfies the deletion-then-readd pattern (defined: same repo path, new content/schema, prior WP referenced as the deletion baseline) should run a `grep -r "<path>"` over the repo at pre-flight time. For each non-test, non-historical-narrative match, decide:
- (a) Does the reference accurately describe post-readd behavior? If yes, leave it.
- (b) Does it describe the deletion as historical context with a clear date / WP reference? If yes, leave it as-is or augment with the readd date.
- (c) Does it document the OLD shape as if current? If yes, the file needs updating — add to EC §Files OR document a deferred-cleanup DECISION.

Option-C ("treat doc-staleness sync as governance close") is the appropriate disposition when the staleness scope is small (≤5 files) and contained to documentation surfaces. For larger scopes, Option B (amend WP/EC bodies) becomes more proportionate.

### 9.4 — Commit-msg hook path-discipline forces topology splits (B-1 / B-2 pattern)

**Lesson:** The project's commit-msg hook enforces `EC-NNN:` prefix for any staged file under `packages/` or `apps/`, regardless of edit content (a comment-only change to `httpRegistry.ts` triggers the same hook as a behavioral edit). When a governance-close commit needs to touch documentation files inside those paths, the hook forces a topology split into a SPEC commit (doc-only files) and a follow-up EC commit (code-path files).

**Future precedent:** When the governance close includes any file under `packages/` or `apps/`, expect to split into B-1 (SPEC) + B-2 (EC-NNN:). This is not a process failure; it's the hook's discipline preventing accidental code-changes-in-SPEC-commits. The commit messages should explicitly note the split rationale.

WP-086's split: B-1 `688acdd` SPEC (8 doc-only files); B-2 `3369b87` EC-086: (httpRegistry.ts comment + apps/registry-viewer/CLAUDE.md table addition).

### 9.5 — Browser-WAF block on operator IP ≠ feature-broken (curl-UA vs real-browser)

**Lesson:** During R2 verification, both my Bash-tool curl AND the operator's PowerShell curl returned `403 Forbidden` from Cloudflare WAF on `images.barefootbetters.com` — same IP (`24.117.186.121`). Initial reading: "the operator's IP is blocked at zone level; the actual feature might be broken in their browser too." Actual reading: "Cloudflare bot-fight rules flag curl's user-agent specifically; real browsers (with full UA + JS) pass cleanly."

This was confirmed when the browser smoke test showed `[cardTypes] load complete` with `durationMs: 122 ms` — fetch worked perfectly in Chrome.

**Future precedent:** When verifying R2 publishes, don't conflate curl-UA WAF blocks with feature-broken status. If curl is blocked, fall back to one of:
- (a) `rclone cat <path>` against the S3 endpoint (bypasses CDN+WAF entirely)
- (b) Browser open + DevTools Network tab (passes WAF if real-browser)
- (c) Curl with browser UA: `curl -A "Mozilla/5.0 ..." <url>` (passes basic UA-block rules)

In WP-086's case, option (a) was used to verify body content; option (b) was used to verify headers + actual feature behavior.
