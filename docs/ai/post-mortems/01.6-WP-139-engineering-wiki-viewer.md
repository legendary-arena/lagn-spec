## 5. Representation & Determinism Audit

### 5a. Determinism (byte-identical output across consecutive builds)

```pwsh
$first = (Get-ChildItem apps/wiki-viewer/public -Recurse -Include *.html, *.css `
  | Sort-Object FullName | ForEach-Object { (Get-FileHash $_.FullName).Hash }) -join "`n"
pnpm wiki-viewer:build
$second = (Get-ChildItem apps/wiki-viewer/public -Recurse -Include *.html, *.css `
  | Sort-Object FullName | ForEach-Object { (Get-FileHash $_.FullName).Hash }) -join "`n"
$first -eq $second  # True
```

Equivalently `diff -r` between two consecutive build outputs returns zero
diff across 15 files (14 HTML + 1 CSS).

### 5b. Locked Hugo determinism knobs

In `apps/wiki-viewer/hugo.toml`:

- `enableGitInfo = false` — git-derived `lastmod` fields excluded from
  HTML (any HEAD move would otherwise flip the determinism check).
- `disableKinds = ["RSS", "sitemap", "taxonomy", "term"]` — RSS / sitemap
  outputs (which embed timestamps) and taxonomy outputs are off.
- `disableHugoGeneratorInject = true` — no Hugo-version meta tag.
- `markup.highlight.codeFences = false` — Chroma syntax highlighting
  disabled (see §5c for rationale).
- `markup.goldmark.parser.autoHeadingIDType = "github"` — GitHub-style
  heading anchors so internal `.md#anchor` links resolve.

### 5c. Determinism finding (caught at first determinism run, fixed before commit)

The initial Hugo configuration (with default Chroma syntax highlighting
enabled) failed the determinism check on the first run with two distinct
non-deterministic surfaces:

1. **Chroma inline CSS.** `<pre>` blocks emitted `-webkit-text-size-adjust:none`
   in one build and not the other. Reproducible across consecutive
   builds; not a build-cache artefact (running with `--cleanDestinationDir`
   reproduced the diff).
2. **Goldmark table cell rendering.** Table cells received
   `style=text-align:left` in one build and not the other.

Fix: set `markup.highlight.codeFences = false` (disable syntax
highlighting on fenced code blocks). The wiki content's ```ts``` blocks
now render as plain `<pre><code>` — no syntax colours, but no determinism
hazard. The verification step now passes consistently.

A secondary lock — `markup.highlight.noClasses = true` — was added so
that any future re-enabling of Chroma defaults to inline-style highlighting
(which is at least scoped to one CSS surface). A class-based highlighter
would also work but requires shipping a Chroma stylesheet, which is
deferred to a future WP if syntax colour ever becomes load-bearing.

### 5d. Projection-is-copy invariant

Pre-build `docs/wiki/INDEX.md` SHA-256:
`BE44B105BE05CC9EC71F18A71522BCAE61E5D323D0776B8AE4DA4091F40B5E3A`

Post-build hash: identical. The projection script uses `copyFileSync`
exclusively (no `renameSync`). The script's own existence-assertion
(`if (!existsSync(sourceIndex)) throw ...`) is a cheap regression guard
against a future `mv`-vs-`cp` drift.

## 11. Findings worth carrying forward

### Finding 1 — Hugo version pin is load-bearing for determinism

Verification Step 2 (byte-identical output across two consecutive builds)
is fundamentally a property of the Hugo binary used. Hugo's render output
has shifted between minor versions before; the version pin
(`apps/wiki-viewer/.hugo-version` = `0.135.0`) prevents silent drift and
is read by both the GHA workflow (`peaceiris/actions-hugo@v3` honours the
file) and the Render `static_site` build command (curls the matching
release archive).

A future WP that bumps the Hugo version MUST include a re-run of the
determinism check as part of acceptance — bumping `.hugo-version` is not
a free operation.

### Finding 2 — Hugo template gotcha: `dict` + `merge` + `append` for grouping

The first draft of `baseof.html` grouped sidebar entries by
`Params.type` using a hand-rolled accumulator with `dict` / `merge` /
`append`. Hugo rejected the template at first build with:

```
error calling append: expected a slice, got *hugolib.pageState
```

The root cause is subtle: when `index $byType $type` returns nil (no
entry yet for that type), `default slice` returns the function reference
to `slice` (a Hugo function), not an empty slice. `append` then fails
because its first argument isn't a slice.

The fix was switching to Hugo's first-class `Pages.GroupByParam "type"`
method, which returns `[]PageGroup{ Key, Pages }`. Idiomatic, deterministic
across consecutive builds, and works at the v0.135.0 pin.

Lesson: prefer Hugo's first-party grouping helpers over hand-rolled
accumulator patterns in templates. The hand-rolled form looks correct
in Go template syntax but trips on subtle nil-vs-zero-value semantics.

### Finding 3 — Verification Step 5's grep is over-broad

The WP-stated `Get-ChildItem ... -Include *.html ...` grep recursively
scans `apps/wiki-viewer/`, which includes the rendered `public/` output.
Wiki source content legitimately mentions internal package names in
prose (`docs/wiki/cardextid.md` discusses `@legendary-arena/registry`'s
boundary; the rendered HTML at `/wiki/cardextid/` carries the same
prose). On a literal reading of the grep, that surfaces as a match.

The substantive invariant — no runtime imports of internal packages
anywhere under `apps/wiki-viewer/` source / template / config — holds,
and a refined grep that excludes `public/` and `content/` returns zero
matches. Future EC revisions should narrow the grep accordingly. The
fix is documented in §4a above.

### Finding 4 — Path-mishap during initial authoring (caught + recovered)

The session ran inside a git worktree at
`C:\pcloud\BB\DEV\legendary-arena\.claude\worktrees\trusting-wiles-0a9aa4\`.
The first round of Write tool invocations targeted absolute paths under
`C:\pcloud\BB\DEV\legendary-arena\` (the parent repo) instead of the
worktree path. The mismatch surfaced when `pnpm wiki-viewer:project`
failed with `ERR_PNPM_NO_SCRIPT` because the worktree's `package.json`
was unchanged (the script add went to the parent repo).

Recovery: `mv apps/wiki-viewer/` and `mv .github/workflows/wiki-viewer.yml`
from parent → worktree, then `git checkout -- package.json render.yaml`
on the parent repo to revert the parent's tracked-file edits. The
worktree was then patched cleanly.

Lesson for future worktree-based sessions: always compose absolute file
paths from the worktree root, not the parent repo root. The system
prompt's "Primary working directory" line is the canonical anchor; copy
it verbatim into Write/Edit invocations rather than typing the
non-worktree path from memory.

### Finding 5 — Bundling `// why:` comments at file headers vs at use sites

The session prompt's six required `// why:` comments (anchors 1–6 in
§Required `// why:` Comments) all landed at file headers rather than
inline at specific code sites. This matches the spirit of the locked
rules (each anchor explains a per-file invariant — the projection
contract, the link-integrity rule, the render-hook posture, the
determinism knobs, the framework choice, the front-matter field surface
lock) and keeps the comments discoverable without scrolling.

The metadata-panel partial's `// why:` lives at the top of `single.html`
(the partial's call site) rather than inside `partials/metadata-panel.html`
itself, because the surface lock is enforced where the partial is
invoked — adding a new front-matter field requires adding a new `<dt>`
in the partial AND removing the surface-lock comment in `single.html`
to acknowledge the schema expansion. That coupling is more visible at
the call site.
