### Amendment 2 — Two-file scope expansion (`check-links.mjs` + `render-link.html`)

**What happened.** The EC-145 §Files to Produce list enumerated `wiki/SCHEMA.md` as the only file gaining a C1 amendment under the SCHEMA accommodation. During execution, when the inventory file was generated and the wiki-viewer build was re-run for the first verification pass, the `pnpm wiki-viewer:check-links` step failed:

```
Broken internal link: architecture-inventory.md → docs/ai/work-packets/WP-049-par-simulation-engine.md (resolved target: ...content/docs/ai/work-packets/WP-049-par-simulation-engine.md does not exist case-sensitively)
Link-integrity check failed with 1 broken internal link(s).
```

The inventory script emits repo-rooted paths (`docs/...`, `packages/...`, `apps/...`) for citing WP / DECISIONS / source files — those paths are valid from the repo root but **not** from the projected wiki tree where every wiki page is sibling-flat under `apps/wiki-viewer/content/`. Even if the link-integrity script were bypassed, the existing render hook (`apps/wiki-viewer/layouts/_default/_markup/render-link.html`) would have routed those links to broken `/docs/...` Hugo URLs because the render hook only special-cases `../`-prefixed links as GitHub-blob rewrites; bare-relative paths are treated as wiki-internal slugs.

**Resolution.** Surfaced as Option 1 / Option 2 / Option 3 to the user; user selected **Option 1** (modify both `check-links.mjs` and `render-link.html` minimally). Two changes:

- `apps/wiki-viewer/scripts/check-links.mjs`: early-continue at the per-file loop when `fileName === 'architecture-inventory.md'`, with a `// why:` block anchored to D-14503 explaining the SCHEMA exemption extends to link-integrity for this single file.
- `apps/wiki-viewer/layouts/_default/_markup/render-link.html`: add a `$isInventoryPage` flag (`eq .Page.File.LogicalName "architecture-inventory.md"`) and a third branch in the existing if/else-if chain that routes every non-external, non-anchor link from this page to the same GitHub-blob URL shape as `../`-prefixed links elsewhere. The inventory page's bare paths are absolute repo paths; the existing `path.Join "wiki"` is not applied.

**Codification.** D-14503's body now explicitly extends the SCHEMA exemption to cover internal-link resolution. WP-145 §Open Decisions C anticipated "the viewer's templates handle as a special case"; the WP did not anticipate the link-integrity script as a separate enforcement seam, but the exemption-extension is the natural broadening of the same pattern.

**Future-WP precedent.** When the SCHEMA exempts a generated file from front-matter / required-section conformance, the exemption should be evaluated against **all** wiki-viewer pipeline gates (link-integrity, render-link, project-wiki, hugo template selection), not just the SCHEMA itself. Generated files crossing the wiki source boundary surface integration gaps that hand-authored exempt files (`SCHEMA.md`, `README.md`, `INDEX.md`) don't expose because the latter happen to use wiki-relative link shapes by convention.

### Amendment 3 — Inventory regeneration after wiki-tree expansion

**Resolution.** Re-ran the script after creating the file to capture the steady-state output. The committed file now matches `node scripts/architecture-inventory.mjs --out -` exactly (modulo the midnight-UTC header straddle, which did not occur during the verification window).

**Future-WP precedent.** When a generated artifact is being added to a tree that the same generator scans, the generator must be re-invoked **after** the artifact lands so the report reflects steady-state. A first-pass invocation before the file exists will under-report by exactly one entry. EC-145 §Common Failure Smells implicitly covered this via the "committed bytes must match script's stdout exactly" verification — the verification step caught it.
