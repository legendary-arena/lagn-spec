## Section 1 — Overview

WP-115 completes the network-surface phase of the public leaderboard subsystem: the read library functions WP-054 cherry-picked to `main` earlier on 2026-05-01 (`getScenarioLeaderboard`, `getPublicScoreByReplayHash`, `listScenarioKeys`) are now reachable over HTTP at three locked paths:

- `GET /api/leaderboards/scenarios` — discoverability seam returning `{ scenarioKeys: string[] }`.
- `GET /api/leaderboards/scenarios/:scenarioKey` — paginated per-scenario leaderboard with `limit` (1–100, default 25) and `offset` (0–10000, default 0).
- `GET /api/leaderboards/scores/:replayHash` — single-score permalink lookup with `null → 404` translation.

The packet is wiring-only. Status-code domain is locked to `{200, 400, 404, 500}`; error envelopes are locked at `{ error: 'invalid_query', message: <sentence> }` / `{ error: 'score_not_found' }` / `{ error: 'internal_error' }`; `Cache-Control: no-store` is set as the **first statement** in every handler body so a thrown WP-054 exception still leaves the header set on the eventual 500 response. Path-parameter validation rejects missing `:scenarioKey` / `:replayHash` with 400 + the locked envelope (per Patch 4); the pagination parser rejects non-integer / out-of-range / array-of-length≠1 query values with 400 + the locked envelope (per Patch 5).

The packet also introduces the long-lived `pg.Pool` singleton at `apps/server/src/db/database.ts` — a new infrastructure seam that the deferred WP-102 profile route (D-10202) and every future request-handler WP will share. The Pool's lifetime is the process lifetime; close-on-SIGTERM is owned by `apps/server/src/index.mjs` (after the HTTP server's graceful-shutdown step resolves, never by a handler). Pool sizing values are locked at `max=10 / idleTimeoutMillis=30000 / connectionTimeoutMillis=5000` per D-11502; the locked Pool-construction log message text is `'[server] pg.Pool constructed (max=10)'` per D-11506 (verbatim — a future observability WP may grep this string to drive a Render log-based metric).

A test-only injection seam (`leaderboardLogic?` 4th parameter on `registerLeaderboardRoutes`) was introduced under WP-115 v1.1 reviewer Patch 3 to keep the test suite logic-pure: tests pass a fake `LeaderboardLogic` whose three methods return canned results, sidestepping any need to mock WP-054's SQL row shapes. This decouples the route-layer transport contract (status codes, envelopes, headers, pagination parsing) from the WP-054 SQL contract — future WP-054 internal SQL changes do not churn these tests.

Three triggers fire 01.6-mandatory authorship per WP-115 §Definition of Done:
1. **New long-lived abstraction** — the `pg.Pool` singleton, owned by the server process across its lifetime.
2. **New code seam** — `apps/server/src/db/`, parallel to `apps/server/src/par/`, `apps/server/src/rules/`, and `apps/server/src/leaderboards/`.
3. **New contract surface** — `LeaderboardLogic` interface (the test-only injection seam), exported from `leaderboard.routes.ts` for tests but not for production callers.

WP-102's profile route wiring stays deferred per D-10202 (reaffirmed under D-11505) even though the long-lived `pg.Pool` introduced here is exactly the lifecycle anchor that follow-up WP needs — the deferral is a scope discipline, not a technical blocker.

## Section 4 — What Surprised Us

### 4.1 The injection seam saved the test-suite from SQL-shape coupling

The v1.0 draft assumed tests could exercise the route layer "without a real DB" by mocking a `database.query(...)` shim that returned canned rows. Reviewer feedback (Patch 3, 2026-05-01) surfaced that this approach would couple the test suite to WP-054's SQL row shape (column names, type coercions, JOIN behavior) — every WP-054 internal SQL refactor would churn the route tests. The optional `leaderboardLogic?` 4th parameter on `registerLeaderboardRoutes` was introduced as a test-only injection seam: production callers omit it and the handler resolves to the imported WP-054 functions; tests pass a fake whose three methods return canned `ScenarioLeaderboard` / `PublicLeaderboardEntry` / `string[]` results.

The seam is contract-safe (optional parameter, default value is the imported binding, production callers continue to use the three-arg form unchanged) and removed all SQL-shape coupling from the route tests. None of the eight tests construct a `pg.Pool`, set `TEST_DATABASE_URL`, or import `pg` for runtime use.

### 4.2 First-statement Cache-Control discipline matters under exception paths

The v1.0 draft set `Cache-Control: no-store` after parsing `query.limit` / `query.offset`. Reviewer feedback (Patch 8) surfaced that this leaves a window where an exception inside the WP-054 helper would surface as a 500 with **no** `Cache-Control` header — a downstream cache (browser, CDN, intermediary proxy) might then cache the error response, turning a transient blip into a sticky outage visible to every user behind the same cache.

The first-statement discipline was locked into both the route file (the `koaContext.set('Cache-Control', 'no-store')` call is the first line of every handler body) and Test #8 (which asserts call-ordering across success + 400 + 500 paths on a mock context that records writes). Future maintainers that want to reorder will hit Test #8.

### 4.3 Two `new Pool(` matches under the static-grep, not one

The pre-existing rules loader at `apps/server/src/rules/loader.mjs:114` constructs its own Pool for the one-shot rules-cache load at startup, then tears it down via `finally { await pool.end(); }` before request traffic begins. The WP-115 EC §After Completing gate text reads "exactly one match (in `db/database.ts`)" — strictly interpreted, the rules-loader match would fail the gate. The gate's intent is "exactly one long-lived request-time Pool," which is satisfied: only `database.ts` constructs a Pool that survives past startup.

This divergence between the literal grep result and the gate text was documented in §2.4 of this post-mortem so a future audit can match the on-disk state to the intent. A future hardening WP may consolidate the rules loader to use the long-lived Pool, removing the second `new Pool(` site; that is out of scope here per WP-115 §Out of Scope (no rules-loader refactor).

### 4.4 Path-parameter validation belongs in the route layer, not WP-054

The v1.0 draft assumed missing `:scenarioKey` would surface from WP-054 as `null` PAR (empty leaderboard, 200) or a SQL error (500). Reviewer feedback (Patch 4) surfaced that routing a missing or empty path parameter to the WP-054 helper would surface as a 500 from the SQL layer instead of the locked 400 envelope — a transport-level shape error (bad request) being masked as an infrastructure error (server fault). Validation was relocated to the route layer (the first thing after `Cache-Control` is set) so the locked 400 envelope (`{ error: 'invalid_query', message: 'Scenario key is required.' }` / `'Replay hash is required.'`) is the only way these conditions surface.

This separation matches the pagination-parser pattern: query-string parsing lives in the route layer, not in WP-054.
