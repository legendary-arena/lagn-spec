## New Patterns Discovered

### Per-copy fan-out as a key-set proliferation precedent for sibling-snapshot maps

WP-018 (`G.cardStats`) and WP-111 (`G.cardDisplayData`) established the sibling-snapshot pattern of building ext_id-keyed lookup maps once at `Game.setup()`. WP-137 extends that pattern with **per-copy fan-out** — each `(hero, card)` pair now contributes N entries (one per copy) instead of one. The locked-author choice for future sibling-snapshot maps that need per-copy fan-out is:

1. The canonical emitter (`buildHeroDeckCards`) owns the suffixing logic and any throw surface.
2. Fan-out callers import a shared resolution helper from the canonical emitter file (RS-4 lock — `resolveHeroCardCopyCount` + `buildCardCountsNameLookup` exported from `setup/buildHeroDeck.ts`).
3. Fan-out callers treat `null` from the resolver as silent fall-through; only the canonical emitter throws.
4. Per-copy entries use **fresh object literals** (no aliasing across keys) so each copy is mutation-isolated (WP-028 D-2802 aliasing prevention extended to setup-time fan-out).

This is now the precedent for any future engine WP that needs to project per-copy state into a sibling-snapshot map.

### Shared exported helper from one of three sites as the cross-file consistency precedent

The pre-flight RS-4 resolution chose **shared exported helper** over byte-for-byte duplication. The locked rationale (WP §Non-Negotiable Constraints) cites:

- Patch locality is improved (a future bug fix in count resolution touches one file, not three).
- The grep gate becomes a positive assertion ("each of the three sites imports `resolveHeroCardCopyCount`") rather than a divergence-detection assertion.
- The `RegistryReader` precedent (already shared between `buildHeroDeck.ts`, `economy.logic.ts`, and `buildCardDisplayData.ts`) shows the pattern works.

This is now the default pattern for cross-file consistency inside the engine package when three or more sites need identical per-card resolution logic. Byte-for-byte duplication remains a viable choice when a positive grep gate is impractical (e.g., the duplicated logic is tightly coupled to surrounding state in each site).

### Additive optional registry-schema field with verbatim name-vs-slug sentence

`HeroSchema.cardCounts: z.record(z.string(), z.number().int().min(1)).nullable().optional()` is the additive field landed under D-13701. The schema-comment block above `HeroSchema` carries the verbatim sentence:

> *"`cardCounts` keys are card display names from the upstream dataset; the engine resolves them against `cards[].name` and emits ext_ids using `cards[].slug`."*

This is the precedent for any future `HeroSchema` extension that introduces a name-vs-slug or display-vs-canonical asymmetry: lock a one-sentence verbatim comment at the schema definition site so the contract is clear from the registry, not from engine implementation. The `// why:` comment cites the originating D-decision (D-13701).

### First migration registered in `migrationRegistry` as the precedent for future replay-format migrations

`migrateHeroExtIdsForCopyIndex(payload: unknown): unknown` is the first entry in `migrationRegistry` (previously `Object.freeze({})`). The locked contract:

- **Pure-transform**: never throws. The throw surface is owned by `migrateArtifact` (the load-boundary exception per the existing comment at `versioning.migrate.ts:31-39`).
- **Best-effort schema compatibility**: prevents `#`-absent crashes in legacy ext_ids by appending `#0`. Does NOT guarantee semantic equivalence — pre-WP-137 replays are expected to fail re-verification because shuffle order shifts under the new copy-index convention (per D-0802 fail-loud at the load boundary).
- **Aliasing prevention**: returns a new payload reference (WP-028 D-2802 extended); the recursive rewriter (`rewriteArgsValue`) returns new arrays/objects whenever any nested value is rewritten.
- **Narrow grammar matcher**: the three-predicate gate (exactly two `/`, no `#`, regex `^[a-z0-9-]+/[a-z0-9-]+/[a-z0-9-]+$`) prevents accidental mutation of villain / mastermind / henchman / scheme hyphen-form ext_ids.

Future migration WPs should follow this template: register at the engine-version-bump key, never throw inside the `MigrationFn`, return a new payload reference, use a narrow matcher to avoid false matches against other ext_id grammars.

## Anti-Patterns Discovered

### Reverse the lookup direction (`cardCounts[card.slug]` always wrong)

The data ships keyed by display name (NOT slug). A reflex `cardCounts[card.slug]` lookup misses everywhere except cards whose display name happens to equal their slug, and the bug is masked in fixtures whose rarity-map values happen to match cardCounts. The grep gate `cardCounts\[\w+\.slug\]` catches this; the only legitimate match in the production code is the documentary `// why:` comment in `buildHeroDeck.ts` explaining why NOT to do it.

### Throw inside `migrateHeroExtIdsForCopyIndex`

The `MigrationFn` signature is `(payload: unknown) => unknown` — pure-transform contract. Throwing inside the function would bypass `migrateArtifact`'s structured error envelope (which names both versions and the artifact's `savedAt` timestamp) and crash callers without operator-actionable context. The locked discipline: shape mismatch returns the input unchanged; only `migrateArtifact` throws.

### Aliasing the per-copy entry across keys (`stats[extId#0] === stats[extId#1]`)

Initial implementation built one `entry` literal per `(hero, card)` pair and assigned it to N keys via `stats[extId#i] = entry`. This shared the reference across all per-copy keys; downstream mutation of any key would mutate all. Fixed by hoisting the scalar values (`attack`, `recruit`, `cost`) and constructing a fresh object literal per copy. WP-028 D-2802 aliasing prevention extends to setup-time sibling-snapshot fan-out.

### "Fix" the captain-america cardCounts in the data layer

`data/cards/core.json/captain-america.cardCounts` ships sum=20 (anomalous) but the upstream input `scripts/convert-cards/inputs/hero-card-counts.json` says sum=14 (canonical). The temptation during execution was to "fix" the data file. WP-137 §Out of Scope + §Known Data Anomaly RS-1 explicitly prohibit modifications to `data/cards/*.json` or `scripts/convert-cards/**`. The pipeline drift is a separate spawned task; WP-137 ships with the bake-in.

## Lessons for Future Migration WPs

1. **`MigrationFn` is pure-transform.** Never throw inside a registered migration. The `migrateArtifact` wrapper is the throw surface.
2. **Engine + data axis bumps in lockstep.** When the wire shape of any persisted artifact changes, the engine reducer behavior also changed; both axes bump together. The `package.json:version` mirror obligation is enforced by the comment at `versioning.check.ts:22-28`.
3. **Best-effort vs strict semantic equivalence (per D-0802).** Migrations that prevent crashes by reshaping legacy data DO NOT guarantee semantic equivalence. Document the guarantee (or lack thereof) in the migration's `// why:` comment so future operators reading old replays understand they're historical artifacts, not re-verifiable runs.
4. **Aliasing prevention extends to load-boundary wrappers.** Returning a new payload object from the migration prevents downstream mutation from leaking back into stored artifacts (per WP-028 D-2802 extended).
5. **Narrow grammar matchers.** Migrations that rewrite ext_id strings should use a tightly-scoped predicate (e.g., the three-predicate gate in WP-137: exactly two `/`, no `#`, regex match) to avoid false matches against other ext_id grammars (villain hyphen-form, mastermind hyphen-form, etc.).
6. **First-migration testing template.** The four `versioning.test.ts` tests added by WP-137 (registry-presence, rewrite-correctness, leaves-other-grammars-untouched, returns-input-unchanged-on-shape-mismatch, never-throws, returns-new-reference) form the testing template for every future registered migration.
