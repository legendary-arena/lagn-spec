## §Vite `onwarn` Regression Probe

Two probe variants executed in scratch state (both reverted before commit):

**Variant A — literal session-prompt §13 step 9 shape** (`import { readFile } from 'node:fs/promises'` with no binding usage):
- Result: build SUCCEEDED with exit 0 and **zero warnings** in the log.
- Cause: `"sideEffects": false` on the engine package causes Rollup to tree-shake the unused import entirely before the resolver runs; the import never reaches the `onwarn` handler.
- Conclusion: this shape is silently safe under the new contract — the structural enforcement is even stronger than the session prompt anticipated. The handler doesn't fire because there's nothing for it to fire on.

**Variant B — binding-used variant** (`import { readFile as __probe } from 'node:fs/promises'; export const __holder = __probe;`):
- Result: build FAILED with **exit 1**.
- Error message: `"readFile" is not exported by "__vite-browser-external"`.
- Mechanism: Rollup raises a hard error during binding resolution because the externalized `node:*` stub doesn't expose the named binding. The error short-circuits the build before the `onwarn` handler executes; the structural enforcement is satisfied either way.

Conclusion: the `onwarn` handler IS the documented enforcement layer, but the actual hard-fail mechanism for the binding-used case is Rollup's resolver, not the handler. The handler remains in place as a backstop for shapes that produce warnings rather than errors (e.g., a Vite plugin emitting a warning containing `__vite-browser-external`).

## §TS Path-Guard Regression Probe

Probe injected at `apps/arena-client/src/main.ts`: `import { loadParIndex } from '@legendary-arena/game-engine/setup'; console.log(typeof loadParIndex);` (binding-used).

**Result:** typecheck reported the same 41 pre-existing errors; **zero new errors** attributable to the probe. The path guard at `apps/arena-client/tsconfig.json` `compilerOptions.paths` was not honored by vue-tsc 2.x.

**Diagnosis:** vue-tsc 2.x with `moduleResolution: Bundler` resolves `@legendary-arena/game-engine/setup` via the engine package's `exports` field directly, bypassing the `paths` mapping. This is a documented vue-tsc limitation when a scoped package has an `exports` map that resolves the subpath natively.

**Mitigation:** the path guard remains installed (it is one of three documented enforcement layers and is harmless; it may catch other shapes under different toolchains). The verify-time grep gate `grep -rn "@legendary-arena/game-engine/setup" apps/arena-client/` is the working backstop and is enforced as part of WP-144 §11 verification steps. Going forward, any arena-client commit that adds a `/setup` import will be caught by the grep gate at code review or by verifying the build log's structural enforcement (Vite `onwarn` + the runtime-purity grep gate on engine source).

D-14401 clause 5 documents this limitation explicitly: "Known limitation: vue-tsc 2.x with `moduleResolution: Bundler` resolves `@legendary-arena/game-engine/setup` via the package `exports` field, bypassing `paths` mapping; the path guard is documented best-effort, with the verify-time grep gate as the backstop."

## §Forward Signal

D-14401 establishes precedent for future workspace packages that need to split their public surface by audience:

1. **Subpath naming:** use a single descriptive subpath name (`./setup`, `./server`, etc.) per audience; avoid wildcards.
2. **`exports` map shape:** include `types` + `import` + `default` conditions per subpath for tooling compatibility (Vite, Vitest, downstream consumers that probe `default`); export `./package.json` so introspection tools can read it.
3. **`"sideEffects": false`:** add ONLY after the binary six-item audit passes; document the audit result + module enumeration in the corresponding D-NNNNN.
4. **Closed-list quarantine:** if any audience contains Node-IO modules, codify the directory list in the decision and require future modules to be authored there.
5. **Three-layer structural enforcement:** for any audience that must NEVER reach a forbidden surface, install (a) the subpath split, (b) a build-time hard-fail (Vite `onwarn` for browser bundles, equivalent for other build systems), AND (c) a compile-time guard (TS path mapping or declaration override). The verify-time grep gate is the backstop for any layer that fails for toolchain-specific reasons.

The WP-144 surface area + commit topology (single Commit A for production/config, single Commit B for governance close) is reusable for future similar refactors.
