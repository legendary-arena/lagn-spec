## Section 2 ‚Äî What Worked Well

- **Single-template-constant discipline for the WP-093 error.** `UNSUPPORTED_HERO_SELECTION_MODE_TEMPLATE` in `setupContract.types.ts` is the only source of the WP-093 error-message template. Step 1 (zod-error upgrade) and Step 1b (defensive fallback) both reference it, so byte-for-byte equality with D-9301 is guaranteed regardless of which detector branch fires. Tests assert exact-string equality via `assert.strictEqual`; `assert.match` / `.includes()` / substring checks are absent from the suite.
- **Engine-identical lookup algorithm.** `buildKnownExtIds` logic is replicated byte-for-byte: `registry.listCards()` once ‚Üí `new Set<string>()` + `for...of` (no `.reduce()`) ‚Üí `Set.has()` for each composition ext_id surface. Because the registry-side validator and the engine-side `validateMatchSetup()` share the algorithm structurally, a document accepted registry-side is accepted engine-side by construction.
- **Single-site default normalization.** `heroSelectionMode: undefined ‚Üí "GROUP_STANDARD"` happens in exactly one place (Step 3 inside `validateMatchSetupDocument()`). The composable reads `draft.heroSelectionMode` directly and the builder seeds it to `"GROUP_STANDARD"` explicitly at draft creation so the absent-default is never relied on for authoring output (L4). This keeps the default-handling layer unambiguous and unpunishable by duplicate `?? "GROUP_STANDARD"` fallbacks.
- **Spread-copy discipline in `prefillFromTheme`.** Each array field (`villainGroupIds`, `henchmanGroupIds`, `heroDeckIds`) is spread-copied before assignment to the draft (L10). The `// why:` comment cites the WP-028 projection-aliasing post-mortem precedent so future readers understand this is a deliberate correctness guardrail, not boilerplate.
- **Drift-detection test uses compile-time + runtime assertion.** The 9-field composition lock is enforced two ways: a `AssertEqual<CompositionFieldName, keyof SetupCompositionInput>` type-level assertion (catches any structural drift at build time) and a runtime `assert.deepStrictEqual` on the tuple order (catches any reordering that still type-checks). No engine import is required ‚Äî the engine side is governed by its own drift-detection test, and the registry-side mirror governs itself.
- **Narrow-subpath import discipline for the viewer.** `useLoadoutDraft.ts` and `LoadoutBuilder.vue` both import from `@legendary-arena/registry/setupContract` (the new subpath export), not the root `@legendary-arena/registry` barrel. Same mitigation precedent as `themeClient.ts` using `./schema` + `./theme.schema` for node:fs-free imports. The viewer's Vite production build completes without any `node:fs/promises` externalization errors.

## Section 3 ‚Äî What Surprised Us

- **The root barrel pulled in `node:fs/promises` via `localRegistry.ts`.** Initial drafts of `useLoadoutDraft.ts` imported from `@legendary-arena/registry` (the root barrel), which re-exports `createRegistryFromLocalFiles` (node-only). Vite's production build failed at rollup with `Cannot resolve '__vite-browser-external'`. The fix was to add a dedicated `./setupContract` subpath export to `packages/registry/package.json` mirroring the `./schema` / `./theme.schema` pattern already used by `themeClient.ts`. This surfaced late because the registry package's own build (tsc) happily compiles the barrel ‚Äî the failure only appears downstream in the viewer's rollup graph.
- **`issue.received` typing on `invalid_enum_value`.** TypeScript's zod issue types declare `issue.received` as `string | number` on enum issues, which triggered a TS2769 overload-mismatch when it was passed directly into `String.prototype.replace(...)` as the substitution argument. Resolved by coercing through `String(rawValue ?? "")` on the explicit raw-input extraction path (so the replacement is always a plain string) rather than relying on zod's `issue.received` field as the substitution source.
- **`noUncheckedIndexedAccess: true` in `packages/registry/tsconfig.json`.** The `checkArrayExtIds` helper's `values[index]` expression is typed as `string | undefined` under this flag. Added an explicit `if (extId === undefined) continue;` guard so the `Set.has()` call receives a narrowed `string`. No change to runtime behavior ‚Äî arrays are never sparse after zod parsing ‚Äî but the explicit guard keeps the code honest about the declared input type.

## Section 5 ‚Äî Precedent to Establish

- **D-1209 pattern extended from engine-side to registry-side.** The `CardRegistryReader` minimal structural interface was originally declared engine-side to avoid `packages/game-engine/**` importing `@legendary-arena/registry`. WP-091 applies the same pattern in reverse: a registry-side copy of the interface avoids `packages/registry/**` (or `apps/registry-viewer/**`) importing `@legendary-arena/game-engine`. The real `CardRegistry` satisfies both copies structurally via `FlatCard.key`. Future registry-side mirrors of engine contracts should follow this same pattern.
- **Zod-upgrade-detector defensive fallback as a reusable pattern.** Any validator that upgrades a framework library's default error message to a governed template should carry a raw-input fallback (deduplicated by `(code, field)` pair) so the upgrade still fires if the framework's issue shape drifts between versions. WP-091's Step 1b is the first application; WP-092's shape guard and any future zod-based validator in the repo should use the same pattern.
- **Spread-copy discipline in composable `prefill*` methods.** Any composable that pre-fills draft state from a registry-loaded singleton must spread-copy every array field before assignment. Direct assignment aliases the singleton and lets subsequent draft mutations corrupt every consumer holding that reference. WP-028 was the original precedent; WP-091 is the second application. Future composables (WP-092 lobby intake, any future authoring surface) should honor the same rule.
- **Single-site default normalization at the validator boundary.** When a contract document has optional fields with backward-compat defaults, normalize at exactly one place ‚Äî the boundary where the structured validator returns its result ‚Äî so the UI / composable / serializer layers can read the field as always-set. Duplicate `?? DEFAULT` fallbacks elsewhere mask validator bugs and create drift between layers.
- **Browser-safe subpath pattern for new registry submodules.** Any new registry submodule that needs to be consumed by `apps/registry-viewer` (or any future browser-only app) should ship a dedicated `./<name>` subpath export alongside a `<name>/index.ts` barrel, so the viewer never imports the root barrel (which unavoidably re-exports the node-only `localRegistry` factory). `./schema`, `./theme.schema`, and `./setupContract` now form the precedent.

## Section 6 ‚Äî Lessons Learned (Feed Forward to 01.4 ¬ßEstablished Patterns)

- **`packages/<pkg>/package.json` `exports` is part of scope on any new consumer-surface WP.** The scope lock in the session invocation didn't explicitly list `packages/registry/package.json` as a modifiable file; the new `./setupContract` subpath entry was added as a necessary glue for the browser build to work. Future WPs that ship a new registry submodule should anticipate a `package.json` `exports` edit in the scope inventory from the start, not as a discovered necessity mid-execution.
- **Byte-for-byte verification greps must cover UTF-8 multi-byte characters.** The session's verification Step 9 uses a regex `.` in PowerShell `Select-String` to match the em-dash between "GROUP_STANDARD" and "Classic Legendary hero groups". In bash `grep`, a single `.` matches one byte, not the three-byte UTF-8 em-dash, so the same check required either `..` (lucky match) or the literal em-dash character. Cross-shell verification scripts should use the literal character or `rg` (which is UTF-8-aware by default) rather than counting on `.` wildcard behavior.
- **Pre-existing `localStorage` usage in unrelated viewer files (e.g., `useCardViewMode.ts`) makes the EC-091 ¬ßStep 5 "no output" expectation ambiguous.** The intent is "no new `localStorage` usage in WP-091 files", but the grep as specified searches the entire `apps\registry-viewer\src` tree and returns pre-existing hits. Future EC-authoring should either (a) scope the grep to the new file set, (b) allow-list pre-existing patterns, or (c) explicitly document the expected non-zero baseline.
- **The zod-upgrade-detector fallback is cheap insurance.** The extra ~20 lines in `validateMatchSetupDocument.ts` cost nothing at runtime for valid documents and protect against a class of failure (framework minor-version drift silently dropping the error upgrade) that would be invisible to every unit test unless a specific compound-failure test case exists. Future validators that upgrade framework errors should include both the upgrade path and the fallback.
