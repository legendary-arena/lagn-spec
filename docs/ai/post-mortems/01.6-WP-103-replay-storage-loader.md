## Section 1 — Overview

WP-103 lands the smallest viable predecessor that closes the WP-053 replay-loader prerequisite. It introduces two functions — `storeReplay` and `loadReplay` — backed by a new content-addressed PostgreSQL table `legendary.replay_blobs`, exposing a clean `Promise<void>` / `Promise<ReplayInput | null>` interface that future request handlers (WP-053 submission, WP-054 leaderboards, any replay-by-hash retrieval surface) consume without re-deriving SQL or cracking the storage layer open.

The packet is deliberately narrow: four files in Commit A — three TypeScript files (`replay.types.ts`, `replay.logic.ts`, `replay.logic.test.ts`) under `apps/server/src/replay/` and one SQL migration (`006_create_replay_blobs_table.sql`). None of the new functions are wired into `server.mjs`, `boardgame.io`'s lifecycle, any move, or any phase hook — a deliberate lifecycle prohibition recorded in WP-103's session prompt. The library is established; consumer wiring is the next packet's responsibility.

The schema diverges from WP-052's `bigserial + ext_id text UNIQUE` precedent: the `replay_hash text` IS the natural key, so the PK is `replay_hash` directly. No separate internal bigint identifier exists, and there is no use case for one — replays are content-addressed by hash. The blob column is `jsonb` (not `bytea` / `text` / `json`) for shape queryability and storage efficiency. There is no row-mutation timestamp by design (content-addressed rows are immutable; mutation would invalidate the canonical identifier) and no FK to `legendary.replay_ownership` (WP-052's contract is locked; application logic in WP-053 will ensure `storeReplay` precedes `assignReplayOwnership` on the submission path).

### 2.4 Aliasing audit

`pg`'s `jsonb` codec deserializes column values at the driver level into a fresh JavaScript object on every query. The implementation `result.rows[0].replay_input as ReplayInput` returns whatever fresh object the driver produced — no shared reference is materialized between calls, no in-memory cache is consulted, and no mutable source object is exposed by the function surface. The aliasing risk that drove D-2801 / WP-028 (engine projection aliasing) does not apply here: server-layer DB queries always materialize fresh JavaScript values.

The application contract is that `legendary.replay_blobs` only ever contains `ReplayInput`-shaped rows — enforced at write time by the `storeReplay` parameter type (`replayInput: ReplayInput`), and at the schema level by the `jsonb NOT NULL` column type. There is no other write path that could insert a non-`ReplayInput` shape.

The locked SQL pattern from EC-111 §Locked Values is used verbatim. `DO NOTHING` is the correct semantic for content-addressed immutability: a SHA-256 collision producing different content for the same hash is statistically infeasible and would be a cryptographic anomaly, not expected behavior. Observing two distinct `replay_input` values for the same `replay_hash` should be treated as a security incident, not a data-update opportunity. ## Section 3 — Lessons & Carry-Forward

### 3.1 Hard-Stop greps match comment text — reword instead of deleting comments

The session prompt's Hard Stops §11–§13 use simple regex matches against the entire file (e.g., `grep -nE "ON CONFLICT.*DO UPDATE"`, `grep -nE "JSON\.parse"`, `grep -nE "updated_at"`). My initial draft of `replay.logic.ts` and the migration carried `// why:` / `-- why:` comments that explained *why those forbidden patterns were avoided* — and the comments themselves contained the literal forbidden substrings. The grep gates fired on the comment text, not on actual code or DDL.

The reconciliation was to reword the comments without losing intent: "(not DO UPDATE)" became "(never the update-on-conflict alternative)"; "manual `JSON.parse`" became "a manual JSON-parsing call"; "no `updated_at` column" became "no row-modification timestamp column". Future packets with similar literal-substring grep gates should pre-screen `// why:` text the same way — the gate is mechanical, not semantic, and helpful explanatory phrases can trip it. Worth recording so the next executor doesn't waste a round-trip rediscovering this.

### 3.2 Skip-pattern gate is *exactly 4*, not "≥ 4"

The session prompt's Hard Stop §16 is asymmetrically stricter than the same gate in §Verification Steps: the verification step says "at least 4 lines (one per DB-dependent test)" while the Hard Stop says "does NOT match exactly 4 lines". My initial draft had a JSDoc reference to the literal `skip: 'requires test database'` substring — bringing the count to 5 — which would have tripped Hard Stop §16 even though the verification gate was satisfied.

The fix was a one-line JSDoc rewording: `` `{ skip: 'requires test database' }` `` → "non-silent skip option". Future executors writing test files under this contract should keep the literal substring confined to the four test definition sites. WP-052 §3.1 already flagged this asymmetry; recording it again with the F-3 fixture-isolation finding folded in.

### 3.3 `Promise<void>` vs `Result<T>` — content-addressed surfaces don't need wrappers

WP-052 used `Result<T>` for fallible identity operations (`createPlayerAccount` can fail with `duplicate_email`, `assignReplayOwnership` can fail with `unknown_account`). WP-103 deliberately diverges: `storeReplay` returns `Promise<void>` and `loadReplay` returns `Promise<ReplayInput | null>`, with no `Result<T>` wrapper on either surface. The reason is structural — there are no expected application-side failure modes:

- `storeReplay`: idempotent insert always succeeds, or throws on infra error
- `loadReplay`: returns `null` on miss (the only documented "miss" signal) or the deserialized `ReplayInput` on hit; never throws on absence

Infra-level failures (connection lost, permission denied, malformed SQL) propagate via thrown exceptions for the caller (a future request handler) to translate into HTTP responses or operator alerts. No `Result<T>` branch would carry useful information beyond what an exception already carries — adding the wrapper would be ceremonial cost without payoff. Recording this so future server-layer storage WPs don't reach for `Result<T>` reflexively when the underlying surface has no application-side failure modes.

### 3.4 Test fixture isolation — F-3 guardrail

WP-103 §F-3 lock prohibited importing fixtures from `apps/replay-producer/samples/` (or anywhere outside `apps/server/src/replay/`) into the test file. The minimal `ReplayInput` covering all four fields fits in ≤ 20 lines of TypeScript inline literal — the inline form is preferable for both layer-boundary hygiene (no cross-app coupling) and locality of reasoning (the fixture is right there in the test file, not in a remote producer sample). Future server-side WPs that touch `ReplayInput` shapes should follow the same pattern unless they have a structural reason to consume a real producer artifact (e.g., a deserialization-format-compatibility test).

### 3.5 `.ts` extension carries forward — `node --import tsx --test` is the runtime

Per WP-052's precedent, the new replay files are `.ts` rather than `.mjs`. They load cleanly under `apps/server`'s test script (`node --import tsx --test scripts/**/*.test.ts src/**/*.test.ts`), but plain `node` (used by `server.mjs`'s runtime path) cannot import them directly. WP-103 does not wire `replay.logic.ts` into `server.mjs`, so this is invisible at execution time. Any future WP that wires `replay.logic` into `server.mjs` (or any other plain-`node`-loaded entrypoint) will need to either add `tsx` to the production runtime, add a build step, or convert these files to `.mjs`. WP-103 does not solve that future problem.
