## 6. Lessons Learned

**New pattern — two-source projection with an orthogonal coherence flag.** `useTriageStatus` is the first dashboard composable that joins two independent endpoints. The sweep composables each project ONE resource; triage projects two whose freshness can legitimately diverge. The lesson the WP-235/WP-238 pattern did not cover: when two sources are fed by separate pipelines, don't overload the 4-arm fetch state machine with cross-source coherence — keep `error` for fetch failures and add an orthogonal `coherence` flag that degrades gracefully. Folding the skew into `error` (the original audit reviewer's instinct) would blank a freshly-submitted report's findings during the routine inspection→handoff sync lag.

**Recurrence — the layer-boundary grep self-trips on documentation prose.** WP-239's close-out grep for `apps/server` / `@legendary-arena/game-engine` expects 0, but a file that *documents* "must not import apps/server (layer boundary)" trips it. Same class as the WP-226/233/236 comment-vs-grep self-trips already in user memory (`feedback_grep_gate_comment_self_trip`). Resolved by paraphrasing the policed tokens in `triage.ts` and keeping the exact server paths only in the drift test's reconcile messages (out of grep scope). The `sweep.ts` precedent freely names `apps/server` in its header — the discipline (paraphrase in graded files) is what keeps the gate honest.

**Projection types belong with the consumer, not the producer.** `TriageProjection` holds `PipelineItem[]`, so it must live where `PipelineItem` lives (`useAgentPipeline.ts`), exactly as `PipelineSweepData` does — even though `useTriageStatus` is the producer. Declaring it with the producer creates a circular type import. Caught by the copilot check, not by typecheck (TS tolerates circular *type* imports), which is why the preventative review earned its keep.
