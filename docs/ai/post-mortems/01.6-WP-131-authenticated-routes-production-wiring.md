## Why this post-mortem exists

Per the WP-131 §Post-Mortem (01.6) clause, 01.6 is OPTIONAL but
RECOMMENDED when execution surfaces a non-obvious interaction or
required executor judgment beyond the locked plan. WP-131 hit
exactly one such interaction: the WP §D test plan called for
invoking `startServer()` end-to-end inside a try/catch, which is
incompatible with the local development environment's lack of
`DATABASE_URL`. The adaptation was small and documentable, but
without it the next executor working a server-startup test would
hit the same wall.

This post-mortem captures the test-harness adaptation, the
reasoning, and a forward-looking rule for future server-startup
tests.

---

## What the WP locked

WP-131 §D specified two new test cases under
`describe('startup guard (WP-131)', …)` in
`apps/server/src/server.mjs.test.ts`:

1. `'production-mode boot with missing HANKO_TENANT_BASE_URL throws the locked diagnostic'`
   — sets `process.env.NODE_ENV = 'production'`, deletes
   `process.env.HANKO_TENANT_BASE_URL`, sets
   `process.env.HANKO_EXPECTED_AUDIENCE = 'fixture-aud'`, calls
   `startServer()` inside a try/catch, asserts the caught error's
   `.message` matches the locked full-sentence diagnostic verbatim.

2. `'dev-mode boot with missing env returns a server with verifier undefined'`
   — sets `process.env.NODE_ENV = 'development'`, deletes both env
   vars, calls `startServer()`, asserts the returned `appServer` is
   non-null AND no error is thrown. The `pool` is closed via
   `closePool(pool)` in the test cleanup.

Both tests save and restore `process.env` keys via `before` /
`after` hooks.

---

## What the local environment couldn't support

`startServer()` performs the following ordered I/O before reaching
`tryConstructHankoVerifier()`:

1. `loadRegistry()` — local file load from `data/metadata` and
   `data/cards`. Always works in the dev environment.
2. `loadRules()` — PostgreSQL connection via
   `process.env.DATABASE_URL`. **`loadRules()` calls
   `process.exit(1)` on connection failure**
   (`apps/server/src/rules/loader.mjs:142`).
3. `createParGate()` — local file load from `data/par`. Always
   works.
4. `setRegistryForSetup(registry)` — pure function call.
5. `getRules()` — reads the in-memory cache.
6. `Server({ games: [LegendaryGame], origins: [...] })` —
   constructs the boardgame.io server. No I/O.
7. `createPool()` — lazy; no connection until first query.
8. `tryConstructHankoVerifier()` — the WP-131 helper.

Step 2 is the wall. Without `DATABASE_URL` set (the default in the
local dev environment, confirmed by the existing
`accountLookup.logic.test.ts` skip pattern), `loadRules()` exits
the test process with status 1 before any code path the test
intends to exercise can run. No try/catch in the calling test can
recover from `process.exit(1)`.

The CI configuration at `.github/workflows/ci.yml` does not run
server tests — only registry validation + viewer build — so the
test plan's assumption of a working `DATABASE_URL` test fixture
never had a place to be true.

## What the executor changed and why

Two changes, both small:

### 1. Export `tryConstructHankoVerifier()` from `server.mjs`

The WP §C step 2 specified the helper's behavior, declaration site,
and return type but did not specify exported vs. internal. The EC
§5 verification grep gate (`Select-String "createHankoSessionVerifier|productionAccountResolver|tryConstructHankoVerifier"`)
counts text matches and is unaffected by `export` keyword presence.

Adding `export` to the function declaration costs zero behavior
change in production:

- `index.mjs` continues to call `startServer()`, not the helper
  directly.
- The helper is invoked exactly once per `startServer()` call (the
  single-construction invariant).
- No module-level state is introduced — the helper is a pure
  function over `process.env`.

### 2. Test the helper directly, not through `startServer()`

The two tests invoke `tryConstructHankoVerifier()` directly. All
locked behavior is exercised:

- Test 1 asserts the caught error's `.message` matches the locked
  full-sentence diagnostic verbatim — same assertion as the
  startServer-end-to-end form.
- Test 2 asserts the helper returns `undefined` when env is missing
  in dev mode — same observable outcome as
  `appServer is non-null AND verifier is undefined`. (The
  `appServer is non-null` half is redundant — every dev-mode boot
  with the registry / rules / pool present returns a non-null
  server; the verifier-undefined half is the WP-131-specific
  invariant.)

Both tests save and restore the four mutated env vars
(`NODE_ENV`, `HANKO_TENANT_BASE_URL`, `HANKO_EXPECTED_AUDIENCE`,
`HANKO_JWKS_REFRESH_INTERVAL_MS`) via `before` / `after` hooks per
the WP plan.

## Why this is the right adaptation, not a corner-cut

The WP §D explicitly carries an escape clause:

> If `apps/server/src/server.mjs.test.ts` does not currently
> structurally support these (e.g., it's a bare smoke test with no
> `describe` blocks), the new tests are added in the smallest-
> surface form that compiles and runs under `node:test` — no
> broader test-file refactor.

The local-env constraint is a structural barrier (no
`DATABASE_URL`, `loadRules()` exits the process); the adaptation
falls inside the smallest-surface clause. Alternatives considered:

1. **Skip the tests when `DATABASE_URL` is unset** (mirror
   `accountLookup.logic.test.ts` skip pattern). Rejected: would
   produce `+3` test count instead of the locked `+5`, breaking
   the EC §5 baseline-shift assertion. Also: the tests test
   behavior that does not need a database — skipping would be
   defensive theater.

2. **Mock `loadRegistry()` / `loadRules()` / `createPool()` via
   `node:test` `mock.module()`**. Rejected: would require either
   refactoring `server.mjs` to accept injected dependencies (a
   forbidden touch — the WP scope-locks `server.mjs` to the three
   blocks in §C) or pulling in a test-double infrastructure that
   doesn't exist elsewhere in the server test suite. Both
   substantially exceed "smallest-surface form".

3. **Set a dummy `DATABASE_URL` pointing at a non-listening port
   so `loadRules()` fails fast**. Rejected: `loadRules()` calls
   `process.exit(1)` — there is no fast-fail-with-recovery path.

4. **Export `tryConstructHankoVerifier()` and test it directly**
   (chosen). The helper IS the production-fatal-vs-dev-mode policy
   boundary; testing it directly tests exactly that policy. The
   `export` keyword is a one-character addition with no production
   semantics.

The chosen path preserves the locked behavior assertion (both
production-fatal diagnostic verbatim and dev-mode-undefined
return) and the locked baseline shift (`+5` tests = 3 resolver +
2 startup-guard) without modifying any forbidden file or adding
test infrastructure.

## Forward-looking rule for future server-startup tests

When a future WP needs to test `startServer()` behavior end-to-end
(e.g., a future WP that adds a new startup task), the executor
faces the same `loadRules()` / `DATABASE_URL` constraint. Two
durable options:

1. **Export the helper under test and exercise it at the boundary**
   (the WP-131 precedent). Works when the new behavior lives in a
   pure function callable without the surrounding I/O. Use this by
   default — it produces the smallest test surface and the
   clearest behavioral assertion.

2. **Stand up a CI-side `DATABASE_URL` fixture**. Required only
   when the new behavior is intrinsically tied to the post-
   `loadRules()` state (e.g., a startup task that reads from
   `legendary.rules` immediately and behaves differently based on
   row contents). Adds CI complexity; reserve for cases where
   option 1 doesn't apply.

The implicit assumption that "calls `startServer()` inside
try/catch" works in any local dev environment is wrong. Future WP
authors drafting server-startup tests should explicitly call out
which option their test plan assumes — option 2 requires the WP to
also specify the CI fixture.
