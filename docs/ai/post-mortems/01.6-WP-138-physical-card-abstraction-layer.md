## New Patterns Discovered

### Schema-internal cross-field validation via `superRefine`

`HeroSchema.superRefine` enforces three cross-field invariants (orphan-side, duplicate-membership, drift) that span `cards[]` × `physicalCards[]` × `cardCounts`. The choice was between three options:

1. **Loader-side validation** — both `localRegistry.ts` and `httpRegistry.ts` call a shared validator helper (e.g., in `shared.ts`).
2. **`superRefine` on `HeroSchema`** — every `SetDataSchema.safeParse` inherits the cross-field check.
3. **Inline duplication in both loaders** — keeps scope tight at the cost of two copies.

Phase 1a adopts (2) — `superRefine` is the single seam for all three checks. Benefits:

- Cross-field invariants live next to the schema they constrain. A reviewer reading `HeroSchema` sees the entire contract in one place.
- Both loaders pick up the validation automatically with zero loader-side change. The errors flow through the existing `errors[]` array and surface in `validate()`'s `HealthReport`.
- Tests can call `HeroSchema.safeParse({...})` or `PhysicalCardSchema.safeParse({...})` directly without spinning up a loader, file system, or in-memory mock.

This is now the default pattern for any future cross-field invariant on a registry schema. Defer loader-side validation to scenarios where the check needs runtime context not present in the parsed object (e.g., cross-set or cross-loader state).

### Patch-declared structure consumed via separate file-load pass

`buildPhysicalCards` reads the patch file via a new `loadPatchFile(setAbbr)` helper rather than threading the patch object through `applyPatch`. The two reads are kept independent:

- `applyPatch` walks sections imperatively and merges fields into `result` — its mental model is "field-by-field merge."
- `buildPhysicalCards` reads `patch.heroes[].physicalCards` declaratively and synthesises the per-hero output — its mental model is "deck-composition primitive build."

Coupling them would force `applyPatch` to either expose its internal state (the loaded patch JSON) or grow a callback hook for the new behavior. The double-read costs one extra disk read per set on a dev tooling script — acceptable. This is now the precedent for any future converter-stage extension that needs structured patch consumption: read the patch fresh, do the work, never thread through the merge pipeline.

### Solo-auto-path as the uniform-model fallback for un-curated split heroes

D-13803 mandates that every hero with `cards[]` non-empty has a non-empty `physicalCards[]`. Curating per-hero patches for ~25 split-side heroes is high-touch work deferred to Phase 1b. The auto-path (one single-side physicalCard per `cards[]` entry) makes the data structurally valid for un-curated heroes while the audit-warning surface produces the worklist:

- Structurally-valid output flows through `HeroSchema.safeParse` cleanly. The `superRefine` checks pass because solo-auto-path emits exactly one physicalCard per side, with `count` matching `cardCounts` for that side (when populated).
- Semantically-wrong output for split heroes is non-blocking under Phase 1a because no consumer reads `physicalCards[]` as authoritative yet. Engine + viewer code continues reading `cards[].imageUrl` and per-side `cardCounts`.
- The audit-warning surface (262 candidates emitted from this run) is the Phase 1b worklist authoring input — every paired-equal `cardCounts` pattern is named with its set, hero, value, candidate card names, and the patch file to edit.

This is the locked-author pattern for any future schema migration that adds a required field with structurally-valid auto-fill while consumer migration is staged. The auto-path's wrongness is bounded to "doesn't yet match physical reality"; correctness restored when curated patches land.

The full surface is the Phase 1b WP's authoring input. Some candidates are likely false positives (heroes with two genuine independent cards that happen to have the same count). Phase 1b drafting must distinguish:

- **True split pairs** — two cards with the same count + same rarity that print on the front/back of one physical artifact (Falcon/Winter Soldier pattern). Get an explicit `physicalCards[]` block.
- **False positives** — two independent cards whose count matches by coincidence. Get a declarative annotation suppressing the warning, OR no annotation if the false-positive cost (one warning per re-conversion) is acceptable.

The `--strict` flag is the gate that forces Phase 1b's completeness: CI green-state under `--strict` requires every audit warning to be either resolved (declared physicalCards) or suppressed (whitelisted false positive). Phase 1b's WP §Acceptance Criteria will lock the resolution.

## Scope-Discipline Surprises

### One: pCloud-shadowed worktree paths

The session ran inside a git worktree at `.claude/worktrees/stupefied-bhaskara-f7dcc6/`. The Read / Edit / Write tools accept absolute paths; passing `C:\pcloud\BB\DEV\legendary-arena\packages\registry\src\schema.ts` (without the worktree prefix) targets the **main repo** at `main`, not the worktree branch. Bash via `pwd` correctly resolves to the worktree, but the Read tool transparently follows whatever absolute path is passed.

The first ~12 file edits in this session went to the main repo by mistake (the same paths the gitStatus block used to describe "Recent commits"). Symptom: the `node` script reading the patch file showed `physicalCards: undefined` even though the Read tool showed `physicalCards: [...]`. Diagnosis: divergent reads — Read was reading the (edited) main file; Bash + Node were reading the (un-edited) worktree file.

Recovery: copied all my modifications from main → worktree via `cp`, then reverted the main repo's WP-138-related files via `git checkout --`. Re-ran `convert-cards-v15.mjs` and `apply-card-counts.mjs` in the worktree to regenerate `data/cards/*.json` against the worktree's (correct) patch file. All gates re-verified; no leakage of the misdirected edits into the main repo's eventual commit.

This is a class hazard for any worktree-based session where the worktree path differs from the user's stated absolute path. The locked-author defense going forward: when working in a git worktree, every Read / Edit / Write absolute path must include the worktree directory prefix. The gitStatus block at session start lists the worktree path explicitly — read it, use it. Bash `pwd` is the source of truth for which directory the session is operating in; Read / Edit must match.

### Two: §18 prose-vs-grep retroactive softening

Initial JSDoc / `// why:` comments enumerated `localeCompare` verbatim while citing D-13802. The session prompt's §11 grep gate `grep -rn "localeCompare" packages/registry/src/ scripts/convert-cards/` returned three matches — all in JSDoc prose, none in actual usage. Per §18 of `00.3-prompt-lint-checklist.md`, prose discussing forbidden tokens must cite the governing decision rather than enumerate verbatim. Retroactively softened three sites (schema.ts, convert-cards-v15.mjs, apply-card-counts.mjs JSDoc; patches/README.md format docs) to cite "D-13802 for the full forbidden list of locale-aware comparison APIs" instead. Grep gate now empty. No governance intent lost.

This is a tooling-ergonomics quirk of literal-string-scoped greps: prose discussing a forbidden token can trip the gate as a false positive. The §18 fix pattern (cite the decision, don't enumerate) is the locked author guidance.

### Sort-comparator coverage

No non-ASCII slugs surfaced in real data during this run. Every hero / card slug across 40 sets used `[a-z0-9-]+` ASCII characters. The D-13802 sort lock (UTF-16 code-unit ordering) is therefore equivalent to ASCII byte ordering for the current data set. If a future card is published with a non-ASCII slug (accented Latin, Cyrillic, CJK), the sort behavior remains deterministic across environments — UTF-16 code-unit ordering is well-defined by the JS spec and locale-independent. No surprises observed; the lock is not load-bearing for current data but is load-bearing for future-proofing.
