- **Migration idempotency.** `CREATE TABLE IF NOT EXISTS` + `CREATE UNIQUE INDEX IF NOT EXISTS` + `CREATE INDEX IF NOT EXISTS`. Re-applying the migration twice is a no-op verified at §5.
- **Safe Knob Posture (per WP-132 §Safe Knob Posture).** Zero runtime feature flags, zero env-var-based behavior switches, zero conditional-on-account-attribute branches in the route handler. The endpoint's behavior is a pure function of `legendary.entitlements` rows. Adding a new entitlement key is a data edit + WP + DECISIONS.md entry — never a runtime toggle.

## Common Failure Smells

- `entitlements.logic.ts` contains an `INSERT INTO legendary.entitlements` or `UPDATE legendary.entitlements` "for completeness" / "in preparation for WP-134" → read-only invariant violated. The INSERT site is WP-134's authoring scope by lock; introducing it here is a layer-of-abstraction-creep that defeats the WP-132 / WP-134 split.
- Migration file references a non-existent `account_id` column on `legendary.players` (e.g., `account_id text NOT NULL REFERENCES legendary.players(account_id)`) → FK contract drift — `legendary.players` PK is `player_id bigserial` and its UNIQUE alternate key is `ext_id text`; there is no `account_id` column. Migration would fail at apply-time with `column "account_id" does not exist`. Fix: adopt the WP-104 D-10402 + WP-109 D-10906 `bigint`-FK-on-`player_id` precedent verbatim. Helper does the `AccountId` → `player_id` lookup at the application layer (mirrors [ownerProfile.logic.ts:123](apps/server/src/profile/ownerProfile.logic.ts:123)).
- Helper SQL uses `WHERE account_id = $1` instead of `WHERE player_id = $1` → same FK contract drift; query returns zero rows on a healthy DB because the `account_id` column doesn't exist or doesn't carry the brand value. Fix: Step 1 maps `AccountId` → `player_id`; Step 2 filters on `player_id`. Both queries verified by §5 grep gates.
- Helper short-circuits Step 1 by passing `accountId` directly into Step 2's `$1` (`WHERE player_id = $1` with `$1 = accountId`) → type-mismatch (`accountId` is `text`/`AccountId`; column is `bigint`). Postgres errors at execution time. Fix: Step 1 must produce a `player_id bigint` from the `players.player_id` column; bind that value into Step 2.
- `entitlements.routes.ts` accepts an `accountId` query parameter or path parameter (instead of resolving it from `requireAuthenticatedSession`) → cross-account read pathway opened. Vision §3 violated. Fix: `accountId` MUST come exclusively from the validated session result.
- `Cache-Control: no-store` set inside an `if (success)` branch → header missing on error responses; production cache infrastructure can mask 500 fault state. Fix: header is the FIRST statement of the handler body, before any branching, mirroring WP-115 D-11504.

Fix: the `default: const _: never = key` branch is the load-bearing primitive; without it the test is a soft check.
