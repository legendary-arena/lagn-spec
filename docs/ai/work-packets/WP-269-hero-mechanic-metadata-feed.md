**Status:** Draft (candidate — not yet reviewed, not ready to execute)

**Dependencies:** WP-251-era hero mechanic ledger (`docs/ai/coverage/hero-mechanic-ledger.json` — the committed, CI-gated source of truth); WP-086 / WP-183 / WP-184 metadata-publish precedent (`data/metadata/*.json` staged in-repo, fetched from R2 by the viewer)

---

## Session Context

The dashboard `/coverage` page surfaces 134 distinct **hero** mechanics from a committed, CI-gated ledger at `docs/ai/coverage/hero-mechanic-ledger.json` (623 hero-card × mechanic rows). The registry-viewer (cards.legendary-arena.com) has no access to that taxonomy: it is generated by repo-root scripts that import `@legendary-arena/game-engine`, which the viewer's layer boundary forbids it to import, and the dashboard's copy lives in `apps/dashboard` (another app the viewer must not import).

The viewer's established pattern for every taxonomy it already shows (card types, ability-effect tags, WP-183/184 patterns) is to **fetch a Zod-validated JSON from R2** (`/metadata/*.json`, staged in-repo at `data/metadata/`). This WP produces that feed for hero mechanics. It is the **producer half** of a two-WP slice (the consumer half — the viewer filter surface — is WP-270, hard-dep'd on this WP). The split follows the operator's "keep data-production and viewer-consumption separate" principle.

The raw ledger mechanic set is **not** UI-safe: free-text `[keyword:X]` markers are slugified into near-duplicate tokens (`cyber-mod`, `cyber-mod-wound`, `cyber-mod-4-wounds`), an `(unmarked)` sentinel, and punctuation artifacts. This WP introduces a **normalization layer** so the published feed carries only curated, display-ready mechanics, with raw/diagnostic tokens flagged `hidden: true` rather than shipped as primary UI chips.

---

## Goal

After this session, a deterministic transform script generates `data/metadata/card-mechanics.json` — a **versioned, viewer-safe, normalized hero-mechanic index** — from the committed hero ledger, validated by a new `CardMechanicsIndexSchema` exported from `@legendary-arena/registry/schema`. The published file carries (a) a `mechanics[]` collection (each with `slug`, `label`, `scope`, `source`, `cardCount`, `cardIds`, `hidden`) and (b) a per-card `cards{ extId: { mechanics: string[] } }` mapping so the viewer never parses ability text at runtime. A CI freshness gate (`mechanics:metadata:check`) fails if the committed file drifts from a fresh regeneration, mirroring `ledger:heroes:check`. The contract shape + normalization + hidden-by-default policy are locked by D-24046. **No `apps/registry-viewer` file is touched** — that is WP-270.

## Contract (Locked by D-24046)

The published `data/metadata/card-mechanics.json` shape:

```json
{
  "version": 1,
  "scope": "hero",
  "generatedAt": "<ISO-8601 UTC, sourced from the ledger or a fixed build stamp>",
  "mechanics": [
    {
      "slug": "berserk",
      "label": "Berserk",
      "scope": "hero",
      "source": "keyword | composition-marker | free-text",
      "cardCount": 12,
      "cardIds": ["core/hulk", "..."],
      "hidden": false
    }
  ],
  "cards": {
    "core/hulk": { "mechanics": ["berserk", "draw"] }
  }
}
```

- `version` integer; `scope` the literal `"hero"`; `mechanics[]` sorted by `slug`; `cardIds` sorted; `cards` keys are engine ext_ids.
- `source` is a closed 3-value union (`keyword` | `composition-marker` | `free-text`).
- `hidden` defaults `true` for any mechanic not present in `scripts/coverage/mechanic-labels.json` (fail-closed). Visible mechanics are an explicit curation decision.
- `generatedAt` is deterministic (no `Date.now()` in the transform — derive from the ledger's stamp or a fixed input) so `--check` is byte-stable.

7. `packages/registry/src/schema.ts` does NOT import `@legendary-arena/game-engine` (data-only schema).
8. `pnpm mechanics:metadata:check` exits 0 against the committed file and non-zero after a deliberate edit (freshness gate works).
