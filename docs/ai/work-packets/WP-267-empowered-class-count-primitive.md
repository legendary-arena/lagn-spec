## Session Context

WP-256 ratified the composable-primitive substrate (D-24029): the engine interprets a small, closed, drift-tested registry of *primitives*, and card mechanics become *data* that composes them. Berserk was the **first proof case** — a *parameter-less* mechanic whose composition is a static `HERO_COMPOSITION_MARKERS` row.

This packet is the **first parameterized mechanic**: **Empowered**. It is the highest-leverage next step because it forces two substrate capabilities most remaining hollow mechanics reuse:

1. A **value expression that counts cards by class in a board zone** (`count-cards-by-class-in-zone`) — the building block of every "+X per Y" buff.
2. The **first parameterized composition** — a mechanic whose composition AST depends on a parameter parsed from adjacent card text, not a static registry row.

Empowered's rule (`data/metadata/keywords-full.json`): *"'You get Empowered by Strength' means 'You get +1 Attack for each Strength card in the HQ.'"* — i.e. `gain Attack equal to the count of cards of the named class currently in the HQ`.

Today `[keyword:Empowered]` parses to nothing — it is one of the unsupported mechanics the coverage probe enumerates and (per the WP-265 sweep) one of the runtime-observed hollows (`empowered` — 13 observations, `parse-unrecognized`). Implementing it the *old* way would mean a new `HeroKeyword` + handler + drift ceremony. Instead Empowered is authored as a **parameterized composition of primitives**.

### Scope is the UNCONDITIONAL CORE only

The Empowered corpus is variadic (47 marker instances). This WP implements ONLY the dominant deterministic, unconditional text form:

```txt
[keyword:Empowered] by [hc:FIXEDCOLOR]
```

That form means:

```txt
+Attack = count of FIXEDCOLOR cards in the HQ
```

Every other variant is explicitly deferred and MUST keep its runtime hollow signal.

**Load-bearing boundary:** the implemented form is only valid when the Empowered marker itself is **not** condition-prefixed AND the marker tail immediately matches `by [hc:COLOR]`. A hook such as:

```txt
[hc:strength]: You get [keyword:Empowered] by [hc:tech].
```

is a deferred conditional-prefix variant for this WP, even though it contains the fixed-color tail. It MUST NOT be silently implemented by leaving the prefix as a hook condition. This protects the Honest-Partial Invariant and prevents partial implementation from becoming an accidental behavior commitment.

## The Composition (target descriptor)

Empowered's composition is **built per parsed color**, not a static row. For `[keyword:Empowered] by [hc:strength]` the parser synthesizes:

```ts
{
  type: 'gain-resource',
  resource: 'attack',
  amount: {
    type: 'count-cards-by-class-in-zone',
    heroClass: 'strength',          // normalizeTraitSlug of the [hc:COLOR] parsed from the marker tail "by [hc:...]"
    zone: 'hq'                      // the first shared/board-zone reference
  }
}
```

The engine gains: one value-expression primitive (`count-cards-by-class-in-zone`), one new parameter union (`EffectCountZoneKind = ['hq']` — the first **shared** zone, distinct from the per-player `EFFECT_ZONE_KINDS = [deck, discard]`), and a builder + parse step for the parameterized-composition pattern. **Empowered's AST is data; only the color is parsed.**

The values exist in `G` deterministically: `G.hq` is the 5-slot recruit row (`HqZone = [HqSlot×5]`, each `CardExtId | null`), and `G.cardTraits[id].heroClass` carries each card's setup-resolved class (the WP-179 `heroClassMatch` evaluator already reads exactly this). So `count-cards-by-class-in-zone` is replay-safe.

## Load-Bearing Invariant — the Honest-Partial Invariant

> Implementing only the unconditional core MUST NOT silence the hollow signal for the deferred variants. A card whose Empowered is a deferred variant (color-of-choice, Double/Triple, conditional-prefix, recruit-variant, multi-class, villain/mastermind) MUST still surface as a `parse-unrecognized` runtime hollow.

The mechanism: the parser treats `empowered` as **resolved only when it successfully extracts an unconditional `by [hc:COLOR]` and emits a composition.** When it cannot (the deferred forms), `empowered` is recorded as an **unresolved marker** exactly as today, so the WP-257 hollow detector still flags it at runtime. Recognizing the mechanic globally (short-circuiting the unresolved-marker path for every `[keyword:Empowered]`) would convert the deferred variants into **silent** no-ops — the precise failure the hollow-detection initiative exists to prevent. This is the highest-risk invariant in the WP; it is repeated in the EC Guardrails.

## User-Visible Impact

**Surface: play.legendary-arena.com (in-match) → dashboard.legendary-arena.com/coverage.** Before this WP, a hero card whose printed ability is the core `Empowered by [hc:COLOR]` form does **nothing** when played — a silent dead end for the player. After it, that card grants `+Attack` equal to the count of that-class cards in the HQ, exactly as the rulebook states. The change is observable two ways: (1) in a real match, the affected hero cards now contribute Attack; (2) on the `/coverage` dashboard, the `empowered` mechanic flips from Unsupported to Executable and its "Observed in play" count drops by the cleared core-form hits. The deferred Empowered variants remain visibly hollow on `/coverage` (honest partial coverage). Because the in-match form is hard to exercise deterministically on the live site, the **live-on-surface verification is the `/coverage` reflection** (the D-24026 check used across this initiative).

## Non-Negotiable Constraints

**Engine-wide:** `ctx.random.*` only (Empowered needs no randomness); never throw in effect execution (unknown value type, missing `G.hq`, missing `cardTraits` → warn to `G.messages` + continue); `G` JSON-serializable (the AST is plain data; the ImplementationMaps + composition registry are module-level consts outside `G`); ESM, Node v22+, `.test.ts`; **full file contents for every modified file — no diffs, no snippets, no "show only the changed section"**; human-style code per `docs/ai/REFERENCE/00.6-code-style.md` (no `.reduce()`, no nested ternaries, no dynamic known-key access, descriptive names, `// why:` on non-obvious decisions); no `boardgame.io`/registry import in the interpreter (it reads `G.hq`/`G.cardTraits` + uses `addResources`).

**Packet-specific:**
- **`HeroKeyword` is frozen.** No `empowered` member; `HERO_KEYWORDS` stays 17; its drift test needs no edit. Empowered attaches to `hook.primitiveEffects` (the WP-256 field), never to `hook.keywords`.
- **The count is a new VALUE expression, not a node type.** `EFFECT_NODE_TYPES` stays 3 (`sequence`/`move-card`/`gain-resource`); only `VALUE_EXPRESSION_TYPES` gains `count-cards-by-class-in-zone`. Each new closed union (the value-expression type + `EffectCountZoneKind`) gets a canonical readonly array + drift test (array/union parity; the value-expression registry asserted bidirectionally against its ImplementationMap keys — the WP-251/WP-256 pattern).
- **`EffectCountZoneKind` is separate from `EFFECT_ZONE_KINDS`.** `EFFECT_ZONE_KINDS = [deck, discard]` is the per-player move-card endpoint union and stays byte-unchanged. The count expression references a **shared** zone (`hq`) via its own union — do NOT overload the move-card zone union with a board zone (the interpreter's `getZoneArray`/`setZoneArray` resolve per-player `playerZones`; HQ is `G.hq`).
- **No mechanic-shaped macro.** A value expression named `empowered-amount` would FAIL D-24029 §6. The primitive is the orthogonal `count-cards-by-class-in-zone(heroClass, zone)`; Empowered is its composition with `gain-resource`.
- **`data/cards/**` byte-unchanged.** The `[keyword:Empowered] by [hc:X]` markers already exist; the parser learns to translate them. Zero re-marking.
- **Missing/empty HQ or `cardTraits` = `+0` (deterministic).** `count-cards-by-class-in-zone` returns 0 when `G.hq` is absent/not-an-array, when `G.cardTraits` is absent, or when no HQ slot matches (empty slots `null` are skipped); never throws.
- **The Honest-Partial Invariant** (above) — deferred variants stay `parse-unrecognized` hollows.
- **No broad parser capture** — the core parameter is anchored to the marker tail (§D); no cross-sentence scan binds a later `[hc:...]` to Empowered.

## Scope (In)

### A) Contract — `rules/effectPrimitive.types.ts` (modified)
- Add `'count-cards-by-class-in-zone'` to the `ValueExpressionType` union AND `VALUE_EXPRESSION_TYPES` (now 2).

- New interface `CountCardsByClassInZoneExpression { type: 'count-cards-by-class-in-zone'; heroClass: string; zone: EffectCountZoneKind }`, added to the `ValueExpression` union.

### B) Interpreter — `hero/effectPrimitive.interpret.ts` (modified)
- `evaluateCountCardsByClassInZone(G, expression, context)` — reads `G.hq`; for each non-`null` slot, reads `G.cardTraits[slot]?.heroClass` and counts equality with `expression.heroClass`; returns the count. **No self-exclusion** (Empowered counts every HQ card of the class, unlike `heroClassMatch`). Missing `G.hq` (non-array) or missing `G.cardTraits` → `0`, warn, never throw. Registered in `VALUE_EXPRESSION_EVALUATORS` under `'count-cards-by-class-in-zone'` (keeps the bidirectional drift parity).

### C) Open mechanic space — `rules/heroCompositions.ts` (modified)

- `PARAMETERIZED_COMPOSITION_MARKER_NAMES: readonly string[] = ['empowered']` — parameterized markers (a builder + a parse step), distinct from the static `HERO_COMPOSITION_MARKERS` rows.
- Extend `HERO_COMPOSITION_MARKER_NAMES` to the **de-duplicated** union (via a `Set`) of `Object.keys(HERO_COMPOSITION_MARKERS)` and `PARAMETERIZED_COMPOSITION_MARKER_NAMES`, so the coverage probe + ledger recognize `empowered` as a supported mechanic without duplicating the vocabulary. (Dedupe protects against a name later migrating between static and parameterized.)

### D) Parser — `setup/heroAbility.setup.ts` (modified) — the parameterized-marker step + condition suppression

Add a parameterized-marker branch for `[keyword:X]` tokens whose normalized name is in `PARAMETERIZED_COMPOSITION_MARKER_NAMES`. For this WP, the only seeded name is `empowered`.

#### Core-form recognition

A token resolves only when ALL of the following are true:

1. normalized marker name is exactly `empowered`;
2. the Empowered marker is **not** part of a condition-prefixed Empowered text form deferred by this WP (operationally: after consuming the parameter `[hc:COLOR]`, the line carries no other gating class/team condition token — a residual `[hc:X]:` / `[team:X]:` prefix means defer);
3. the text **immediately following the marker** matches:

```regex
^\s*by\s*\[hc:([a-z0-9-]+)\]
```

The parser MUST NOT do a broad forward scan that can bind a later unrelated `[hc:...]` token to Empowered.

**On match:**
- Compute `normalizedHeroClass = normalizeTraitSlug(color)`.
- Push `buildEmpoweredComposition(normalizedHeroClass)` to `hook.primitiveEffects`.
- Remove exactly ONE matching `{ type: 'heroClassMatch', value: normalizedHeroClass }` from `heroClassConditions`, if present, because the consumed `[hc:COLOR]` is the Empowered parameter, not a gate. (Suppression uses the same normalized value passed to the builder.)
- Do not add `empowered` to `hook.keywords`.

**On no match / deferred form** — record the marker in `unresolvedMarkers` exactly as today. This includes, but is not limited to: `by the color of your choice`; `[keyword:Double Empowered]`; `[keyword:Triple Empowered]`; conditional-prefix Empowered forms; recruit-variant forms; multi-class variants; villain/mastermind Empowered forms.

The parser treats `empowered` as resolved only when it successfully emits the core composition. Any broader recognition that removes `empowered` from `unresolvedMarkers` without emitting an executable composition violates the Honest-Partial Invariant. Empowered NEVER enters `hook.keywords`.
