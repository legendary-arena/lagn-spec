# WP-187 — Villain & Henchman Effect-Marker Enrichment (Card Data)

## Goal

Enrich the card data so that the unambiguous subset of Villain and
Henchman `Ambush:` / `Fight:` ability lines carries a structured
`[effect:<VillainEffectKeyword>]` marker. This is the upstream data
prerequisite that makes WP-185's engine parser able to detect and
execute Fight/Ambush effects: WP-185 reads `[effect:]` markers from
`abilities[]`; today there are none, so WP-185 cannot execute without
this WP. After WP-187, the curated lines (e.g. `Fight: KO one of your
Heroes`) carry `[effect:koHeroCurrentPlayer]` and become engine-readable.

---

## Assumes

- **No hard WP dependency.** WP-187 is the upstream root of the
  Fight/Ambush effect chain. It produces card-data artifacts consumed
  by WP-185; it does not consume any prior WP's runtime code.
- **WP-185 vocabulary lock (spec dependency, not code).** The marker
  vocabulary is exactly the five strings locked in
  `docs/ai/work-packets/WP-185-villain-fight-and-ambush-effects.md
  §Non-Negotiable Constraints` / `EC-212 §Locked Values`:
  `gainWoundEachPlayer | gainWoundCurrentPlayer | koHeroCurrentPlayer |
  heroDeckTopToEscape | captureBystander`. WP-185 need not have executed
  — its spec is on `main` and is the source of truth. The overlay script
  hardcodes a local copy of these five strings and loud-fails on any
  marker outside the set (drift guard).
- **Card-generation pipeline shape (verified 2026-05-28).**
  `data/cards/*.json` (40 sets, tracked in git) is generated by
  `scripts/convert-cards/convert-cards-v15.mjs` and overlaid by
  `scripts/convert-cards/apply-card-counts.mjs`, which reads curated
  `scripts/convert-cards/inputs/*.json` files and writes the set files
  back with `JSON.stringify(..., null, 2)` formatting so diffs stay
  clean. WP-187 adds a sibling curated input + a sibling overlay step
  following that exact precedent.
- **Data finding (verified 2026-05-28).** `grep -rn "\[effect:"
  data/cards/` returns zero matches today. The five effect keywords
  appear nowhere as markup tokens; the effects live in free text. This
  WP is what closes that gap.
- **Drafting baseline:** `origin/main @ ccc79bf` (2026-05-28).

## Context

WP-185's pre-flight surfaced a blocking data gap: the engine parser was
specced to extract effect keywords from `[icon:]` / `[keyword:]` markup,
but those namespaces carry card-mechanic names (`Undercover`, `Prey`,
…) and resource icons (`recruit`, `attack`, …) — **not** effect
vocabulary. The actual Fight/Ambush effects are free text. Per the
engine's no-NL-parsing discipline (hero-ability precedent), the engine
must read structured markers, not parse English. WP-185 was therefore
re-specced to read a dedicated `[effect:]` marker — and WP-187 is the
WP that authors those markers.

**Why curated, not auto-derived.** A phrase-scan of the data
(2026-05-28) matched ~244 candidate lines, but the matches
over-capture: "KO **two** of your Heroes" (magnitude 2), "**Each
villain in the city** captures a Bystander" (multi-source/target), and
"discards a card … **or** gains a Wound" (a choice, not an
unconditional each-player wound) all phrase-match but do **not** reduce
to a single WP-185 MVP effect. Blind injection would mis-mark them.
Therefore WP-187's authoritative artifact is a **human-reviewed marker
map**; an offline `--propose` mode bootstraps the review but is never
authoritative. The engine still reads only reviewed, structured markers
(no NL at runtime) — the build-time script may phrase-scan freely
because it runs offline and its output is committed and reviewable.

**Mechanism is timing-agnostic; v1 curation is Ambush + Fight only.**
The overlay handles `Ambush:` / `Fight:` / `Escape:` / `Overrun:`
prefixes generically, but WP-187 curates only `Ambush:` and `Fight:`
(WP-185's timing union). The `Escape:` / `Overrun:` curation that
WP-186 needs is a later data-only addition to the same map — no new
script WP required.

## Scope (In)

- **Curated marker map** —
  `scripts/convert-cards/inputs/villain-effect-markers.json`. Two
  top-level sections to match the two data shapes WP-185's parser
  handles:
  - `villains: { <setAbbr>: { <villainGroupSlug>: { <cardSlug>:
    { ambush?: VillainEffectKeyword[], fight?: VillainEffectKeyword[] } } } }`
  - `henchmen: { <setAbbr>: { <henchmanGroupSlug>:
    { ambush?: VillainEffectKeyword[], fight?: VillainEffectKeyword[] } } }`
    (henchman ability text is group-level per WP-185 — the marker
    applies to the group's `abilities[]` line.)
  - Each list contains only the five locked keyword strings. A
    `_unassigned` block at the top is a **structured array** (not free
    prose) documenting candidate lines deliberately left unmarked, so it
    is diffable and can feed a future WP-185 vocabulary expansion. Each
    row: `{ set, group, card?, timing, text, reason }` where `reason` ∈
    `{ "magnitude>1", "conditional", "multi-target", "multi-line",
    "other" }` (`card` omitted for henchman group-level entries). Mirrors
    and tightens WP-184's `_unassigned` convention.
- **Overlay script** — `scripts/convert-cards/apply-effect-markers.mjs`:
  - **Apply mode (default):** reads the marker map, locates the matching
    ability line by the deterministic predicate below, and appends
    ` [effect:<keyword>]` token(s) to that line. Iterates sets → groups →
    cards in lexicographic order (deterministic processing). Writes set
    files back with the same `JSON.stringify(..., null, 2)` formatting as
    `apply-card-counts.mjs`.
  - **Match predicate (consistent with WP-185):** a target line is one
    whose leading-whitespace-trimmed form begins with `<Timing>:`
    (case-insensitive; `<Timing>` ∈ `Ambush` / `Fight` for v1). The
    predicate runs on an ephemeral trimmed copy; the stored string is
    never altered except by the marker append. This is the **same**
    timing detection WP-185's parser uses — producer and consumer must
    agree on what counts as a timing line.
  - **Append ordering (canonical):** markers go at the **end** of the
    matched line — original text, then any existing trailing markup
    (e.g. `[rule:Adapt]`, order preserved), then `[effect:]` token(s).
    When a line gets multiple effect markers, they are appended in
    `VILLAIN_EFFECT_KEYWORDS` canonical order (not map-authoring order).
  - **Idempotent, per-keyword:** each distinct `[effect:X]` appears at
    most once per line; presence is checked per-keyword before append, so
    re-running produces a zero-line diff.
  - **Loud-fail** on: a keyword outside the five locked strings; a
    named set/group/card that does not exist; a timing key matching
    **zero or more-than-one** ability line. (Two cards — one in `bkwd`,
    one in `rvlt` — carry two `Fight:` lines apiece; the map shape cannot
    disambiguate which, so they loud-fail and are recorded as
    `multi-line` in `_unassigned` — v1 defers them.)
  - **`--propose` mode (dry-run):** scans all `Ambush:` / `Fight:`
    lines, phrase-matches candidate keywords, and prints one row per
    candidate — `set | group | card | timing | text | proposedKeywords`,
    sorted lexicographically for diffable review. Writes nothing.
    Read-only bootstrap; never authoritative.
- **Regenerated data** — the `data/cards/*.json` set files that contain
  curated cards are rewritten with the injected `[effect:]` markers.
  Diff is bounded to the curated ability lines.
- **v1 curation discipline (locked):** a line is marked **only** when it
  reduces to exactly one application of a single MVP keyword with
  WP-185's exact semantics:
  - **unconditional** — no `may`, no `If …`, no `… or …` choice
  - **magnitude 1** — `KO one of your Heroes` ✅; `KO two …` ✗ (deferred)
  - **single-target matching WP-185** — `captureBystander` = attach to
    *this* card; `Each villain … captures` ✗ (deferred)
  - A line with two genuinely-unconditional distinct MVP effects emits
    two markers; conservatism is preferred — when in doubt, leave
    unmarked (WP-185 safe-skips → `effects: []`).
- **Governance** — `STATUS.md` entry, `DECISIONS.md` D-18701..D-18703,
  `WORK_INDEX.md` flip to `[x]`, `EC_INDEX.md` EC-214 → Done.

## Out of Scope

- **`Escape:` / `Overrun:` marker curation** — deferred to a follow-on
  data-only addition for WP-186. The overlay script handles these
  prefixes generically; only the curated entries are deferred.
- **Engine parsing / execution** — entirely WP-185. WP-187 produces
  data only; it adds no engine, registry, or server code.
- **Expanding the five-keyword vocabulary** — locked by WP-185. A sixth
  keyword is a future WP in WP-185, not here.
- **Magnitude / conditional / multi-target effects** — out of the v1
  MVP. These lines stay unmarked and are documented in the map's
  `_unassigned` block.
- **Modifying `apply-card-counts.mjs`, `convert-cards-v15.mjs`, or any
  other existing converter** — WP-187 adds a sibling overlay, it does
  not alter existing generation steps.
- **`mechanicalPattern` taxonomy (WP-184)** — viewer-side metadata,
  unrelated surface.
- **Image/asset, schema, or registry-runtime changes** — none.

## Non-Negotiable Constraints

**Engine-wide (always apply):**

- ESM only, Node v22+. Full file contents for new files; no diffs/snippets.
- Human-style code per `00.6-code-style.md` — full English names, JSDoc
  on every function, `// why:` on non-obvious decisions.
- No randomness, no clocks, no network in the overlay script — it is a
  deterministic pure-IO transform over local files.

**Packet-specific:**

- The marker token is exactly `[effect:<VillainEffectKeyword>]` where
  `<VillainEffectKeyword>` ∈ the five locked strings. The overlay
  validates every emitted marker against its local copy of that set and
  **loud-fails** on any other value. A `// why:` on that local array
  states it MUST equal WP-185's `VILLAIN_EFFECT_KEYWORDS`.
- The overlay is **idempotent per-keyword**: each `[effect:X]` appears
  at most once per line; a re-run over already-marked data produces a
  zero-line diff.
- The overlay **loud-fails** (never silently skips) on: unknown keyword,
  missing set/group/card, zero-or-multiple matching timing lines.
- **Match predicate = WP-185's timing detection.** A target line is
  located by `line.trimStart()` beginning with `<Timing>:`
  (case-insensitive). The predicate is ephemeral; the stored string is
  altered only by the marker append. Producer (this script) and consumer
  (WP-185 parser) MUST use the same predicate or markers drift onto
  lines the engine reads differently.
- **Append-only, canonical order.** Markers are appended at the end of
  the line — original text, then existing markup (order preserved), then
  `[effect:]` token(s) in `VILLAIN_EFFECT_KEYWORDS` order. The overlay
  MUST NOT insert mid-text, split/reflow the line, edit punctuation,
  reorder words, or rewrite the human-readable effect text.
- **Deterministic iteration:** process sets → groups → cards in
  lexicographic order.
- The local five-keyword array is the validation gate; it is a hardcoded
  copy with **no import/auto-sync** from engine TS (a `.mjs` script
  cannot and must not reach into `packages/`). If WP-185 ever changes the
  vocabulary, this script fails on the new value until manually updated —
  drift is intentional, not silent.
- v1 curation marks only unconditional, magnitude-1, single-target
  lines (see Scope discipline). Conservatism over coverage.
- WP-187 adds **no** engine/registry/server code and modifies **no**
  existing converter script.

**Session protocol:**

- If a `--propose` candidate's semantics are ambiguous, leave it
  unmarked and record it in the map's `_unassigned` block. Do not
  guess. Do not stretch a line to fit a keyword.
- If a real line needs a sixth keyword or a magnitude, that is out of
  scope — leave unmarked; vocabulary expansion is a WP-185-side future
  WP.

**Locked marker vocabulary (mirrors WP-185 — do not re-derive):**

```
gainWoundEachPlayer
gainWoundCurrentPlayer
koHeroCurrentPlayer
heroDeckTopToEscape
captureBystander
```

```pwsh
# Apply the overlay (idempotent) and confirm a clean re-run
node scripts/convert-cards/apply-effect-markers.mjs
git add -A && git stash   # capture, then re-run to prove idempotency
git stash pop
node scripts/convert-cards/apply-effect-markers.mjs
git diff --stat data/cards/   # expected: no changes on second run

# Markers landed and are vocabulary-clean (only the 5 locked strings)
grep -rhoE "\[effect:[^]]+\]" data/cards/ | sort | uniq -c

# Loud-fail behaviors (each must exit non-zero with a clear message)
#  - temporarily add a bogus keyword / missing card to a scratch copy
#    of the input and confirm the script refuses to proceed

# Propose mode writes nothing
node scripts/convert-cards/apply-effect-markers.mjs --propose | head

# Full monorepo build (data-only change)
pnpm -r build
```

Expected: the `uniq -c` output lists only the five locked keywords; the
second apply run shows no `data/cards/` diff; `--propose` prints a table
and leaves the tree clean; `pnpm -r build` exits 0.
