## Session Context

WP-267 authored the **unconditional core** of Empowered (`[keyword:Empowered] by [hc:FIXEDCOLOR]` ‚Üí `+Attack = count of that-class HQ cards`) as the first parameterized composition, and **explicitly deferred** every gated/variadic form, including the **conditional-prefix** form, to keep its scope honest (WP-267 ¬ß"Load-bearing boundary"). That deferral left a class of cards visibly dead in real play: a live diagnostics capture (`gitSha 048e572`, 2026-06-20) shows `antm/wonder-man/ionic-energy` (`[hc:ranged]: You get [keyword:Empowered] by [hc:ranged]`) firing `Unhandled effect observed ‚Ä¶ "empowered" ‚Ä¶ parse-unrecognized` on every play ‚Äî the card does nothing.

This packet implements exactly that deferred form, and **only** the safe class-gated shape, by reusing everything WP-267 built. The conditional-prefix form is the cheapest remaining Empowered variant because its semantics map onto machinery that already exists:

- the **composition** is the same `buildEmpoweredComposition(color)` output (no new primitive/builder);
- the **gate** is an ordinary `heroClassMatch` condition the WP-179 evaluator already handles;
- the **executor** already runs `hook.primitiveEffects` only when the hook's conditions pass (WP-256) ‚Äî so retaining the `[hc:X]:` prefix as a hook condition *is* the conditional behavior, with no executor change.

The only change is in the **parser**: WP-267 resolves Empowered only when the consumed `[hc:COLOR]` param is the line's *sole* condition (clearing all conditions on resolve). This WP adds a sibling resolve path that recognizes the canonical conditional-prefix shape, suppresses **only** the consumed count param, and **retains** the leading class gate.

## The Resolution Gate (structural, not condition-counting)

The naive rule "resolve when the residual conditions after consuming the param are all `heroClassMatch`" is **unsafe**: `wtif/star-lord-tchalla/fight-or-flight` (`Choose one: ‚Ä¶ Empowered by [hc:strength], or ‚Ä¶ Empowered by [hc:covert]`) would leave a residual `heroClassMatch(covert)` that is the *second choose-one branch's count param*, not a gate ‚Äî mis-resolving it as `empowered(strength)` gated by `covert`. The gate must therefore be **structural**.

A `[keyword:Empowered]` token resolves the conditional-prefix form when **ALL** of the following hold (operationally locked in EC-302):

1. the normalized marker name is exactly `empowered`;
2. the ability text contains **exactly one** `[keyword:Empowered]` marker (rejects choose-one / double-marker forms);
3. the ability text **begins with a single class-condition prefix** matching `^\s*\[hc:([a-z0-9-]+)\]\s*:` (the prefix gate class `X`); nothing else precedes the gate;
4. the text **immediately following the marker** matches the anchored tail `EMPOWERED_PARAM_TAIL_PATTERN` (`^\s*by\s*\[hc:([a-z0-9-]+)\]`) ‚Üí the count color `Y` (fixed color);
5. the text immediately after the consumed tail does **not** continue with `and [hc:‚Ä¶]` (rejects multi-class);
6. the ability text contains **no** `[team:‚Ä¶]` token (rejects team-gated forms).

**On resolve:**
- push `buildEmpoweredComposition(normalizeTraitSlug(Y))` to `hook.primitiveEffects`;
- push `empowered` to `resolvedMarkers` (the WP-268 by-hook provenance ‚Äî same gate as the `primitiveEffects` push);
- remove **exactly one** `{ type: 'heroClassMatch', value: normalizeTraitSlug(Y) }` from `heroClassConditions` (the consumed count param), and **retain** the leading `heroClassMatch(normalizeTraitSlug(X))` gate so the hook fires the effect only when that condition passes;
- do **not** add `empowered` to `hook.keywords`.

**On no match / any deferred form** ‚Äî record `empowered` in `unresolvedMarkers` exactly as today (the Honest-Partial Invariant). The unconditional-core path (`tryResolveEmpoweredCore`, sole-condition, empty residual) is unchanged and still handles the WP-267 forms.

> Same-class case (the common one, e.g. Ionic Energy `X == Y == ranged`): Step 1a extracted two `heroClassMatch(ranged)` (prefix + tail). Removing exactly one leaves exactly one ‚Äî the gate. Different-class case (`[hc:strength]: ‚Ä¶ by [hc:tech]`): removing the one `heroClassMatch(tech)` leaves `heroClassMatch(strength)` as the gate. Both correct.
