## Goal

After this session, the operator dashboard has a new top-level navigation item,
**Pipeline**, that opens a single page (`/pipeline`) visualizing the four
checks-and-balances roles from `apps/dashboard/docs/code-checks-and-balances.md`
— **Architect → Builder → Inspector → Evaluator** — as one left-to-right
pipeline of four lane-cards. Each lane shows that role's most recent output and
a status, derived entirely from the existing build-time governance snapshot
(`useGovernanceSnapshot`). The Evaluator lane is an honest placeholder until an
acquisition-readiness data source exists. This makes the abstract
separation-of-duties model observable in one glance — and deliberately keeps it
off the revenue Overview, as its own nav item, per the
`dashboard-operating-system.md` audit (Overview is already governance-heavy).

This is an internal operator-only surface. It is additive: it presents the same
governance data the existing `GovernanceThroughputWidget` / `StatusFeedWidget` /
`GovernanceKpiStrip` already consume — through the four-role lens — and modifies
none of them.

    `blocked()` (the review/dependency gate). Best-available proxy; see Out of
    Scope re: structured review-status.

    Evaluator Placeholder copy.

## Out of Scope

- **Per-agent separate pages/routes** (one route per Architect/Builder/
  Inspector/Evaluator). Explicitly rejected: the four roles are a sequence whose
  handoffs must be seen together, and Inspector/Evaluator have too little data to
  justify their own pages. v1 is one page with four lanes.
- **Any change to `build-governance-snapshot.mjs`** (the snapshot generator).
  Enriching the Inspector lane with structured review-status
  (`Needs review` / `Reviewed`) and the Evaluator lane with acquisition-readiness
  reports both require new snapshot fields — deferred to a follow-up WP. This WP
  consumes only the snapshot's current accessor surface.
- **Modifying the existing governance widgets** (`GovernanceThroughputWidget`,
  `StatusFeedWidget`, `GovernanceKpiStrip`). They stay byte-identical; Pipeline
  re-presents the same data, it does not relocate or restyle them.
- **Adding Pipeline content to the Overview page.** It is a separate nav item,
  not an Overview widget (Overview is already governance-heavy per the audit).
- **Evaluator data plumbing** — acquisition-readiness scoring, report storage,
  or any per-evaluation persistence.
- Any `apps/server`, `game-engine`, `registry`, or `preplan` edit.

## Non-Negotiable Constraints

**Engine-wide (always apply — do not remove):**
- Output the **full file contents** for every new or modified file. Diffs,
  snippets, and "show only the changed section" are forbidden.
- ESM only; Node v22+; no CommonJS, no `require()`.
- Human-style code per `docs/ai/REFERENCE/00.6-code-style.md` (explicit control
  flow with `for...of` not branching `reduce()`, descriptive names, JSDoc on
  every function, `// why:` on non-obvious decisions, full-sentence error
  messages).

**Packet-specific:**
- `PipelineItem` shape is locked: `{ id: string; label: string; meta?: string }` — `id` is a stable key (commit sha or synthetic string), `label` is the primary human-readable text, `meta` is optional supplemental context (e.g., commit kind, counts). The composable returns this shape; tests assert against it.
- `useAgentPipeline.ts` derives **all** lane data from `useGovernanceSnapshot`
  (imported). It MUST NOT import `../data/governance-snapshot.json` directly and
  MUST NOT read `import.meta.env` — single data source.
- The page presents **one** page with four lanes — not four routes. Exactly one
  new route (`/pipeline`) and exactly one new `NAV_ITEMS` entry.
- `PipelinePage.vue` mirrors the existing 4-state Widget Contract:
  - **Loading state:** render a loading indicator before snapshot resolution.
  - **Error state:** if `loadError` is non-null, the page renders a single unified error message; no lane content renders.
  - **Empty state:** each lane renders its `emptyMessage` when it has no items.
  - **Data state:** four lane-cards rendered with their items.
  No fabricated data when the snapshot is absent or errored.
- The Evaluator lane MUST return exactly one item in `items` containing the Locked Evaluator Placeholder string; `emptyMessage` MUST NOT be used for the Evaluator lane (the lane is never "empty").
- If `governanceKpis()` returns `null`, Architect and Builder KPI-derived items are omitted (not fabricated); only commit-derived items remain, or `emptyMessage` if none.
- No item may appear in more than one lane in a single evaluation pass (no cross-lane commit leakage).
- The existing governance widgets and every other page/widget are untouched —
  additive only. `git diff --name-only` at close lists exactly the nine
  `## Files Expected to Change` files; `git diff -- apps/dashboard/src/widgets/` returns empty.
- No new npm dependencies. No `Math.random()` in the new files. No
  `apps/server`, `game-engine`, `registry`, or `preplan` imports.

**Session protocol:**
- If the lane-to-accessor mapping cannot be satisfied from the snapshot's
  current accessors (e.g., a needed field is absent), **stop and ask** — do not
  modify `build-governance-snapshot.mjs` to add fields (that is a follow-up WP).

**Locked Contract Values:**
- Route: `path: 'pipeline'`, `name: 'pipeline'`, `meta.roles: ['admin', 'operator']`.
- Nav entry: `{ to: '/pipeline', label: 'Pipeline', abbreviation: 'Pi' }`
  (label is the operator's choice — `Pipeline` is the locked default;
  alternatives `Agents` / `Governance Pipeline` are a one-line change if Jeff
  prefers).
- Lane order (left to right): Architect, Builder, Inspector, Evaluator.
- Lane data sources (see §Lane Composition below): Architect = `governanceKpis().openDrafts` + `commits(5)` filtered `kind === 'SPEC'`; Builder = `governanceKpis().wpsDoneThisWeek` + `inFlight()` + `commits(5)` filtered `kind === 'WP'`; Inspector = `nextExecutable(5)` + `blocked()`; Evaluator = single locked placeholder item.
- `commits()` is called with explicit limit `5`; items rendered in returned order — no client-side sorting.
- Builder assumes execution commits classify as `kind === 'WP'` (per the generator's classifier); if this invariant changes, the Pipeline updates in a follow-up WP.
- **Locked Evaluator Placeholder** (verbatim):
  `No acquisition-readiness evaluation recorded yet. Run the Evaluator quarterly per code-checks-and-balances.md §7.`

### Lane Composition (Authoritative)

| Lane | KPI Item | State / Execution Items | Commit Items |
|---|---|---|---|
| Architect | `governanceKpis().openDrafts` | — | `commits(5)` where `kind === 'SPEC'` |
| Builder | `governanceKpis().wpsDoneThisWeek` | `inFlight()` | `commits(5)` where `kind === 'WP'` |
| Inspector | — | `nextExecutable(5)` + `blocked()` | — |
| Evaluator | — | Single locked placeholder (never empty) | — |

`commits(5)` returns the 5 most recent commits; items rendered in returned order without client-side sorting. If `governanceKpis()` returns `null`, KPI items are omitted; if no other items remain, the lane shows `emptyMessage`. No cross-lane leakage.
