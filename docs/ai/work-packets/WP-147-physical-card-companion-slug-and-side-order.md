# WP-147 — PhysicalCard `companionSlug` + Physical-Side Order

**Status:** Ready
**Primary Layer:** Registry / Contracts + Scripts (convert-cards) + Data
**Dependencies:** WP-137 (cardCounts authority), WP-138 (physicalCards introduction), WP-140 (`_skipPair[]` annotation grammar)

---

## Session Context

> WP-137 / WP-138 / WP-140 established `hero.physicalCards[]` as the
> authoritative deck-composition surface with `sides[]` (length 1 or 2) and
> a deterministic `imageUrl` derived from `Array.prototype.sort()` of side
> slugs (D-13801, D-13802, D-13803). This packet extends that model with a
> per-physical-card `companionSlug?` field (for hero-with-sidekick artwork)
> and overrides D-13802's alphabetical sort lock for two-side cards in
> favor of source-data physical-side order. The only data fix in this
> packet is Drax in `mgtg.json`; an audit of the 37 other existing
> two-side physical cards is explicit out-of-scope follow-up.

---

## Goal

After this session, `packages/registry/src/schema.ts` accepts an optional
`companionSlug` on every `physicalCard`, the `heroImageUrl()` function in
`scripts/convert-cards/convert-cards-v15.mjs` emits filenames that include
the companion slug between hero slug and side slugs when present and
preserves **source-data `sides[]` array order** for two-side cards (no
`.sort()`), and `data/cards/mgtg.json` contains the corrected Drax hero
with two split physical cards carrying their companion slugs and
physical-side ordering.
The change is additive: the 1245 existing single-side physical cards keep
their current `imageUrl` values; the 39 existing two-side physical cards
keep their current `imageUrl` values unless their `sides[]` order diverges
from physical-side order (audit follow-up out of scope per §Out of Scope).

## Debuggability & Diagnostics

All behavior introduced by this packet must be debuggable via
deterministic reproduction and state inspection.

- Given the same `data/cards/<set>.json` input file and the same
  `heroImageUrl()` arguments, the function returns a byte-identical
  URL string. Determinism comes from:
  - `sides[]` order in source data (authoritative for two-side cards)
  - String literal `companionSlug` (when present)
  - No locale-aware comparison APIs anywhere in the path
- Schema validation failures must produce human-readable Zod errors
  that name the offending field (`companionSlug`) and include a path
  identifying the physical-card location (e.g.,
  `heroes[n].physicalCards[m].companionSlug`). If existing registry
  validation already annotates hero slug + physicalCard id, preserve
  that behavior — but this WP does not require adding new
  cross-object error annotation machinery. The Zod messages for
  `companionSlug` validation failures (empty string, whitespace,
  invalid characters) must follow the same full-sentence format used
  elsewhere in `PhysicalCardSchema`.
- The Drax data fix must round-trip: load `mgtg.json`, validate
  via `RegistrySchema`, re-serialize, re-load — identical output.

### B) Factor `heroImageUrl` into the registry package

**`packages/registry/src/heroImageUrl.ts`** — **new**:

- TypeScript module exporting one function:
  `heroImageUrl(setAbbr: string, heroSlug: string, sides: readonly string[], companionSlug?: string): string`
- Internal logic:
  - Length guard: throw if `sides.length < 1 || > 2` with the
    existing full-sentence error message (preserve D-13802 ceiling
    lock — raising it requires its own DECISIONS entry). The error
    message must be **byte-identical** to the previous implementation
    in `convert-cards-v15.mjs` to preserve test and diagnostic
    stability.
  - When `companionSlug` is set: insert `-{companionSlug}` between
    `-{heroSlug}` and the side segment. Apply uniformly to single-
    side and two-side cards. Validate that `companionSlug` matches
    `/^[a-z0-9-]+$/` and throw a full-sentence error otherwise.
    This validation acts as **defense-in-depth at the module
    boundary**; the Zod schema validation in
    `PhysicalCardSchema.companionSlug` remains the primary
    enforcement for registry data.
  - When `sides.length === 2`: use `sides.join('-')` directly
    (preserving source order). NO `.sort()`. Add `// why:` comment
    citing D-14702: source-data `sides[]` order is authoritative
    (physical-side order: side A on the left/top of the printed
    card is first in the array).
  - When `sides.length === 1`: emit
    `{setAbbr}-hr-{heroSlug}[-{companionSlug}]-{sides[0]}.webp`.
- Export the `R2_BASE_URL` constant
  (`https://images.barefootbetters.com`) from this module — the
  convert script currently defines it as a local literal; moving it
  here keeps the URL host as a single source of truth for whoever
  lands the host-migration WP next. (`// why:` comment notes that
  this constant is used only for URL string assembly — no module
  in this WP performs network access against it — and that any
  change to the host is a separate orthogonal WP.)
- Full JSDoc on the function: signature, parameters, return value,
  thrown errors, and a `// why:` citing D-14701 (companionSlug
  introduction) and D-14702 (physical-side order).

Viewer.

making the registry capable of expressing artwork that depicts the
hero plus a named non-hero companion (a property of the printed
physical card that the previous schema could not represent). It
strengthens §10a by giving the Registry Viewer the data hook
(`companionSlug` in `imageUrl`) needed to render the correct
artwork for split-side companion cards.

or persuasion. It does not introduce user-visible copy of any kind.

determinism. `companionSlug` is a string literal carried in source
data. The two-side filename order, after D-14702, is the order of
`sides[]` in source JSON — identical across all environments,
since JSON files are the same on Windows, Linux, and CI. No
locale-aware comparison APIs are added anywhere. The 1245
existing single-side imageUrls remain bit-identical after the
change; the 39 existing two-side imageUrls remain bit-identical
when `sides[]` is already in alphabetical order (which is
currently the case for all 39 by construction — re-ordering of
those entries is the audit-scope follow-up explicitly out of
scope here).
