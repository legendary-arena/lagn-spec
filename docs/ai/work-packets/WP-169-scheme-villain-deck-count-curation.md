# WP-169 ‚Äî Scheme Villain-Deck Twist & Bystander Count Curation (Pipeline Data)

**Status:** Draft ‚Äî pending pre-flight + copilot gates and human review
**Primary Layer:** Registry / Card Data Pipeline (`scripts/convert-cards/`)
**Dependencies:** WP-167 (complete) ‚Äî provides the optional `SchemeSchema.villainDeckTwistCount` / `villainDeckBystanderCount` fields (D-16702), the `inputs/scheme-deck-counts.json` input file, and the `applySchemeDeckCounts` converter wiring + loud-fail. Does **not** depend on WP-168.

---

## Session Context

WP-167 added optional `villainDeckTwistCount` / `villainDeckBystanderCount` to
`SchemeSchema` and wired `convert-cards-v15.mjs` to apply them from
`inputs/scheme-deck-counts.json` (D-16702/D-16703). It populated exactly **one**
entry ‚Äî `core.midtown-bank-robbery ‚Üí { 8, 12 }` ‚Äî as a proof of pipeline. Every
other scheme emits neither field, so WP-168's engine resolution falls back to
the locked defaults: **8** twists (D-1411) and **`numPlayers`** bystanders
(D-1412).

A data-curation investigation against the printed Setup text in all 40
`data/cards/*.json` found that the 8-twist default is correct for only about a
third of schemes; roughly two-thirds print a different fixed twist count
(7, 9, 10, 11, 6, 5, 4, ‚Ä¶). The `numPlayers` bystander default is correct for
almost every scheme ‚Äî only a small tail prints an explicit villain-deck
bystander count, including one scheme that prints **zero** (where the default
would wrongly add `numPlayers`). This packet curates the input file so the
registry data matches the printed setup, and fixes two latent gaps in the
WP-167 pipeline that block that curation.

---

## Goal

After this session, `inputs/scheme-deck-counts.json` carries a curated
villain-deck twist count for every scheme whose printed Setup specifies a single
fixed twist count other than 8, and a curated bystander count for every scheme
whose printed Setup specifies an explicit villain-deck bystander count
(including the explicit-zero case). All 40 `data/cards/*.json` are regenerated so
those schemes carry the printed counts; schemes that match the defaults, or
whose twist count is player-count-dependent, emit neither field and keep the
engine fallback. Two pipeline fixes make this possible and correct: the
`applySchemeDeckCounts` overlay assigns each count **independently** (an omitted
count keeps its engine default rather than being forced), and `apply-card-counts.mjs`
gains the same scheme-deck-count overlay so the 4 outlier sets
(`2099`, `amwp`, `wpnx`, `wtif`) can be curated at all. New fields remain
additive and optional, so every set file still validates.

- Full file contents for every new or modified file ‚Äî no diffs, no snippets
- Human-style code per `docs/ai/REFERENCE/00.6-code-style.md`

**Packet-specific:**
- **No schema change.** `villainDeckTwistCount` / `villainDeckBystanderCount`
  already exist (WP-167 / D-16702). Do not touch `packages/registry/src/schema.ts`.
- **No engine change.** WP-168 consumes these counts; this packet only supplies
  data and pipeline plumbing.
- `data/cards/*.json` are **generated** ‚Äî never hand-edit them. Change the input
  file and/or converter, then regenerate.
- `villainDeckTwistCount` / `villainDeckBystanderCount` are the locked field
  names; never abbreviate or rename (Rule 14).
- **Curation rule (twists):** encode `villainDeckTwistCount` only for a scheme
  whose printed Setup specifies a single **fixed** twist count **other than 8**.
  A scheme printing exactly 8 stays unencoded (default already correct). Never
  invent a count ‚Äî extract it from the committed Setup line.
- **Carve-out rule (twists):** a scheme whose printed twist count is
  **player-count-dependent**, **additive-per-player**, or otherwise **not a
  single constant** is **NOT encoded** (see Out of Scope). A single integer field
  cannot represent it, and WP-168 reads a single integer.
- **Disambiguation (what counts as a single fixed villain-deck count):** encode
  only when the Setup names a number of twists **shuffled into the villain deck**
  that is constant regardless of player count and conditions. Twists placed
  **outside** the villain deck (e.g. "N additional Twists next to this Scheme")
  are **excluded from the count, not a carve-out trigger** ‚Äî encode only the
  in-deck constant (e.g. Killbots prints "5 Twists ‚Ä¶ 3 additional Twists next to
  this Scheme" ‚áí encode `5`). Force a **carve-out** when the in-deck count itself
  is player-count-dependent, additive/subtractive per player, or conditionally
  extended ("add a 9th Twist if ‚Ä¶"). When in doubt, carve out.
- **Curation rule (bystanders):** encode `villainDeckBystanderCount` only for a
  scheme whose printed Setup specifies an **explicit** villain-deck bystander
  count, including an explicit **zero** ("No Bystanders in the Villain Deck" ‚áí
  `0`). A scheme that says nothing about villain-deck bystanders stays unencoded
  (the `numPlayers` default is correct).
- **At-least-one-count rule (relaxed contract, D-16803):** an entry in
  `scheme-deck-counts.json` must carry **at least one** of the two counts; it may
  carry only one. An omitted count keeps that scheme's engine default. The
  converter assigns each count independently and must never write an
  `undefined`-valued key.
- **Loud-fail contract (preserved + extended):** matching is exact `schemeSlug`
  equality, never fuzzy or name-normalized. Any entry whose slug matches no
  scheme in the set throws a full-sentence error naming the set and slug and
  exits non-zero, **before** that set's output is written. This already holds in
  `convert-cards-v15.mjs`; `apply-card-counts.mjs` must gain the identical guard
  for the 4 outlier sets.
- **Empty-entry loud-fail:** an entry present for a `schemeSlug` but carrying
  neither count is malformed input. Both converters throw a full-sentence error
  naming the set and slug and exit non-zero ‚Äî never silently skip. `// why:`
  enforces the at-least-one-count contract (D-16803).
- **Field placement invariant:** the converter appends the count field(s) **after
  the scheme's existing keys** (matching WP-167's Midtown output ‚Äî i.e. after
  `cards`), with `villainDeckTwistCount` before `villainDeckBystanderCount` when
  both are present. It must not reorder or move any existing scheme key. `// why:`
  stable key order keeps regeneration diff-free.
- **Mutation boundary:** `applySchemeDeckCounts` and the `apply-card-counts.mjs`
  overlay mutate **only** the two count fields on the matched scheme, in place;
  they must not clone, reconstruct, or re-serialize the `scheme` object or touch
  any other key. `// why:` prevents accidental key reordering or loss of unrelated
  fields.
- **Clean-diff / determinism:** new fields are appended in the existing fixed
  position; no key reordering or re-sorting. Regenerating twice yields zero diff
  (idempotent).
- **No unintended mutation:** the only deltas in regenerated `data/cards/*.json`
  are the curated scheme `villainDeckTwistCount` / `villainDeckBystanderCount`.
  WP-167's villain `copies`, `alwaysLeads[]` / `ledBy[]`, and every hero /
  keyword / image URL / ability / henchman / `physicalCards[]` field are
  unchanged.

**Session protocol:**
- If a scheme's Setup text is ambiguous about whether a printed count is a single
  constant or player-count-dependent, classify it as a **carve-out** (do not
  encode) and record it in the D-16804 carve-out list ‚Äî never guess a flat value.

**Locked contract values (relevant subset):**
- **Engine fallbacks (do not encode when the printed value equals these):**
  twists `8` (D-1411); bystanders `numPlayers` (D-1412).
- **Field names:** `villainDeckTwistCount`, `villainDeckBystanderCount`.
- **Input shape:** `{ setAbbr: { schemeSlug: { villainDeckTwistCount?, villainDeckBystanderCount? } } }`
  (at least one count present per entry, per D-16803).
- **Midtown Bank Robbery (existing, keep):** `core.midtown-bank-robbery ‚Üí
  { villainDeckTwistCount: 8, villainDeckBystanderCount: 12 }`. (Twist 8 is the
  default but is retained because the entry also carries the bystander override.)
