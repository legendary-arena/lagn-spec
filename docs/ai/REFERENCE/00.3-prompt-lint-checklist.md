# 00.3 — Prompt Lint Checklist
# Legendary Arena · Work Packet Quality Gate

> **REFERENCE DOCUMENT — Not an execution prompt. No output required.**
>
> **Purpose:** Every Work Packet must pass this checklist before it is handed
> to Claude Code for execution. A packet that fails any applicable section
> must be fixed first. Executing a failing packet is the primary cause of
> wasted sessions and context drift.
>
> **When to run this checklist:**
> - Before executing any Work Packet for the first time
> - After modifying an existing Work Packet
> - When a session produces unexpected output — lint the packet retroactively
>   to find the gap that caused it
>
> **Relationship to 00.1:** This checklist enforces the Work Packet structure
> and constraints defined in `00.1-master-coordination-prompt.md`. When the
> two documents conflict, 00.1 is authoritative — update this checklist to match.
>
> **Subordination:** This checklist is subordinate to:
> - `docs/ai/ARCHITECTURE.md` (system architecture — Section 1, Section 5)
> - `.claude/rules/*.md` (runtime enforcement during execution)
>
> The checklist gates Work Packet quality before execution.
> The `.claude/rules/*.md` files enforce constraints during execution.
> If this checklist conflicts with either authority, the checklist must be
> updated to match — not the other way around.

## §1 — Work Packet Structure

Every Work Packet must contain all of the following sections.
A missing section is an automatic FAIL — do not mark N/A.

- [ ] `## Goal` — one paragraph describing what will exist after the session
- [ ] `## Assumes` — explicit list of prior packets, files, and external state
- [ ] `## Context (Read First)` — specific docs and sections Claude must read
- [ ] `## Scope (In)` — explicit bullet list of what is in scope
- [ ] `## Out of Scope` — explicit bullet list of what must not be touched
- [ ] `## Files Expected to Change` — every file to be created or modified
- [ ] `## Non-Negotiable Constraints` — flat bullet list with engine-wide and packet-specific constraints (see §2)
- [ ] `## Acceptance Criteria` — 6–12 binary, observable checks
- [ ] `## Verification Steps` — exact pnpm commands with expected output
- [ ] `## Definition of Done` — checklist including STATUS.md, DECISIONS.md, WORK_INDEX.md updates

❌ FAIL if any section is missing
❌ FAIL if any section is present but empty
❌ FAIL if `## Out of Scope` is missing — an empty out-of-scope list is not acceptable;
   at least two things that might seem related must be explicitly excluded

## §2 — Non-Negotiable Constraints Block

The `## Non-Negotiable Constraints` section uses a flat bullet list organized
into engine-wide constraints, packet-specific constraints, session protocol,
and locked contract values. This matches the structure in `PACKET-TEMPLATE.md`.

- [ ] Section contains **Engine-wide** constraints (always apply — do not remove)
- [ ] Engine-wide constraints require full file contents for every new or modified file
- [ ] Engine-wide constraints explicitly forbid diffs and snippets
- [ ] Engine-wide constraints state: ESM only, Node v22+
- [ ] Engine-wide constraints state: Human-style code — see `docs/ai/REFERENCE/00.6-code-style.md`
- [ ] Section contains **Packet-specific** constraints, e.g.:
  - "Read-only — no files may be created or modified"
  - "No Math.random()"
  - "No WebSocket logic"
- [ ] Constraints do not contradict the packet body
- [ ] Section contains **Session protocol** (stop and ask on unclear items)
- [ ] Section contains **Locked contract values** where applicable

❌ FAIL if partial output (diffs, snippets, "show only the changed section") is permitted
❌ FAIL if constraints do not reference `00.6-code-style.md`

## §7 — Dependency Discipline

- [ ] No new npm dependencies are introduced without explicit documentation
- [ ] If a new dependency is required:
  - [ ] Exact package name and version (or range) is stated
  - [ ] The packet requires `package.json` to be updated
  - [ ] The full updated `package.json` is required as output
- [ ] No "check if X is installed, otherwise use Y" logic —
      the packet must make a hard decision
- [ ] Forbidden packages are explicitly excluded where relevant:

Auth0 / Clerk / Passport remain forbidden.

❌ FAIL if a forbidden package could be used without explicit rejection
❌ FAIL if a new dependency is introduced without version and `package.json` update

## §8 — Architectural Boundaries

> Architectural boundaries are authoritatively defined in
> `docs/ai/ARCHITECTURE.md` — Section 1 ("Monorepo Package Boundaries",
> "Package Import Rules") and Section 5 ("Package Dependency Rules").
> The synthesized enforcement view is in `.claude/rules/architecture.md`
> ("Layer Boundary (Authoritative)").
> Runtime enforcement is handled by `.claude/rules/*.md`.
> This section gates Work Packet quality — it does not redefine boundaries.

- [ ] The packet must not instruct Claude to add logic to a layer that
      violates the Layer Boundary defined in `.claude/rules/architecture.md`

### Backend
- [ ] PostgreSQL used only for rules/configuration data — never live game state
- [ ] `G` and `ctx` are never stored in the database
- [ ] No database queries inside boardgame.io move functions
- [ ] No WebSocket logic outside `boardgame.io Server()`
- [ ] `pg` pool used for all database connections (not a single client)

### Game Logic
- [ ] All randomness uses `ctx.random.*` — never `Math.random()`
- [ ] `G` is JSON-serializable (no class instances, no functions)
- [ ] Moves are pure functions (same inputs → same outputs, always)
- [ ] Deck builder logic lives in `src/game/deckBuilder.mjs`, not `legendary.mjs`

### Frontend
- [ ] Components do not fetch R2 data independently — use `cardRegistry.js`
- [ ] Components contain no game logic
- [ ] boardgame.io client is in `src/game/client.js`, not in Vue components
- [ ] `boardgame.io/react` is never imported
- [ ] All moves dispatched through `gameStore.move()`

### Scripts and Diagnostics
- [ ] Diagnostic scripts are explicitly marked read-only in their contract
- [ ] Seeder scripts use upserts only — never `DELETE`
- [ ] Migration scripts are idempotent

❌ FAIL if any boundary is violated or left ambiguous in the packet

## §16 — Code Style (Human-Style / Junior Maintainability)

> Code style rules are defined in `docs/ai/REFERENCE/00.6-code-style.md`.
> Runtime enforcement is handled by `.claude/rules/code-style.md`.
> This section gates Work Packet output quality.

Enforces the rules in `docs/ai/REFERENCE/00.6-code-style.md`.
Every file produced by the packet must satisfy these checks.

### §16.1 — No premature abstraction
- [ ] No helper function created for logic that appears fewer than 3 times
- [ ] No factory functions for objects created in only one place
- [ ] No higher-order functions (functions returning functions) unless the
      framework explicitly requires them

❌ FAIL if shared helpers are created for logic used only once or twice

### §16.2 — Explicit control flow
- [ ] No nested ternaries — multi-branch logic uses `if/else` blocks
- [ ] No `Array.reduce()` with complex branching — use `for...of` instead
- [ ] No dynamic property access (`obj[key]`) when keys are known at write time

❌ FAIL if control flow requires more than one read to understand

### §16.3 — Readable names
- [ ] No abbreviations in variable or function names
      (`config` not `cfg`, `villainGroup` not `vg`, `error` not `e`)
- [ ] Boolean names start with `is`, `has`, or `can`
- [ ] Loop variables are descriptive: `for (const villainGroup of villainGroups)`

❌ FAIL if a junior developer would need to look up what a name means

### §16.4 — Small functions
- [ ] No function exceeds 30 lines (not counting JSDoc)
- [ ] Each function does one thing; its name describes that one thing
- [ ] Every function has a JSDoc comment documenting parameters, return value,
      and error conditions

❌ FAIL if any function cannot fit on a single screen

### §16.5 — Comments explain WHY
- [ ] Non-obvious code has a `// why:` comment explaining the decision
- [ ] No comments that restate what the code does
- [ ] Unusual defaults and magic numbers are explained inline

❌ FAIL if a junior developer would not understand why a block exists

### §16.6 — No magic imports
- [ ] No `import * as` — use named exports only
- [ ] No barrel re-exports that hide where something lives
- [ ] Every import path is a real, unambiguous file path

❌ FAIL if the import structure obscures where code lives

### §16.7 — Error messages are full sentences
- [ ] All `throw new Error(...)` messages identify what failed and what to check
- [ ] No single-word or terse messages (`throw new Error('invalid')`)

❌ FAIL if any error message leaves a junior developer guessing what broke

## §18 — Prose-vs-Grep Discipline

A Work Packet, Execution Checklist, or session prompt that includes
grep-based Verification Steps **must not restate forbidden tokens
verbatim in adjacent prose** when the grep is literal-string-scoped
(i.e., lacks `from ['"]` or function-call scoping that would exclude
documentation hits).

### §18.1 — When this rule applies

This rule applies when a WP / EC / session prompt includes:

- A `## Verification Steps` grep pattern targeting tokens like
  `Math\.random`, `Date\.now`, `performance\.now`, `new Date(`,
  `boardgame\.io`, `@legendary-arena/registry`, `apps/server`, or any
  other forbidden import / call identifier.
- AND prose in the same file OR in any file under the Verification
  Step's grep path (JSDoc module headers, markdown documentation,
  commentary blocks) that discusses those tokens in any context —
  governance rationale, forbidden-list enumeration, prohibited-use
  explanation, or precedent citation.

### §18.2 — Required form

Prose discussing forbidden tokens **must cite the governing
decision** (e.g., *"see D-3701 for the full forbidden list of
imports, I/O surfaces, wall-clock reads, and randomness sources"*)
rather than enumerate the tokens verbatim. Decision-ID citations
preserve governance content auditably without creating false-positive
grep hits.

### §18.3 — Failure symptom

If a Verification Step grep returns matches that are **all inside
prose** (JSDoc comments, markdown paragraphs, block comments), the
rule has been violated. The grep is not wrong; the prose is. Soften
the prose to cite a D-entry and re-run the verification step.

### §18.4 — Rule precedent

Surfaced during WP-037 / EC-037 execution (2026-04-22) when
`packages/game-engine/src/beta/beta.types.ts` module-header JSDoc
enumerated forbidden tokens (`boardgame.io`, `@legendary-arena/registry`,
`apps/server/`, `Math.random`, `Date.now`, `performance.now`,
`new Date()`) and tripped the session prompt's Verification Steps 5b
and 6 as false positives. Resolution: prose softened to cite D-3701
for the full forbidden list. No governance intent lost — D-3701
already enumerates the forbidden tokens exhaustively. Documented in
[`docs/ai/post-mortems/01.6-WP-037-public-beta-strategy.md`](../post-mortems/01.6-WP-037-public-beta-strategy.md)
§8 (reality reconciliation) and §10 (fixes applied during
post-mortem).

## §19 — Bridge-vs-HEAD Staleness Rule

Any **repo-state-summarizing artifact** — session-context bridges,
status reports, cross-session hand-off documents, or any document
whose content is a snapshot of commit history or file state — must
be reconciled against `HEAD` in the same atomic shell operation as
its commit. The window between a drafting-time `git log` check and
the subsequent `git commit` is a staleness vector; parallel sessions
may land commits in that window, and the committed artifact will
canonically reference a state that does not match actual `HEAD`.

### §19.1 — When this rule applies

This rule applies when authoring or committing:

- `docs/ai/session-context/session-context-wp*.md` bridges
- `docs/ai/STATUS.md` §Current State entries
- `docs/ai/post-mortems/01.6-*.md` §2 Final Baseline / §4 Scope-Lock
  Adherence / §5 Pre-Flight RS-# Dispositions sections
- Commit messages that enumerate parent commits, intervening state,
  or "recent commits" chains
- Any document section labeled "Commits landed this session",
  "Repo-State Anchor", "What changed", "Recent commits", or
  equivalent

### §19.2 — Required discipline

Before staging such an artifact, **re-run `git log --oneline -N` in
the same shell** as the subsequent `git commit`. If the working tree
or `HEAD` has moved since the drafting-time snapshot, reconcile the
artifact's content against actual `HEAD` before committing. The goal
is a commit whose authored content is factually accurate **at the
moment of its own landing**, not at an earlier snapshot time.

### §19.3 — Failure symptom

If an authored artifact claims "pending" or "queued" status for
something that has already landed at `HEAD`, references commit
hashes that are not immediate ancestors of `HEAD`, or enumerates a
commit list that skips parallel-session commits landed between
drafting and commit, the rule has been violated. Two remediation
paths:

1. **Preferred:** Revert the stale commit and re-author the artifact
   against current `HEAD`. Appropriate when the stale commit has not
   yet been pushed or built upon.
2. **Acceptable:** Add a follow-up reconciliation commit that
   explicitly updates the stale sections with strikethrough
   preservation of the original claims and a terminal Reconciliation
   Note documenting which sections were corrected and why.

### §19.4 — Rule precedent

Surfaced during the WP-085 governance-bundle session (2026-04-22)
when [`docs/ai/session-context/session-context-wp038.md`](../session-context/session-context-wp038.md)
was authored against `main @ 8b84587` but committed at `main @
f53af27`. Five parallel-session commits landed in the drafting
window (`604eaaa` through `485d0d0`), resolving two items that the
bridge's Pending Follow-Ups section still claimed as pending.
Corrected at SPEC commit `98c8a3f` via the "acceptable" remediation
path (reconciliation commit with strikethrough + terminal note),
since the stale commit had not been built upon and revert-plus-
re-author would have discarded no downstream work. The Reconciliation
Note at the foot of the bridge documents the specific sections
corrected and why.

- [ ] User profile or account funding attribution surfaces (per WP-097 §C)
- [ ] Tournament-specific funding-channel integrations (Open Collective,
      PayPal, equivalent per `docs/TOURNAMENT-FUNDING.md`)
- [ ] User-visible copy referencing "donate", "support tournaments",
      "tournament funding", or equivalent terms — **as part of a
      proposed or implemented user interaction** (per WP-097 §F G-6).
      The "user interaction" qualifier scopes the trigger to copy that
      the user reads or acts on; it does not capture incidental
      mentions in code comments, error messages quoting policy text,
      or developer-facing documentation strings.

A WP that touches **none** of the above MAY mark §20 as **N/A**, but
MUST include a one-line justification per WP-097 §F *"Applicability
is declared, never inferred."* A bare `N/A` is a §20 lint **FAIL**.
The justification MUST name *why* none of the §20.1 trigger surfaces
are present (e.g., *"docs-only governance update; no UI surfaces, no
user-visible copy, no funding channels referenced"*). Tautological
placeholders such as *"N/A — not applicable"* or *"this WP does not
trigger §20"* are also a §20 lint FAIL — they restate the conclusion
without naming the reason.

- **Governance-doc exclusion.** Pure governance / reference documents
  (other WPs, ECs, STATUS updates, DECISIONS entries,
  `docs/ai/REFERENCE/*` files) that *mention* funding conceptually —
  for example, by citing WP-097, D-9701, or §20 itself — without
  defining, proposing, or implementing user-facing surfaces do **not**
  trigger §20. The N/A path with justification still applies; this
  carve-out only confirms that conceptual mention alone is not a
  trigger. Without it, reviewers risk over-applying §20 to every
  meta-document that names the funding policy.
- **Analytical / retrospective mention is not a trigger.** Mentioning
  funding in a purely analytical or retrospective context —
  summarizing WP-097, citing D-9701, describing historical policy
  decisions, or post-mortem analysis of an already-shipped surface —
  without proposing, defining, or implementing any user-visible
  surface does **not** by itself trigger §20. This complements the
  prior carve-out: that one lists the document classes that are exempt
  by their own nature; this clarification covers the kind of mention
  that is exempt regardless of which document it appears in. The N/A
  path with justification still applies.

### §20.2 — Required content of `## Funding Surface Gate`

When triggered, the WP must contain a `## Funding Surface Gate`
block (or fold the gate into its `## Vision Alignment` block per
WP-097 §F Audit discipline) with:

- [ ] **Surface inventory** — explicit list of every implemented
      surface mapped to its WP-097 §A / §B / §C source.
- [ ] **G-1 through G-7 disposition** — per implemented surface, an
      explicit pass / N/A statement with rationale, citing WP-097 §F
      by ID rather than paraphrasing. **Partial mapping is a FAIL**
      equivalent to missing the gate entirely — listing G-1 through
      G-5 and silently omitting G-6 / G-7 (or any other subset) does
      not satisfy this requirement. All seven gate items must be
      addressed; an item that genuinely does not apply must be
      explicitly marked N/A with rationale, not omitted.
- [ ] **Copy deferral declaration** — explicit statement that
      user-visible copy matches `docs/TOURNAMENT-FUNDING.md §Public
      Blurb (Reusable)` verbatim, OR a `D-NNNN` carve-out citation
      per WP-097 §F *"Copy deferral."*
- [ ] **Authority citation** — `WP-097`, `D-9701`, and `D-9801`
      cited as the authority chain.

❌ FAIL if §20 is triggered and the WP has no `## Funding Surface Gate` section (or equivalent block inside `## Vision Alignment`)
❌ FAIL if the gate block lacks the G-1 through G-7 disposition mapping per WP-097 §F
❌ FAIL if the disposition mapping covers only a subset (Partial mapping is a FAIL equivalent to missing the gate entirely)
❌ FAIL if §20 N/A is declared with a bare or tautological justification
❌ FAIL if user-visible copy paraphrases the Public Blurb without a `D-NNNN` carve-out citation

### §20.3 — What §20 does **not** do

- It does not redefine WP-097 §F G-1..G-7. WP-097 is the source of
  truth; §20 is the trigger that forces the mapping.
- It does not replace §17 Vision Alignment. A WP that triggers both
  §17 and §20 must satisfy both.
- It does not apply to WPs that touch no §20.1 trigger surface.
- It does not apply retroactively. Pre-existing WPs (WP-097 included)
  are not required to add `## Funding Surface Gate` blocks.
- It does not imply or require automated static analysis tooling.
  §20 is a documentation-level lint gate enforced by the WP author
  and reviewer at draft time, the same as §17, §18, and §19. Future
  tooling may automate any of the §20 checks, but the absence of
  such tooling is not itself a §20 failure; the gate is satisfied by
  author / reviewer discipline alone.

## §21 — API Catalog Update

> Introduced by WP-118 (single `SPEC:` commit, 2026-04-30) per
> `D-11804`. Mirrors §17 / §20 in structure: §21.1 lists trigger
> conditions; §21.2 lists required content; §21.3 lists FAIL
> conditions; §21.4 documents the N/A path. The authoritative
> catalog this section enforces is
> [`docs/ai/REFERENCE/api-endpoints.md`](api-endpoints.md). The
> companion enforcement rule lives at
> [`.claude/rules/work-packets.md`](../../../.claude/rules/work-packets.md).

### §21.1 — Trigger conditions

A Work Packet **must** update `docs/ai/REFERENCE/api-endpoints.md` in
the same commit that lands the code change if it touches any of the
following surfaces:

- [ ] Adds a new HTTP endpoint to `apps/server` (whether registered
      directly via `router.<verb>(...)` in `apps/server/src/server.mjs`
      or via a `register*Routes(...)` helper invoked from
      `server.mjs`).
- [ ] Modifies the URL, method, request shape, response shape, status
      codes, or auth posture of an existing HTTP endpoint already in
      the catalog.
- [ ] Removes or deprecates an HTTP endpoint already in the catalog.
- [ ] Changes the `Status` of an existing endpoint (e.g.,
      `Shipped-but-unwired` → `Wired` when a deferred route registers
      with `Server({...}).router`; `Pending: WP-NNN (...)` → `Wired`
      when the drafting WP executes).
- [ ] Adds, modifies, removes, or changes the status of a library
      function reachable via direct import from `apps/server/src/**`
      that the catalog records as `Library-only`.

### §21.2 — Required content of the catalog update

When triggered, the catalog update MUST satisfy all of the following:

- [ ] **Replace-whole-row merge semantics (per D-11804).** The
      affected catalog row is replaced **entirely** in the same
      commit. Partial-update of an existing row (changing one column
      while leaving others stale) is FAIL. Status transitions are
      full-row replacements, never field-level edits — every status
      change carries the row's other columns forward intact (or
      re-populates them for the new shape) so reviewers compare a
      complete-row before vs. complete-row after.
- [ ] **Status closed set.** Every catalog row's `Status` column
      carries exactly one of the four values `Wired |
      Shipped-but-unwired | Library-only | Pending` (per the catalog's
      Status Taxonomy header). `Pending` rows use the format
      `Pending: WP-NNN (STATE YYYY-MM-DD)` to name both the drafting
      WP and its state at catalog-write time.
- [ ] **Auth closed set (per D-9905).** Every catalog row's `Auth`
      column carries exactly one of the three values `guest |
      handle-required | authenticated-session-required` (per the
      catalog's Auth Taxonomy header).
- [ ] **Canonical field names.** Every field name appearing in the
      row's request or response schema (`accountId`, `handle`,
      `matchId`, `replayHash`, etc.) matches the spelling in
      [`docs/ai/REFERENCE/00.2-data-requirements.md`](00.2-data-requirements.md)
      exactly.
- [ ] **Authorizing WP citation.** The row's `Authorizing WP` column
      names the WP that authorized the surface (e.g., `WP-011`,
      `WP-102`, `WP-115`). For boardgame.io built-ins surfaced by the
      framework with no explicit project WP, the value is
      `boardgame.io built-in`.
- [ ] **Drift / Undocumented annotations.** If an endpoint exists in
      code but no WP documents it, the row carries an `Undocumented:`
      annotation (auth posture / request shape / response shape
      inferred from code) and the WP body flags the gap so a
      follow-up WP can backfill the governance record. If an
      endpoint exists in code but contradicts its authorizing WP's
      contract, the row carries a `Drift: <one-line description>`
      annotation; reconciliation is a follow-up WP, not in scope of
      the triggering WP.

### §21.3 — FAIL conditions

❌ FAIL if §21 is triggered and the WP does not update
`docs/ai/REFERENCE/api-endpoints.md` in the same commit (a missing row
for a new endpoint; a missing row update for a modified endpoint; a
missing row removal for a removed endpoint).

❌ FAIL if any catalog row's `Status` column carries a value outside
the four-value closed set `Wired | Shipped-but-unwired | Library-only
| Pending`.

❌ FAIL if any catalog row's `Auth` column carries a value outside the
three-value closed set `guest | handle-required |
authenticated-session-required` (per D-9905).

❌ FAIL if any field name in a catalog row's request or response
schema diverges from the canonical spelling in
`docs/ai/REFERENCE/00.2-data-requirements.md`.

❌ FAIL if a catalog update partially edits an existing row instead
of replacing it wholesale (per D-11804 replace-whole-row merge
semantics — partial-update is FAIL).

❌ FAIL if the WP claims §21 N/A while touching any §21.1 trigger
surface (the N/A path applies only when no §21.1 surface is
touched).

### §21.4 — N/A path

A WP that touches **none** of the §21.1 trigger surfaces MAY mark §21
as **N/A**, but MUST include a one-line justification per the §17 /
§20 N/A discipline. A bare `N/A` is a §21 lint FAIL. The justification
MUST name *why* none of the §21.1 trigger surfaces are present (e.g.,
*"docs-only governance update; no HTTP endpoints touched, no
`apps/server/src/**` library functions added or modified"*).
Tautological placeholders such as *"N/A — not applicable"* or *"this
WP does not trigger §21"* are also a §21 lint FAIL — they restate the
conclusion without naming the reason.

## How to Lint a Work Packet

1. Open the Work Packet and this checklist side by side
2. Run §1 first — if any required section is missing, stop and add it
3. Work through §2–§15 in order, marking each checkbox
4. Run §16 last — code style is easiest to check after structural issues are resolved
5. Run §17 against the trigger list — if any §17.1 surface is touched, the WP
   must include `## Vision Alignment`; otherwise mark §17 as N/A explicitly
6. Run §18 against every Verification Step that uses a literal-string-scoped
   grep — cross-check that adjacent prose (JSDoc headers, markdown commentary
   under the grep's path) cites a `D-xxxx` entry instead of enumerating the
   forbidden tokens verbatim. If the prose is verbatim, soften it to cite the
   governing decision and re-run the Verification Step
7. Mark §11 and §12 as N/A if the packet does not touch authentication or tests
7. If any ❌ FAIL condition triggers: fix the packet, then re-lint from §1

**After execution**, verify the output against the packet's own
`## Acceptance Criteria` checklist. If any output violates a §16 rule, reject
it and say:
```
This violates Code Style Rule §16.N from 00.6-code-style.md.
Please rewrite [specific function/section] to follow that rule.
```

**If a session produces unexpected output**, re-lint the packet retroactively.
The gap in the checklist that allowed the unexpected output should be fixed
before the next session runs.

## Drift Detection During Execution

If Claude's output deviates from the authoritative documents mid-session,
stop immediately and do not accept the output. Use this response:

```
DRIFT DETECTED: [describe the contradiction].
The authoritative source is [document name, section number].
Please re-read that section and revise your output to match it.
```

context drift detection protocol.

underscore card URLs). Not a Work Packet and not part of any gate — delete
freely if it stops being useful.
