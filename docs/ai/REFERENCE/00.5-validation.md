## Code Style Mandate

All output must follow `00.6-code-style.md`. Key rules for this prompt:

- **The script is a linear top-to-bottom sequence** — four named async functions
  (one per phase), called in order from a `main()` function. No class, no factory,
  no runner abstraction.

- **Phase functions are named by what they check**, not by their phase number:
  `checkRegistry()`, `checkMetadataForAllSets()`, `spotCheckImages()`,
  `findCrossSetDuplicateSlugs()`.

- **Validation checks are explicit `if` blocks** — each check is its own `if`
  with a full-sentence message pushed to the errors or warnings array:
  ```js
  // BAD — compressed into one expression
  if (!setJson.heroes?.length) errors.push(`[${abbr}] missing heroes`);

  // GOOD — separate checks with clear, distinct messages
  if (!setJson.heroes) {
    errors.push(`[${setAbbreviation}] metadata JSON is missing the "heroes" key entirely.`);
  } else if (setJson.heroes.length === 0) {
    errors.push(`[${setAbbreviation}] "heroes" array is present but empty.`);
  }
  ```

- **No abbreviations in variable names** — use `setAbbreviation` not `abbr`,
  `mastermindsData` not `mm`, `villainGroupData` not `vg`, `errorMessage` not `msg`.
  The only exception: the literal `abbr` field value read from JSON may be stored
  as `setAbbreviation` once and reused.

- **Slug checks use explicit `for...of` loops**, not `.filter()` chains.

- **The 50ms delay between sets is a named constant with a `// why:` comment**:
  ```js
  const DELAY_BETWEEN_SETS_MS = 50;
  // why: R2 rate-limits aggressive parallel fetches. A 50ms pause between
  // sequential set fetches keeps us well under the limit for 40+ sets.
  await sleep(DELAY_BETWEEN_SETS_MS);
  ```

- **`// why:` comments** are required on:
  - The HEAD request strategy for image checks
  - The sample-only approach for image checks
  - The exit code behavior
  - The 50ms delay constant

## Deliverables

### 1. Validation script (`scripts/validate-r2.mjs`)

A standalone Node.js ESM script. Read-only throughout.
Written as four named async phase functions called from `main()`.

#### Phase 1 — `checkRegistry()`

- Fetch `metadata/sets.json` from R2 root
- Verify valid JSON containing an array of set abbreviations
- Print total set count and list of abbreviations found
- Return the array of abbreviations for use by subsequent phases

#### Phase 2 — `checkMetadataForAllSets(setAbbreviations)`

For each abbreviation, fetch `/metadata/{abbr}.json` and check the following.

Read `00.2-data-requirements.md §1` before implementing these checks — the
field names here are authoritative and must match exactly.

**Structural checks (ERROR if missing or wrong type):**

Top-level set JSON:
- `heroes` key is present and is an array (may be empty for scheme-only sets)
- `villains` key is present and is an array
- `masterminds` key is present and is an array
- `schemes` key is present and is an array

Hero deck records (each item in `heroes[]`):
- Has `slug` — string, non-empty
- Has `name` — string, non-empty
- Has `cards` — array with length between 1 and 14
  (hero decks vary in size — do not hard-code 1–4)
- Each card in `cards[]` has `imageUrl` — string, starts with `https://`
- Each card in `cards[]` has `cost` — number
- Each card in `cards[]` has `hc` — string, non-empty

Mastermind records (each item in `masterminds[]`):
- Has `slug` — string, non-empty
- Has `name` — string, non-empty
- Has `vp` — number (victory points when defeated, per 00.2 §1.3)
- Has `cards` — array with length >= 1

Villain group records (each item in `villains[]`):
- Has `slug` — string, non-empty
- Has `name` — string, non-empty
- Has `cards` — array with length >= 1
- Each card in `cards[]` has `slug` — string, non-empty

Scheme records (each item in `schemes[]`):
- Has `slug` — string, non-empty
- Has `name` — string, non-empty

Each check is its own `if` block with a full-sentence error message that
includes the set abbreviation and the specific field or value that is missing
or invalid. Do not compress multiple field checks into a single condition.

**Known data quality issues — WARN, do not ERROR (per 00.2 §12):**

- Any `abilities` array entry that is exactly `"[object Object]"`:
  warn with the set, card name, and position index
- Any scheme with `id: null`:
  warn that this is a known transform reverse-side card pattern — not a bug

**Slug format checks (WARN if violated):**

- All slugs are lowercase with hyphens only — no spaces, underscores, or uppercase
- No duplicate slugs within a single set's JSON (check across all card types)
- No slug is an empty string

Check slugs using an explicit `for...of` loop over each card type array —
not `.flatMap()` or `.map().flat()`.

**Relationship checks (WARN if unexpected):**

- If a mastermind has a non-empty `alwaysLeads` array, each entry must match
  a `slug` in that set's `villains` array or `henchmen` array
- If `alwaysLeads` is present but empty on a mastermind, that is acceptable —
  some masterminds lead any group (do not warn)

#### Phase 3 — `spotCheckImages(setAbbreviations)`

**Read `00.2-data-requirements.md §3.2` before implementing this phase.**
The URL construction rules and suffix legend are defined there.
Do not construct URLs from memory — read the document.

For each set, HEAD-check one image per card type:
- First mastermind's card: `https://images.barefootbetters.com/{abbr}/{abbr}-mm-{mastermindsSlug}.webp`
- First villain group's first card: `https://images.barefootbetters.com/{abbr}/{abbr}-vi-{groupSlug}-{cardSlug}.webp`
- First hero deck's first card: use the `imageUrl` field directly from the card
  JSON — do not reconstruct the URL; the stored `imageUrl` is the authority
  (per 00.2 §3.2: "always prefer the stored `imageUrl` field")

Do NOT check every image — that is 1,000+ requests. Sample only.
Report any 404s found and the total sample size.

Each HEAD request:
```js
// why: HEAD requests check URL reachability without downloading image data.
// A 200 response confirms the image exists at R2; we do not need the bytes.
const headResponse = await fetch(imageUrl, { method: 'HEAD' });
```

Wrap each HEAD request in its own try/catch block. A network error on a single
image must not abort the entire phase.

#### Phase 4 — `findCrossSetDuplicateSlugs(allSetMetadata)`

Collect all slugs across all sets (heroes, villains, masterminds, schemes) using
an explicit `for...of` loop for each card type — not `.flatMap()`.

Report any slug appearing in more than one set. These cause upsert collisions
in PostgreSQL and must be resolved before seeding.

```js
// why: Cross-set slug duplicates are silent bugs during PostgreSQL seeding.
// An upsert on a duplicate slug overwrites the first set's data with the
// second set's data, producing wrong card records with no error thrown.
```
