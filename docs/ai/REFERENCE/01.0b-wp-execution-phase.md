## Phase 2 — Execution (Session 2, new Claude Code session)

### Step 8: Open execution session

- Open a **fresh** Claude Code session. A new worktree+branch auto-spawns.
- `SessionStart` hook fires (`01.8`). Expected: silent (you just cleaned
  up at the end of Phase 1).
- **If main has moved since pre-flight passed in Phase 1**, REVALIDATE
  before doing any work. This applies when (a) Phase 1 closed days
  ago and other PRs have merged since, or (b) you're running in
  parallel mode and another WP merged after this WP's pre-flight.
  Procedure:

  ```pwsh
  git fetch origin main
  git rebase origin/main   # or merge if rebase introduces noise
  # Re-run pre-flight per 01.4. If verdict is now NOT READY,
  # address the new blockers before executing.
  ```

  See `## Parallel WP execution` below for the full rebase +
  revalidation discipline when running multiple WPs concurrently.

- Open the session-prompt file written in Phase 1 Step 6
  (`docs/ai/invocations/session-wpNNN-*.md`). Paste its content as the
  first message to Claude.
- Claude reads the authority chain per the prompt: CLAUDE.md →
  ARCHITECTURE.md → .claude/rules/ → WP-NNN → EC-NNN.
- Apply `01.1-how-to-use-ecs-while-coding.md` for in-execution
  discipline (one WP per session, EC clauses drive debugging, etc.).

### Step 9: Implement + post-mortem (if triggered)

Execute the work clause-by-clause per the EC. Specific gates fire as
needed:

- **`01.5` runtime wiring** — if a file outside the allowlist needs
  modification to keep tests green, invoke `01.5` per the authorization
  in the session prompt. Document each wiring change.
- **`01.6` post-mortem** — author IF any trigger fires per
  `01.6-post-mortem-checklist.md` (new contract, new builder, new
  abstraction, new runtime wiring, new code category). Most
  post-mortems can be authored inside Session 2; only split out to a
  dedicated session if the work generated substantial open questions.

### Step 10: Commit + governance close

**Two-commit topology is REQUIRED** (per `01.3 §Two-commit topology`).
The implementation commit (Commit A) and the governance-close commit
(Commit B) MUST be separate commits on the branch. Combining them
into a single commit is forbidden — it conflates code review with
governance review and makes per-commit auditability impossible.

A subsequent **squash merge to `main`** at PR-merge time is fine
(repo policy); the two-commit constraint applies to the branch
state, not to `origin/main`'s history.

#### Commit A — the implementation

```
EC-NNN: <imperative subject under 70 chars>
```

Body: what changed, locked values used, any `01.5` wiring invocations,
any post-mortem invocations.

#### Commit B — governance close

```
SPEC: governance close for WP-NNN / EC-NNN
```

Body lists updates. **Files always updated in Commit B:**

- `docs/ai/work-packets/WORK_INDEX.md` — flip the `[ ]` row to `[x]`,
  add Done date, EC commit reference, baselines pre/post
- `docs/ai/execution-checklists/EC_INDEX.md` — Status → `Done`
- `docs/ai/DECISIONS.md` — land any reserved `D-NNNNN` entries
- `docs/05-ROADMAP-MINDMAP.md` — flip the WP's status icon
  (📝 Drafted → ✅ Done, or 📦 Queued → ✅ Done)
- `docs/STATUS.md` — if it tracks WP-level progress

**Commit prefix conventions** (per `01.3`):

| Prefix | When |
|---|---|
| `EC-NNN:` | Implementation commits inside an EC's scope |
| `SPEC:` | Governance-only commits (draft, close, ledger updates) |
| `INFRA:` | Infrastructure / hygiene changes outside any WP/EC scope |

### Step 11: Push + cleanup

> **Done is observable, not merged (D-24026).** Steps 1–3 put code on `main`;
> they do NOT make a *user-facing* packet "done." If the WP's
> `User-Visible Surface` is anything other than `none — infrastructure`, the
> packet is not complete until the change is confirmed **live on that surface**
> after the deploy — a real match on play.legendary-arena.com, the deployed
> cards/dashboard page — with observable evidence (the Definition of Done's
> live-verification item). A merge that turns green CI but is never seen on the
> live surface is a half-shipped packet. For `none — infrastructure` packets,
> there is nothing to observe — STATUS.md says so explicitly and this gate is N/A.

1. Push the execution branch.
2. Open the PR. Title: `WP-NNN: <title>`. Description references the WP
   file and the EC file.
3. Merge to `main` (squash typical).
4. **Verify the branch is safe to delete** (REQUIRED post-merge gate).
   From inside the execution worktree:

   ```pwsh
   node scripts/prune-empty-claude-branch.mjs --verify-current
   ```

   Expected: `VERIFY PASS: ...`. The check accepts **all three GitHub
   PR-merge styles** — merge-commit (true fast-forward / 0 commits
   ahead), squash-merge (PR exists with `state=MERGED` on origin),
   and rebase-merge (same gh fallback path; the original branch tip
   is non-ancestor of main with new SHAs on main). The script's
   "squash-merge pattern" label in its PASS message is a holdover;
   semantically it means "a merged PR is recorded on origin for this
   branch, regardless of merge style." Failures (uncommitted changes,
   unmerged work, missing PR) STOP cleanup until resolved.

5. **Delete the execution branch** (local + remote), same pattern as
   Phase 1 Step 7.
6. Run `node scripts/prune-empty-claude-branch.mjs --report` from
   canonical. Expected: silent.
7. (Optional) If the hook reports any accumulation from earlier
   sessions, run `--execute`.

Phase 2 complete. The implementation is on `origin/main`; both draft
and execution branches are deleted; the worktree state is clean for
the next WP.

## Parallel WP execution

**Sequential is the default. Parallel is permitted only in narrow cases.**

The two-session pattern assumes serial execution of a single WP. The
rules below cover the cases where running multiple WPs in flight at
the same time is genuinely cheaper than serial — but those cases are
narrower than they look, and the operator should choose serial by
default unless one of the narrow-case conditions clearly applies.

### Why sequential is the default (empirical, not theoretical)

**Same-repo parallel work almost always costs more than it saves.**
Even when two WPs touch disjoint code files, they share a governance
surface that they both have to update: `WORK_INDEX.md`,
`EC_INDEX.md`, `DECISIONS.md`, `STATUS.md`, the WP file itself
(when an audit/hardening pass intersects an execution session, as
happened with WP-007b). Every parallel session bumps into this
surface. The "first-to-READY merges first → others rebase + revalidate"
discipline below works in theory but the lived cost is exactly what
WP-007b's 2026-05-11 session experienced: ~50% of operator time goes
to conflict resolution + force-push + rebase cycles, not progress.
The rules manage the friction; they don't eliminate it. **Sequential
within a single repo is structurally simpler and faster in practice.**

**Cross-repo parallel work is genuinely cheap.** Engine repo and
marketing repo have fully independent governance ledgers; nothing
they touch is shared. Two WPs — one per repo — can be in flight
simultaneously without any coordination cost beyond the operator's
own attention budget. Today's WP-148 (engine) / WP-149 (marketing
queued) / WP-150 (engine) sequencing demonstrated this: each repo's
main moved independently, no rebases needed within each repo, clean
merges all around.

**Bottom line:**

- **One repo, one WP at a time** — default, always works
- **Two repos, one WP each** — fine, no extra coordination cost
- **One repo, two+ WPs in parallel** — exception only; needs the
  full discipline below; expect overhead

The rules in this section describe the discipline required when the
exception applies. The mere fact that the rules exist isn't an
invitation to invoke the exception — most days, sequential is the
right call.

### What's structurally parallel-safe

- **Cross-repo work** (engine repo + marketing repo) is fully
  parallel-safe. The repos have independent governance ledgers; no
  shared files. Order of cross-repo merges doesn't matter.
- **Cross-layer work in the same repo** (e.g., `registry-viewer` +
  `game-engine`) is parallel-safe at the *code* level — different
  package trees, different test surfaces, no import cycles by Layer
  Boundary rule. The conflict surface reduces to governance files
  only.

### What WILL conflict in same-repo parallel mode

| File | Why it conflicts |
|---|---|
| `docs/ai/work-packets/WORK_INDEX.md` | Both WPs add rows in the same phase section, often on adjacent lines |
| `docs/ai/execution-checklists/EC_INDEX.md` | Both EC rows land in the same index |
| `docs/ai/DECISIONS.md` | Both reserved `D-NNNNN` from the same `origin/main` tip — collision is hard (same number, different meanings) |
| `docs/STATUS.md` | Running narrative; every WP touches it |
| `docs/05-ROADMAP-MINDMAP.md` | Both flip status icons; conflicts on adjacent lines |

### Rules for parallel mode

1. **First-to-READY merges first.** When multiple WPs reach READY
   simultaneously, the first one through pre-flight + copilot + lint
   gates merges first. Don't "save the easy one for last" — that
   creates an incentive to rush gates on harder WPs.

2. **All other in-flight WPs MUST rebase + revalidate after every
   merge to main.** Procedure, run in each non-merged worktree:

   ```pwsh
   git fetch origin main
   git rebase origin/main
   # Resolve governance-file conflicts mechanically (keep both rows;
   # preserve phase ordering).
   # Re-run pre-flight per 01.4. The previous READY verdict is now
   # stale; the new main may have shifted dependencies or baselines.
   ```

   Pre-flight verdicts go stale when main moves. A READY from before
   another WP merged is no longer valid for execution OR for merging
   the WP itself.

3. **Numbering reservation honors merge order, not draft order.** If
   WP-A and WP-B both reserved `D-13901` from the same starting
   commit, and WP-A merges first, WP-B's `D-13901` is now invalid.
   WP-B MUST renumber to next-free (e.g., `D-13902`) during rebase —
   update every reference in WP-B's WP file, EC file, code comments,
   and `DECISIONS.md` draft entry. This is the worst kind of
   conflict to resolve; minimize by checking origin's draft branches
   for in-flight reservations BEFORE drafting (see "Numbering
   reservation ownership" in 01.0a Step 2).

4. **Cross-repo parallel follows the same rules within each repo
   independently.** Engine repo and marketing repo (`C:\www\legendary-arena-com\`)
   have separate governance ledgers; their `WORK_INDEX`-equivalent
   files don't share state. Each side runs its own
   first-to-READY-merges-first discipline.

5. **No hard concurrency cap.** The limit is operator cognitive load.
   Practically: 2–3 WPs in flight is manageable; beyond that,
   mid-execution scope tends to drift and rebase cost compounds.

### Anti-patterns (parallel-mode specific)

- **Drafting two WPs simultaneously without checking origin for
  in-flight reservations.** Open two draft sessions, both reserve
  `D-13901`, both reach READY, both try to merge. The second-to-merge
  has a mass-renumber problem touching every WP/EC/code/DECISIONS
  reference.
- **Skipping the rebase + revalidate step between merges.** "It's
  just docs" is not a justification — main is the baseline for every
  gate; if main moved, the gate verdicts are invalid.
- **Letting governance-file conflicts pile up.** Resolve immediately
  when they appear. Deferred conflict resolution means the operator
  has to reconstruct the conflicting branches' intent from memory.
- **Cross-repo work in a single PR.** Engine repo and marketing repo
  are separate `git remote`s. PRs cannot span repos. Each repo gets
  its own WP, its own PR, its own merge.
- **Open `SPEC:` PR on a WP file + parallel execution session for the
  same WP.** The execution session's authority chain depends on the
  WP body's final form. An unmerged SPEC PR (e.g., a hardening pass
  surfacing audit improvements) signals the WP body isn't final.
  Either merge the SPEC first OR fold the SPEC improvements directly
  into the execution session itself (single commit, no separate SPEC).
  Running both in parallel produces a deadlock surfacing at pre-flight
  ("cited authority not on main") or mid-execution ("WP body changed
  underneath me"). **Empirical precedent (2026-05-11):** WP-007b's
  hardening pass (9 audit improvements) and execution session opened
  in parallel; execution correctly stopped at pre-flight; operator
  abandoned the hardening PR to end the coordination cost — accepted
  the loss of audit refinements in exchange for serial progress.
- **Folding more than ~5 distinct corrections inline into a single
  execution session.** Option D (fold-inline) is the right escape for
  small spec-vs-reality drift, but if execution surfaces 7+ distinct
  corrections to the WP body during one session, the WP body was
  structurally wrong — not lightly buggy. Indicators: corrections
  span multiple WP sections; multiple references to other WPs'
  precedents (WP-144, EC-148, etc.) needed to ground decisions;
  iterative discovery where each amendment uncovers the next gap.
  Past ~5 amendments, STOP and open a fresh draft cycle instead of
  continuing to fold. Hardening passes against WP bodies similarly
  have diminishing returns past ~3 audit findings — beyond that, the
  WP body needs a fresh draft, not iterative hardening.
  **Empirical precedent (2026-05-11):** WP-007b's execution session
  reached 7 inline amendments before locking; the hardening PR that
  preceded it caught 9 issues but missed 7 more. In retrospect,
  opening a fresh draft after the 3rd or 4th amendment would have
  been less painful than continuing to fold inline. Both the
  hardening pass and the execution session's fold-inline budget
  saturated; the WP body's quality, not the workflow, was the
  load-bearing issue.

### When parallel mode is more trouble than it's worth

Sequential mode (one WP at a time) is always safe. Pick sequential
when ANY of:

- The parallel WPs would touch the same files in their `§Files
  Expected to Change` allowlists (not just governance ledgers, but
  actual production files).
- The parallel WPs depend on each other (one's `§Assumes` block
  cites the other).
- You have less than ~30 minutes of focused time and rebase costs
  would dominate productive work.
- A long-running WP (engine refactor, multi-day work) is in flight;
  spinning up a parallel small WP and rebasing the long one against
  it incurs cascading rebase cost on the long one.
- **A SPEC PR is open against the WP file you want to execute.** This
  is the WP-007b case from 2026-05-11: the WP file itself is the
  authority the execution session reads; an unmerged SPEC PR means
  the authority isn't final. Don't parallel — sequence them (merge
  SPEC first) or abandon one side (give up the SPEC, execute against
  current main).

## Common deviations from the happy path

### "This is a small, single-layer, additive change — do I need the full two-session pattern?"

Probably not. If it passes the eligibility gate in `01.0a §Lightweight Lane
(single session)` — single layer, ≤ 4 code/test files (+ ≤ 1 same-layer
runtime-wiring file), no new contract, additive or a mechanical migration,
no determinism/sentinel-hash impact, a narrow validation/UX surface — run it
in the **lightweight lane**: one session, one branch, one PR, inline EC
amendments allowed. The mandatory scaffold run replaces the two-session
split's independence (empirical over procedural), and the lane self-demotes
to the full pattern the moment any eligibility criterion breaks. Default to
the two-session pattern whenever eligibility is ambiguous. **Empirical
precedent: WP-254 (2026-06-16)** ran the heavyweight path — three PRs, a
redraft cycle — for ~2h of code; it was lightweight-eligible the whole time.

### "I want to draft a WP but I'm not sure if it's actually one WP"

Discuss in 01.0a Step 3 until you can lock the framing. If you can't
decide between one-WP and paired-WP after a short conversation: draft
as one, note the split-vs-merge question explicitly in §Scope (Out),
and let pre-flight surface it. Pre-flight is the right gate to catch
scope-too-large issues.

### "Pre-flight says NOT READY because a dep WP is still in flight"

You drafted too early. Park this WP in WORK_INDEX.md as a placeholder
row (`[ ]` with status `BLOCKED on WP-NNN`), close the draft session
without merging anything, and revisit when the dep clears.

### "The work is just a config knob — do I need a WP?"

Read `SAFE-KNOBS.md`. Tier-1 / Tier-2 knob adjustments documented
there don't need a full WP; a small SPEC commit updating the relevant
config + a `DECISIONS.md` D-entry (if the change locks rationale) is
sufficient. Tier-3 (gated) knobs DO need a WP because they touch
engine surfaces.

### "I'm in the middle of execution and realized the EC's allowlist is wrong"

Stop. Either (a) the work is in-scope but the EC missed a file —
amend the EC mid-execution per `01.1 §Mid-execution amendment` rules,
or (b) the work is genuinely larger than the WP — split or re-scope
per `00.1 §Splitting`. **Do not** silently expand scope.

### "The post-mortem triggers fired late and I don't have time"

`01.6` allows authoring the post-mortem in a separate session within
24h of the execution close. Mark the EC's status `Done — post-mortem
pending` in the governance-close commit, then author the post-mortem
in a small follow-up SPEC commit.

### "I forgot to delete the draft branch after Step 7"

Next session's hook will catch it (`01.8`). Run
`node scripts/prune-empty-claude-branch.mjs --execute` from canonical.
The script's safety contract verifies the branch is 0 commits ahead,
no remote, dirt-only worktree before deleting — so a missed cleanup
auto-resolves at the next session boundary with no risk of losing
real work.

### "The EC number I want to reserve is already taken"

Engine-repo ECs with letter suffixes (`EC-NNNA`, `EC-NNNB`) were
pre-allocated for the early WP-005 / WP-006 / WP-007 / WP-008 / WP-009
contract/implementation splits. Many of those slots are still occupied
by their original ECs (e.g., `EC-007B` is `turn-loop`, not available
for cross-repo WP-007b's engine-side execution).

**Rule:** when an EC number would collide with an existing slot,
default to **plain numeric next-free** (e.g., `EC-154` for WP-007b's
engine-side EC, not `EC-007B`). Cross-repo WP ↔ EC linkage is via
the EC body's `§Source` citation, not via matched naming. Don't try
to force `WP-NNN[a|b]` to map to `EC-NNN[A|B]` — the slot may be
occupied. Plain numeric is the engine convention for everything post
`EC-014A` anyway (`EC-150`, `EC-151`, `EC-152`, `EC-153`...).

**Empirical precedent (2026-05-11):** WP-007b's engine-side EC could
have been `EC-007B` to match cross-repo naming, but that slot was
taken (`turn-loop` checklist from the early WP-007 contract/loop
split). Retargeted to `EC-154` mid-execution as a small inline
amendment; cross-repo link preserved via `§Source: WP-007b` in the
EC body. The engine repo's commit-validation CI accepts the plain
numeric form per the `^EC-[0-9]+[A-Z]?:` regex; no validator changes
needed.

### "The WP body has a Failure Condition that's overly strict for the surface I'm touching"

WP authors sometimes write blanket Failure Conditions like
"Lighthouse score below 90 in any of the four categories, regardless
of cause." For surface-level WPs touching apps with pre-existing
characteristics (perf, library deprecations, documentation gaps,
font-size choices), strict "regardless of cause" gates fail on issues
the WP didn't cause and can't reasonably fix in scope.

**Defaults that age better when authoring a WP:**

- **Delta-attributable**: "Lighthouse delta-attributable-to-this-WP
  must not regress any category by ≥10 points." Holds the WP
  accountable for its impact, not for pre-existing scores.
- **Category-scoped**: "Lighthouse SEO + Accessibility ≥ 90"
  (or whichever categories the WP's surface actually moves). Brand
  integration mostly affects A11y + SEO; Performance + Best Practices
  are dominated by data-pipeline + library characteristics that
  surface WPs can't reach.

**If you inherit a WP with an overly strict gate at execution time,**
fold-inline an Amendment that carves out the out-of-scope failures
to a deferred follow-up WP. Don't expand the current WP's scope to
include them — that's how brand-integration WPs become perf-rewrite
WPs.

**Empirical precedent (2026-05-11):** WP-007b §Failure Conditions
required "Lighthouse ≥ 90 in any of the four categories regardless of
cause." Phase 5 verification surfaced Performance 61 + Best Practices
79 — both pre-existing registry-viewer characteristics. Amendment 7
carved them out as deferred follow-up scope; kept WP-007b focused on
its actual deliverable (brand integration, hash parity, mount + class
colors); shipped clean.

## Anti-patterns

- **Skipping pre-flight to save time.** The 5-minute pre-flight catches
  the 5-hour rewrite. Every WP that skipped pre-flight in practice
  produced a mid-execution amendment or a re-scoped WP.
- **Drafting and executing in the same session.** Conflates draft scope
  decisions with implementation decisions; pre-flight's binary verdict
  loses meaning because the draft author and executor are the same
  agent at the same moment. Two-session pattern is load-bearing.
- **Letting the draft branch linger after PR merge.** The cumulative
  effect after N WPs is the 60-branch accumulation that motivated
  `01.8`. Don't normalize the mess.
- **Updating WORK_INDEX.md and EC_INDEX.md in a separate commit from
  the WP execution.** The status flip is part of Definition of Done;
  separating them means the indices and the code drift out of sync
  during PR review. Governance close is one commit.
- **Force-deleting a worktree that has untracked files you haven't
  read.** Per the cleanup precedent in `feedback_squash_merge_audit.md`
  and the empirical pattern from the 2026-05-11 cleanup: always
  inspect uncommitted state before `git worktree remove --force`. The
  hook's safety contract refuses to auto-prune dirty worktrees for
  exactly this reason.
- **Treating the session prompt as source-of-truth.** The session
  prompt at `docs/ai/invocations/session-wpNNN-*.md` is **ephemeral
  scratchpad** by design (gitignored unless an EC normatively cites
  it). The WP file and EC file are the source-of-truth. If the prompt
  and the WP/EC disagree, the WP/EC wins; the prompt should be
  regenerated to match, not the other way around.
- **Resuming a stale worktree without verifying supersession.**
  Drafting sessions get interrupted — a higher-priority WP appears,
  attention shifts, the operator pauses to think. The untracked
  `WP-*.md` / `EC-*.md` files left behind in the worktree are easy
  to pick up later by reopening it, but the worktree's view of "what
  work needs doing" is frozen at the interruption moment. If days
  passed, the work may have shipped via a different commit, the EC
  number may have been reassigned to a different WP, or the proposed
  file shape may no longer match what's expected post-execution.
  Before resuming work in a worktree more than ~24–48 hours old,
  re-run the supersession check from `01.0a §Step 2: Pre-draft
  sanity` against the current `origin/main`:

  - Has the WP slug landed via a different commit?
  - Has the EC number been reassigned to a different WP?
  - Does the proposed file shape still match what's expected
    post-execution (i.e., does the in-progress draft still describe
    the work that needs doing)?

  **Programmatic alternative:** `node scripts/prune-empty-claude-branch.mjs
  --audit --idle-days 2` will surface this worktree if the slug has
  shipped (`stale-untracked-spec`) or the worktree has been idle past
  the 2-day threshold (`worktree-idle`). The `01.0a §Step 2`
  supersession check is the per-slug equivalent; both are valid entry
  points.

  If any check returns a match, treat the worktree as abandoned (per
  `01.0a §Abandoning a draft`, state (a) — pre-commit) rather than
  resuming. Reopening and continuing the draft would produce
  conflicting artifacts at PR-open time and force a mid-flight
  reconciliation that's strictly more expensive than starting fresh.

  **Empirical precedent (2026-05-23):** the cleanup pass that removed
  9 stale `claude/*` worktrees+branches found 4 dirty worktrees with
  untracked `WP-*.md` / `EC-*.md` draft files fitting this pattern —
  `confident-swartz`, `musing-aryabhata`, `blissful-blackburn`, and
  `serene-pike`. In each case the draft's intended WP / EC had
  shipped via different artifacts (e.g., an `EC-158` slot got
  reassigned mid-flight to a different WP; `confident-swartz`'s
  draft `WP-105` body was the pre-execution version that diverged
  from what actually shipped). Reopening any of these worktrees and
  continuing the draft would have produced conflicting artifacts at
  PR-open time. None of these fit the "forgot to delete after PR
  merge" failure mode `scripts/prune-empty-claude-branch.mjs` is
  designed to catch — they all sat below the hook's safety contract
  (dirty worktree = refuse to prune), accumulating silently.

## Authority

and prescribes the order in which they fire. It does not override any
of them. If a cited doc disagrees with a step here, the cited doc
wins.

This document does NOT carry veto authority over WP/EC execution. It
is a navigation aid. The operator alone decides whether to follow each
step; the harness alone decides whether each hook fires.

If the workflow ever fails (a gate is unreachable, a step blocks
progress, a script misbehaves): the operator may bypass the
problematic step provided they document the bypass in the WP's
post-execution notes. Stuck gates are bugs; report them and route
around.
