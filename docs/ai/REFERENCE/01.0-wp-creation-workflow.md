## Purpose

Every other doc in the 01.x family answers a narrow question:

- `01.1` — how do I use an EC during coding?
- `01.3` — what commit messages are legal?
- `01.4` — is this WP ready to execute?
- `01.5` — am I allowed to touch this file outside scope?
- `01.6` — does this work need a post-mortem?
- `01.7` — did a second pair of eyes review the plan?
- `01.8` — what hook is firing and why?

This document answers the upstream question those assume: **how do you get
from a feature idea to a merged commit on `origin/main`, in the right
order, using the right gates, leaving the worktree state clean?**

It is **not** a substitute for any individual doc. It is the index that
tells you which one to read at which moment, and the convention that
chains them together.

---

## Conformance semantics

Unless explicitly marked `OPTIONAL`, every step in this document is
`REQUIRED`. The three tiers:

- **REQUIRED** — must be completed for the WP to be considered valid.
  Skipping a REQUIRED step is a governance violation; the WP cannot
  flip to Done with REQUIRED steps unsatisfied.
- **CONDITIONAL** — required only if the triggering condition applies
  (e.g., `01.6` post-mortem fires when the trigger criteria are met;
  DECISIONS reservation fires when the WP locks design choices).
- **OPTIONAL** — operator judgment. Skipping is not a violation, but
  the operator should be able to explain why the step was unnecessary
  for this particular WP.

Each step's heading carries its tier in parentheses. If absent, assume
`REQUIRED`.

## The two-session pattern

A Work Packet's full lifecycle is **two Claude Code sessions**, each in
its own auto-spawned worktree:

| Session | Produces | Output committed to |
|---|---|---|
| **1. Drafting** | WP file, EC file, index rows, lint gate, pre-flight, copilot check, session-prompt scratchpad | A SPEC PR merged to main |
| **2. Execution** | The actual feature implementation + governance close | An EC: + SPEC: pair of commits merged to main |

Each session's `SessionStart` hook (`01.8`) fires when the session opens.
Both sessions should observe a **silent** hook — if the warning fires, the
operator runs cleanup first (per `01.8 §Manual override`) before
proceeding.

A **third session is sometimes required** for `01.6` post-mortem authoring
when the work surfaces design tension that warrants a written record,
but most post-mortems can be authored inside Session 2.

## Phase 1 — Drafting (Session 1)

### Step 1: Session start

- Open Claude Code from the **CLI or the Windows desktop app**.
- A new `claude/<adjective>-<scientist>-<hash>` worktree+branch
  auto-spawns. There is no off-toggle.
- `SessionStart` hook fires (`01.8`). It runs
  `scripts/prune-empty-claude-branch.mjs --report` against the canonical
  clone. **Expected: silent** (count < 5 prunable branches).
- If the warning appears: stop, switch to canonical clone, run
  `node scripts/prune-empty-claude-branch.mjs --execute`, then return.
  Drafting on top of accumulated detritus inflates the operator's
  cognitive load for no benefit.

### Step 2: Pre-draft sanity (REQUIRED — hard gate)

Done **inside the new draft worktree**. The goal is to ground the draft
in the current state of `main`, not in stale assumptions.

**Pre-condition for Step 3:** canonical clone MUST be clean and
fast-forward synced to `origin/main`. If not: STOP. Do not proceed
with drafting until the working tree is clean and `git pull --ff-only
origin main` exits 0.

**Record the baseline:** capture `git rev-parse origin/main` and cite
it in the WP's `§Context` or `§Assumes` block. This makes the draft's
baseline reproducible by any future operator reading the WP.

- Confirm canonical clone is on `main`, clean, and synced:
  `git status`, `git pull --ff-only origin main`.
- Re-read the authority chain (top to bottom; takes ~2 minutes):
  - `.claude/CLAUDE.md`
  - `docs/ai/ARCHITECTURE.md` (the relevant section for the work's layer)
  - `.claude/rules/work-packets.md`
  - `.claude/rules/<layer>.md` (e.g., `server.md` if engine-server work)
- Identify the next-free numbers:
  - **WP number** — scan `docs/ai/work-packets/WORK_INDEX.md` for the
    highest landed `WP-NNN`; new draft = `WP-(N+1)`. For paired WPs,
    use sequential numbers (`WP-148` + `WP-149`) not suffixed
    (`WP-148a` + `WP-148b`) — that suffix style is reserved for the
    handful of WP-007-era precedents.
  - **EC number** — scan `docs/ai/execution-checklists/EC_INDEX.md`.
    EC numbers are independent of WP numbers (EC-149 may pair with
    WP-146, etc.).
  - **D-NNNNN decision number(s)** — only if the WP locks
    architectural decisions. Scan `docs/ai/DECISIONS.md` for the
    highest landed D-entry; reserved at draft time, **landed at
    execution time** (the entry is added to DECISIONS.md when the WP
    executes, not when it drafts).

**Number reservation ownership:** a reserved number (WP, EC, or
D-NNNNN) is **owned by the draft branch** until the draft PR merges
to `main`. If a draft is abandoned (branch discarded without merging
the WP/EC files), the reserved number returns to the free pool and
MUST NOT be left as a "burnt" gap in the indices. Abandoned drafts
that already pushed an index row to `main` must be retracted with a
follow-up SPEC commit removing the row before the number can be
re-used.

### Step 3: Frame the work

This step is exploratory but must conclude with complete answers to
all five questions below. Discuss with Claude as long as needed; the
output is the framing that makes Step 4 mechanical file-writing rather
than invention.

- **What is the user-visible outcome?** One or two sentences. If the
  outcome is "we adjust a config knob," check `SAFE-KNOBS.md` first —
  some changes are configuration, not WPs. WPs are for code or
  contract changes; tier-1/tier-2 knob adjustments may not need one.
- **Which layer(s) does this cross?** Per the Layer Boundary in
  ARCHITECTURE.md: Registry / Game Engine / Pre-Plan / Server / App /
  Shared Tooling. Layer crossings mean either a single WP that respects
  the boundary, or a split (paired WPs, one per layer).
- **What are the dependencies?** Other WPs that must land first. If a
  hard-dep WP is still draft-state or in-flight, this WP is BLOCKED;
  don't push past that — drop it back into the queue with a clear
  reason.
- **Is this one WP or several?** A useful rule of thumb: if the
  contract surface is bigger than ~10 files OR crosses a layer
  boundary OR locks more than 3 DECISIONS, consider splitting.
  Split = paired WPs sharing a `## Assumes` chain.
- **Does this touch repos other than this one?** The marketing repo
  at `C:\www\legendary-arena-com\` is independent governance (see
  `reference_dual_repo_layout.md` in user memory). Cross-repo work
  needs one WP per repo, typically sequential.

**Step 3 outputs MUST land in the WP file in Step 4:**

- User-visible outcome → `§Goal`
- Layer crossing decision → `§Scope (In) / (Out)`
- Dependencies → `§Assumes`
- Split-vs-single decision → `§Context` (with the rationale, not just
  the verdict) if non-trivial

If you cannot answer all five questions, Step 4 is not yet authorized.
Loop back here.

### Step 4: Create WP + EC

#### WP file

Write to `docs/ai/work-packets/WP-NNN-{kebab-slug}.md` per the template
in `00.1-master-coordination-prompt.md`. Required sections (in this
order):

1. **Goal** — one paragraph, user-visible outcome
2. **Assumes** — every prerequisite WP, every contract this depends on,
   every external state that must be true. Each line cites the source
   that locks it.
3. **Context** — why now, what's the underlying motivation
4. **Scope (In) / (Out)** — closed enumeration of what's changing and
   explicit list of what's NOT changing
5. **Files Expected to Change** — file allowlist (the EC will mirror
   this with `01.5` runtime-wiring exceptions noted)
6. **Contract** — the API/data/UI surface this WP locks
7. **Acceptance Criteria** — testable bullets
8. **Verification Steps** — operator-runnable steps to confirm
9. **Definition of Done** — the binary gates that flip the
   `WORK_INDEX.md` checkbox

#### EC file

Write to `docs/ai/execution-checklists/EC-NNN-{kebab-slug}.checklist.md`
per `EC-TEMPLATE.md`. Hard limit: **60 non-empty content lines**.
Required sections:

- **Before Starting** — dependencies + baseline assertions
- **Locked Values** — verbatim constants the executor must not
  re-derive
- **Guardrails** — 5–8 invariants the executor must not violate
- **Required Comments** — `// why:` locations the executor must
  produce
- **Files to Produce** — the allowlist (must match the WP's section
  5)
- **After Completing** — what to update in indices
- **Common Failure Smells** — optional; failure modes specific to
  this WP's surface

#### Index rows

- `docs/ai/work-packets/WORK_INDEX.md` — add a `[ ]` row in the correct
  phase section. Format per the file's header rules.
- `docs/ai/execution-checklists/EC_INDEX.md` — add a row mapping EC →
  WP, with Status `Pending`.

#### (Optional) DECISIONS reservations

If the WP locks design choices that need durable rationale, draft
`D-NNNNN` entries at the end of `docs/ai/DECISIONS.md`. Mark them
"Drafted YYYY-MM-DD; not yet landed." They flip to "Active
(post-execution)" in Session 2.

### Step 5: Pre-flight + copilot check + lint gate

**Gate execution order is strict and non-negotiable:**

1. Pre-flight (`01.4`) MUST run first and produce `READY TO EXECUTE`
2. Copilot check (`01.7`) MUST run only after pre-flight = READY
3. Lint gate (`00.3`) MUST be satisfied before commit (Step 7)

**Re-run rule:** if any gate fails or any artifact is edited after a
gate passed, that gate AND all downstream gates MUST be re-run. A
stale PASS verdict is not a PASS — pre-flight READY on a WP that was
subsequently edited is no longer READY.

All three gates run **inside Session 1** (the drafting session),
against the just-written WP+EC.

#### Pre-flight (`01.4`)

Read the full procedure in `01.4-pre-flight-invocation.md`. The pre-flight
produces a binary verdict:

- **READY TO EXECUTE** — dependencies verified, scope locked,
  ambiguities resolved
- **NOT READY** — listing of blocking PS-items + clarifying RS-items

If NOT READY: fix the WP/EC in-place, re-run pre-flight. Iterate until
READY.

#### Copilot check (`01.7`)

Read `01.7-copilot-check.md`. Audits the WP + EC + pre-flight report
against 30 failure modes. Verdict:

- **PASS** — proceed to session-prompt authoring
- **RISK** — minor concerns; document and proceed
- **BLOCK** — substantive issue; HOLD (fix in-place) or SUSPEND
  (re-run pre-flight after rewrites)

If RISK or BLOCK: same in-place-fix loop as pre-flight.

#### 00.3 lint gate self-review

Run the **lint gate** (`00.3-prompt-lint-checklist.md`) against the WP.
All 21 sections must PASS or have an explicit N/A justification.
Document the result inline in the WP body under a `## Lint Gate
Self-Review` heading.

#### Hard invariant before Step 6 / Step 7

Pre-condition for both session-prompt authoring (Step 6) and the
drafting commit (Step 7):

- Pre-flight verdict MUST be `READY TO EXECUTE`
- Copilot check verdict MUST be `PASS` or `RISK` (`RISK` requires the
  concern documented inline in the WP)
- Lint gate self-review MUST be complete with all 21 sections
  resolved

If any of these conditions is not met: Step 6 and Step 7 are
**unauthorized**. Loop back to Step 5 and address the failure before
proceeding.

### Step 6: Write the session prompt

Create `docs/ai/invocations/session-wpNNN-{kebab-slug}.md`. This file
is **gitignored** under the patterns we established (`session-*.md`
under `docs/ai/invocations/`), so it lives as a scratchpad. Override
with `git add -f` only if an EC normatively cites it.

Format the prompt per the examples in `docs/ai/invocations/` (e.g.,
`session-wp005a-*.md`, `session-wp010-*.md`):

1. **Header** — WP ID, title, status reference
2. **Invocation intent** — one paragraph
3. **Authority chain (read order)** — CLAUDE.md → ARCHITECTURE.md →
   .claude/rules/ → WP → EC → reference docs → source files
4. **Pre-execution checks** — baseline assertions
5. **Execution rules** — locked values, forbidden patterns,
   `01.5`-runtime-wiring authorization (if applicable; cite
   `01.5-runtime-wiring-allowance.md` and note which files are
   explicitly authorized for wiring outside the allowlist)
6. **SAFE-KNOBS scope** — if the work touches a knob surface, cite
   `SAFE-KNOBS.md` and note which tier the change falls into
7. **Session task** — the work statement

Hard limit: 200 lines unless justified in the WP's `§Context`. The
session prompt is not a substitute for the WP + EC; it's the
operator's brief to the next Claude session.

**Session prompt scope restriction (REQUIRED):** the prompt MUST NOT
introduce any of the following beyond what the WP + EC already
encode:

- New scope (in or out)
- New files in the allowlist
- New contract elements
- New locked values
- New forbidden patterns

The prompt may **restate** or **operationalize** what the WP + EC
encode (e.g., translating "the EC requires X" into "you MUST do X
because the EC requires it"). It may NOT extend or contradict either.
If new scope or contract surface needs adding, it goes back into the
WP + EC, gates re-run from Step 5, then the prompt regenerates.

### Step 7: Commit + close drafting session

Single commit:

```
SPEC: draft WP-NNN + EC-NNN [+ D-NNNNN..D-MMMMM]
```

Commit format per `01.3-commit-hygiene-under-ec-mode.md`. The body
explains scope, lint-gate result, pre-flight verdict, copilot-check
verdict.

Then:

1. Push the draft branch.
2. Open a small PR. Title matches the commit title.
3. Merge to `main` (squash or rebase per repo policy).
4. **Verify the branch is safe to delete** (REQUIRED post-merge gate).
   From inside the draft worktree:

   ```pwsh
   node scripts/prune-empty-claude-branch.mjs --verify-current
   ```

   Expected: `VERIFY PASS: ...`. The check handles both true-merge
   (0 commits ahead) and squash-merge (merged PR exists on origin)
   cases. If it returns `VERIFY FAIL`, **STOP** — the failure message
   explains what's wrong (uncommitted changes, unmerged work, etc.).
   Fix before proceeding.

5. **Delete the draft branch.** This is the load-bearing step that
   keeps the hook quiet next session. Both local and remote:

   ```pwsh
   git branch -D claude/<draft-branch-name>
   git push origin --delete claude/<draft-branch-name>
   ```

6. Return to canonical, run `node scripts/prune-empty-claude-branch.mjs --report`. Expected: silent.

---

### Phase 1 Definition of Done

Phase 1 is complete when ALL of the following are true:

- [ ] WP file + EC file merged to `main` via a SPEC PR
- [ ] WORK_INDEX.md and EC_INDEX.md rows added in the same SPEC commit
- [ ] Pre-flight verdict on record: `READY TO EXECUTE`
- [ ] Copilot check verdict on record: `PASS` or `RISK` (with concern documented)
- [ ] Lint gate self-review section present in the WP body
- [ ] Session prompt written at `docs/ai/invocations/session-wpNNN-*.md`
- [ ] Draft branch deleted (local + remote)
- [ ] `scripts/prune-empty-claude-branch.mjs --report` reports silent from canonical

If any item is unsatisfied, Phase 1 is NOT done. Do not proceed to
Phase 2 with partial Phase 1 completion.

**Blocking drafts:** if pre-flight surfaces a dependency blocker
(`NOT READY` because hard-dep WP is in-flight), do not merge partial
WP/EC artifacts. Choose exactly one:

- **Complete Step 4–5 fully and merge** if you want the WP to exist
  as a `[ ]` placeholder row with `BLOCKED on WP-NNN` status. This
  is appropriate when you want to reserve the WP/EC numbers and
  signal the dependency.
- **Discard the draft branch entirely** if you'd rather re-draft when
  the dep clears. This returns the WP/EC numbers to the free pool.

Half-merged states (WP file on `main` without EC file, or index row
without WP file) are forbidden.

## Phase 2 — Execution (Session 2, new Claude Code session)

### Step 8: Open execution session

- Open a **fresh** Claude Code session. A new worktree+branch auto-spawns.
- `SessionStart` hook fires (`01.8`). Expected: silent (you just cleaned
  up in Step 7).
- **If main has moved since pre-flight passed in Phase 1**, REVALIDATE
  before doing any work. This applies when (a) Phase 1 closed days
  ago and other PRs have merged since, or (b) you're running in
  parallel mode and another WP merged after this WP's pre-flight.
  Procedure:

  ```pwsh
  git fetch origin main
  git rebase origin/main   # or merge if rebase introduces noise
  # Re-run pre-flight per 01.4. If verdict is now NOT READY,
  # address the new blockers before executing.
  ```

  See `## Parallel WP execution` below for the full rebase +
  revalidation discipline when running multiple WPs concurrently.

- Open the session-prompt file written in Step 6
  (`docs/ai/invocations/session-wpNNN-*.md`). Paste its content as the
  first message to Claude.
- Claude reads the authority chain per the prompt: CLAUDE.md →
  ARCHITECTURE.md → .claude/rules/ → WP-NNN → EC-NNN.
- Apply `01.1-how-to-use-ecs-while-coding.md` for in-execution
  discipline (one WP per session, EC clauses drive debugging, etc.).

### Step 9: Implement + post-mortem (if triggered)

Execute the work clause-by-clause per the EC. Specific gates fire as
needed:

- **`01.5` runtime wiring** — if a file outside the allowlist needs
  modification to keep tests green, invoke `01.5` per the authorization
  in the session prompt. Document each wiring change.
- **`01.6` post-mortem** — author IF any trigger fires per
  `01.6-post-mortem-checklist.md` (new contract, new builder, new
  abstraction, new runtime wiring, new code category). Most
  post-mortems can be authored inside Session 2; only split out to a
  dedicated session if the work generated substantial open questions.

### Step 10: Commit + governance close

**Two-commit topology is REQUIRED** (per `01.3 §Two-commit topology`).
The implementation commit (Commit A) and the governance-close commit
(Commit B) MUST be separate commits on the branch. Combining them
into a single commit is forbidden — it conflates code review with
governance review and makes per-commit auditability impossible.

A subsequent **squash merge to `main`** at PR-merge time is fine
(repo policy); the two-commit constraint applies to the branch
state, not to `origin/main`'s history.

#### Commit A — the implementation

```
EC-NNN: <imperative subject under 70 chars>
```

Body: what changed, locked values used, any `01.5` wiring invocations,
any post-mortem invocations.

#### Commit B — governance close

```
SPEC: governance close for WP-NNN / EC-NNN
```

Body lists updates. **Files always updated in Commit B:**

- `docs/ai/work-packets/WORK_INDEX.md` — flip the `[ ]` row to `[x]`,
  add Done date, EC commit reference, baselines pre/post
- `docs/ai/execution-checklists/EC_INDEX.md` — Status → `Done`
- `docs/ai/DECISIONS.md` — land any reserved `D-NNNNN` entries
- `docs/05-ROADMAP-MINDMAP.md` — flip the WP's status icon
  (📝 Drafted → ✅ Done, or 📦 Queued → ✅ Done)
- `docs/STATUS.md` — if it tracks WP-level progress

**Commit prefix conventions** (per `01.3`):

| Prefix | When |
|---|---|
| `EC-NNN:` | Implementation commits inside an EC's scope |
| `SPEC:` | Governance-only commits (draft, close, ledger updates) |
| `INFRA:` | Infrastructure / hygiene changes outside any WP/EC scope |

### Step 11: Push + cleanup

1. Push the execution branch.
2. Open the PR. Title: `WP-NNN: <title>`. Description references the WP
   file and the EC file.
3. Merge to `main` (squash typical).
4. **Verify the branch is safe to delete** (REQUIRED post-merge gate).
   From inside the execution worktree:

   ```pwsh
   node scripts/prune-empty-claude-branch.mjs --verify-current
   ```

   Expected: `VERIFY PASS: ...`. The check accepts both
   merge-commit and squash-merge patterns; failures (uncommitted
   changes, unmerged work, missing PR) STOP cleanup until resolved.

5. **Delete the execution branch** (local + remote), same as Step 7.
6. Run `node scripts/prune-empty-claude-branch.mjs --report` from
   canonical. Expected: silent.
7. (Optional) If the hook reports any accumulation from earlier
   sessions, run `--execute`.

Phase 2 complete. The implementation is on `origin/main`; both draft
and execution branches are deleted; the worktree state is clean for
the next WP.

## Parallel WP execution

The two-session pattern assumes serial execution of a single WP. In
practice, you may want multiple WPs in flight at the same time — e.g.,
one on the registry viewer, one on the game engine, one on the
marketing repo. This is **permitted** by the `work-packets.md §One
Packet per Session` rule (the rule says one WP per *session*, not one
WP per day or per operator).

The cost is **governance-ledger conflicts**. Every WP touches the same
set of small governance files; those files WILL conflict at merge time
when multiple WPs diverged from the same `origin/main` tip.

### What's structurally parallel-safe

- **Cross-repo work** (engine repo + marketing repo) is fully
  parallel-safe. The repos have independent governance ledgers; no
  shared files. Order of cross-repo merges doesn't matter.
- **Cross-layer work in the same repo** (e.g., `registry-viewer` +
  `game-engine`) is parallel-safe at the *code* level — different
  package trees, different test surfaces, no import cycles by Layer
  Boundary rule. The conflict surface reduces to governance files
  only.

### What WILL conflict in same-repo parallel mode

| File | Why it conflicts |
|---|---|
| `docs/ai/work-packets/WORK_INDEX.md` | Both WPs add rows in the same phase section, often on adjacent lines |
| `docs/ai/execution-checklists/EC_INDEX.md` | Both EC rows land in the same index |
| `docs/ai/DECISIONS.md` | Both reserved `D-NNNNN` from the same `origin/main` tip — collision is hard (same number, different meanings) |
| `docs/STATUS.md` | Running narrative; every WP touches it |
| `docs/05-ROADMAP-MINDMAP.md` | Both flip status icons; conflicts on adjacent lines |

### Rules for parallel mode

1. **First-to-READY merges first.** When multiple WPs reach READY
   simultaneously, the first one through pre-flight + copilot + lint
   gates merges first. Don't "save the easy one for last" — that
   creates an incentive to rush gates on harder WPs.

2. **All other in-flight WPs MUST rebase + revalidate after every
   merge to main.** Procedure, run in each non-merged worktree:

   ```pwsh
   git fetch origin main
   git rebase origin/main
   # Resolve governance-file conflicts mechanically (keep both rows;
   # preserve phase ordering).
   # Re-run pre-flight per 01.4. The previous READY verdict is now
   # stale; the new main may have shifted dependencies or baselines.
   ```

   Pre-flight verdicts go stale when main moves. A READY from before
   another WP merged is no longer valid for execution OR for merging
   the WP itself.

3. **Numbering reservation honors merge order, not draft order.** If
   WP-A and WP-B both reserved `D-13901` from the same starting
   commit, and WP-A merges first, WP-B's `D-13901` is now invalid.
   WP-B MUST renumber to next-free (e.g., `D-13902`) during rebase —
   update every reference in WP-B's WP file, EC file, code comments,
   and `DECISIONS.md` draft entry. This is the worst kind of
   conflict to resolve; minimize by checking origin's draft branches
   for in-flight reservations BEFORE drafting (see "Numbering
   reservation ownership" in Step 2).

4. **Cross-repo parallel follows the same rules within each repo
   independently.** Engine repo and marketing repo (`C:\www\legendary-arena-com\`)
   have separate governance ledgers; their `WORK_INDEX`-equivalent
   files don't share state. Each side runs its own
   first-to-READY-merges-first discipline.

5. **No hard concurrency cap.** The limit is operator cognitive load.
   Practically: 2–3 WPs in flight is manageable; beyond that,
   mid-execution scope tends to drift and rebase cost compounds.

### Anti-patterns (parallel-mode specific)

- **Drafting two WPs simultaneously without checking origin for
  in-flight reservations.** Open two draft sessions, both reserve
  `D-13901`, both reach READY, both try to merge. The second-to-merge
  has a mass-renumber problem touching every WP/EC/code/DECISIONS
  reference.
- **Skipping the rebase + revalidate step between merges.** "It's
  just docs" is not a justification — main is the baseline for every
  gate; if main moved, the gate verdicts are invalid.
- **Letting governance-file conflicts pile up.** Resolve immediately
  when they appear. Deferred conflict resolution means the operator
  has to reconstruct the conflicting branches' intent from memory.
- **Cross-repo work in a single PR.** Engine repo and marketing repo
  are separate `git remote`s. PRs cannot span repos. Each repo gets
  its own WP, its own PR, its own merge.

### When parallel mode is more trouble than it's worth

Sequential mode (one WP at a time) is always safe. Pick sequential
when ANY of:

- The parallel WPs would touch the same files in their `§Files
  Expected to Change` allowlists (not just governance ledgers, but
  actual production files).
- The parallel WPs depend on each other (one's `§Assumes` block
  cites the other).
- You have less than ~30 minutes of focused time and rebase costs
  would dominate productive work.
- A long-running WP (engine refactor, multi-day work) is in flight;
  spinning up a parallel small WP and rebasing the long one against
  it incurs cascading rebase cost on the long one.

## Common deviations from the happy path

### "I want to draft a WP but I'm not sure if it's actually one WP"

Discuss in Step 3 until you can lock the framing. If you can't decide
between one-WP and paired-WP after a short conversation: draft as one,
note the split-vs-merge question explicitly in §Scope (Out), and let
pre-flight surface it. Pre-flight is the right gate to catch
scope-too-large issues.

### "Pre-flight says NOT READY because a dep WP is still in flight"

You drafted too early. Park this WP in WORK_INDEX.md as a placeholder
row (`[ ]` with status `BLOCKED on WP-NNN`), close the draft session
without merging anything, and revisit when the dep clears.

### "The work is just a config knob — do I need a WP?"

Read `SAFE-KNOBS.md`. Tier-1 / Tier-2 knob adjustments documented
there don't need a full WP; a small SPEC commit updating the relevant
config + a `DECISIONS.md` D-entry (if the change locks rationale) is
sufficient. Tier-3 (gated) knobs DO need a WP because they touch
engine surfaces.

### "I'm in the middle of execution and realized the EC's allowlist is wrong"

Stop. Either (a) the work is in-scope but the EC missed a file —
amend the EC mid-execution per `01.1 §Mid-execution amendment` rules,
or (b) the work is genuinely larger than the WP — split or re-scope
per `00.1 §Splitting`. **Do not** silently expand scope.

### "The post-mortem triggers fired late and I don't have time"

`01.6` allows authoring the post-mortem in a separate session within
24h of the execution close. Mark the EC's status `Done — post-mortem
pending` in the governance-close commit, then author the post-mortem
in a small follow-up SPEC commit.

### "I forgot to delete the draft branch after Step 7"

Next session's hook will catch it (`01.8`). Run
`node scripts/prune-empty-claude-branch.mjs --execute` from canonical.
The script's safety contract verifies the branch is 0 commits ahead,
no remote, dirt-only worktree before deleting — so a missed cleanup
auto-resolves at the next session boundary with no risk of losing
real work.

## Anti-patterns

- **Skipping pre-flight to save time.** The 5-minute pre-flight catches
  the 5-hour rewrite. Every WP that skipped pre-flight in practice
  produced a mid-execution amendment or a re-scoped WP.
- **Drafting and executing in the same session.** Conflates draft scope
  decisions with implementation decisions; pre-flight's binary verdict
  loses meaning because the draft author and executor are the same
  agent at the same moment. Two-session pattern is load-bearing.
- **Letting the draft branch linger after PR merge.** The cumulative
  effect after N WPs is the 60-branch accumulation that motivated
  `01.8`. Don't normalize the mess.
- **Updating WORK_INDEX.md and EC_INDEX.md in a separate commit from
  the WP execution.** The status flip is part of Definition of Done;
  separating them means the indices and the code drift out of sync
  during PR review. Governance close is one commit.
- **Force-deleting a worktree that has untracked files you haven't
  read.** Per the cleanup precedent in `feedback_squash_merge_audit.md`
  and the empirical pattern from the 2026-05-11 cleanup: always
  inspect uncommitted state before `git worktree remove --force`. The
  hook's safety contract refuses to auto-prune dirty worktrees for
  exactly this reason.
- **Treating the session prompt as source-of-truth.** The session
  prompt at `docs/ai/invocations/session-wpNNN-*.md` is **ephemeral
  scratchpad** by design (gitignored unless an EC normatively cites
  it). The WP file and EC file are the source-of-truth. If the prompt
  and the WP/EC disagree, the WP/EC wins; the prompt should be
  regenerated to match, not the other way around.

## Authority

This document INDEXES other governance docs and prescribes the order
in which they fire. It does not override any of them. If a cited doc
disagrees with a step here, the cited doc wins.

This document does NOT carry veto authority over WP/EC execution. It
is a navigation aid. The operator alone decides whether to follow each
step; the harness alone decides whether each hook fires.

If the workflow ever fails (a gate is unreachable, a step blocks
progress, a script misbehaves): the operator may bypass the
problematic step provided they document the bypass in the WP's
post-execution notes. Stuck gates are bugs; report them and route
around.
