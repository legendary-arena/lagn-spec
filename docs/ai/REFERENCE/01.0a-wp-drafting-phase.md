## Purpose

Every other doc in the 01.x family answers a narrow question:

- `01.1` — how do I use an EC during coding?
- `01.3` — what commit messages are legal?
- `01.4` — is this WP ready to execute?
- `01.5` — am I allowed to touch this file outside scope?
- `01.6` — does this work need a post-mortem?
- `01.7` — did a second pair of eyes review the plan?
- `01.8` — what hook is firing and why?

Together, **01.0a + 01.0b** answer the upstream question those assume:
**how do you get from a feature idea to a merged commit on `origin/main`,
in the right order, using the right gates, leaving the worktree state
clean?**

01.0a (this file) covers **drafting** — producing the WP file, EC file,
index rows, gate verdicts, and session-prompt scratchpad, then merging
that bundle to main as a SPEC PR.

01.0b covers **execution + cleanup + parallel-mode + reference**.

Why the split: the drafting session is a natural pause-point. The
operator typically reviews the drafted WP/EC, considers edits, and may
let days pass before opening the execution session. Splitting the doc
at this boundary means each phase's reference is self-contained and
small enough to consult mid-session without overwhelming context.

## Conformance semantics

Unless explicitly marked `OPTIONAL`, every step in this document is
`REQUIRED`. The three tiers:

- **REQUIRED** — must be completed for the WP to be considered valid.
  Skipping a REQUIRED step is a governance violation; the WP cannot
  flip to Done with REQUIRED steps unsatisfied.
- **CONDITIONAL** — required only if the triggering condition applies
  (e.g., `01.6` post-mortem fires when the trigger criteria are met;
  DECISIONS reservation fires when the WP locks design choices).
- **OPTIONAL** — operator judgment. Skipping is not a violation, but
  the operator should be able to explain why the step was unnecessary
  for this particular WP.

Each step's heading carries its tier in parentheses. If absent, assume
`REQUIRED`.

## Harness modes (IDE/desktop vs CLI)

This workflow has two first-class invocation patterns. Both are valid;
their differences matter only at Phase 1 Step 1 (session start).

| Mode | What it is | Worktree behavior | SessionStart hook |
|---|---|---|---|
| **Claude Code IDE / desktop app** | Session opened via VS Code extension or Windows desktop app | Auto-spawns `claude/<adj>-<sci>` worktree per session at `C:\claude-worktrees\` | Fires per `01.8` |
| **CLI** | `claude` invoked from any shell in canonical or in a worktree of operator's choice | No auto-spawn; operator manages branches/worktrees explicitly | Does NOT fire |

### When to use which

- **IDE / desktop mode** is structurally better for **parallel WP
  execution** (multiple sessions = multiple auto-spawned worktrees,
  naturally isolated). The accumulation problem `01.8` addresses is
  specific to IDE mode.
- **CLI mode** is better for: single-WP focus sessions, branch
  cleanup work, direct-to-main `INFRA` edits, anything where you
  want the canonical working tree as your edit target rather than
  a fresh worktree.

### Workflow translation for CLI mode

CLI sessions don't auto-spawn worktrees, so Phase 1 Step 1 adapts:

- Operator picks between two sub-modes:
  - **(a) Branch-in-canonical:** `git checkout -b claude/<draft-name>`
    from `main`, in the canonical clone. Simpler, lower setup cost.
    Appropriate when draft work is bounded (markdown-only,
    governance edits, short sessions). pCloud sync conflicts CAN
    occur on canonical; abandon in-place and switch to (b) if
    `[conflicted N]` files appear.
  - **(b) Explicit worktree:** `git worktree add C:\claude-worktrees\<name>
    -b claude/<draft-name>`. Off-pCloud (no sync conflicts), but
    adds worktree-path-discipline overhead per
    `feedback_worktree_path_discipline.md` in user memory.
    Appropriate for parallel WPs or for work expecting >~10 files
    of churn.
- SessionStart hook does NOT fire. To audit accumulation manually,
  run from canonical: `node scripts/prune-empty-claude-branch.mjs --report`.
- All other phase steps (Phase 1 Steps 2–7, Phase 2 Steps 8–11, gates,
  cleanup, parallel-execution rules) apply identically.

The choice between branch-in-canonical and explicit worktree is
operational taste, not correctness. Both reach the same governance
artifacts and pass the same gates.

## The two-session pattern

A Work Packet's full lifecycle is **two Claude Code sessions**, each in
its own auto-spawned worktree:

| Session | Produces | Output committed to | Covered in |
|---|---|---|---|
| **1. Drafting** | WP file, EC file, index rows, lint gate, pre-flight, copilot check, session-prompt scratchpad | A SPEC PR merged to main | **01.0a** (this file) |
| **2. Execution** | The actual feature implementation + governance close | An EC: + SPEC: pair of commits merged to main | **01.0b** |

Each session's `SessionStart` hook (`01.8`) fires when the session opens.
Both sessions should observe a **silent** hook — if the warning fires, the
operator runs cleanup first (per `01.8 §Manual override`) before
proceeding.

A **third session is sometimes required** for `01.6` post-mortem authoring
when the work surfaces design tension that warrants a written record,
but most post-mortems can be authored inside Session 2.

## Lightweight Lane (single session)

> **Opt-in exception to the two-session pattern above. Never the default.**
> The lane is **strictly narrower** than the standard pattern, and **any
> ambiguity resolves against eligibility** — when in doubt, use the
> two-session lane. Locked by `DECISIONS.md` D-24028.

The two-session pattern optimizes for cross-layer coordination, contract
safety, and audit separation. For small, single-scope, additive work it
inverts the cost: ~1–2h of code wrapped in three PRs, avoidable redraft
cycles, and branch/deploy churn (the WP-254 precedent below). The
lightweight lane collapses such work into **one session, one branch, one
PR** — without dropping any load-bearing gate.

### Eligibility gate (ALL must hold — any miss → two-session lane)

Eligibility is **provisional at draft** (the structural criteria, judged
from the proposed scope) and **confirmed at govern-close** (the empirical
criteria, knowable only after implementation). A WP enters the lane on the
provisional check and stays only if the confirmed check passes; any
confirmed-criterion failure triggers self-demotion (below).

**Structural (provisional, at draft):**

1. Single layer, single package/app — no layer-boundary crossing.
2. ≤ 4 code/test files in the allowlist (governance ledgers excluded),
   **plus at most one same-layer runtime-wiring file** (`01.5`). A second
   wiring file, or any cross-layer wiring, demotes.
3. No `01.6` trigger — no new contract, abstraction, builder,
   runtime-wiring category, or code category.
4. No new contract file (`.types.ts` / `.validate.ts` / `.gating.ts`).
5. At most one **scoped** `D-entry` — rationale-only; locks no
   cross-cutting or future-facing invariant.
6. Surface limited to narrow input-validation or UX copy. **Not allowed:**
   scoring / PAR / leaderboards, identity, multiplayer sync, monetization,
   RNG, determinism/persistence.

**Empirical (confirmed, at govern-close):**

7. Strictly additive **or** a mechanical migration — no rewrite of existing
   logic (a refactor disguised as additive is a FAIL).
8. Zero determinism impact: persistence boundary, replay behavior, and
   sentinel/hash surfaces unchanged; `finalStateHash` unchanged or N/A.
9. The file/wiring budget (#2) still holds at the final `git diff --name-only`.

### Collapsed flow — one session, one branch, one PR

1. Draft WP + EC (shorter than standard; the EC still carries the scope
   allowlist + locked values).
2. **Scaffold** the change — prototype it and run the affected package's
   suite — and record the observed result (REQUIRED for any
   validation-tightening change; see `01.4 §Empirical Scaffold`).
3. Implement.
4. Govern-close (two-commit topology, below).
5. Open one PR → merge.

No session boundary exists. Inline EC amendments are expected, not
exceptional — but only within category (see below).

### Kept — non-negotiable, identical to the standard lane

- Tests + typecheck green. Never skipped.
- Scaffold-first for validation-tightening changes.
- Layer-boundary discipline; EC allowlist + locked values.
- `00.3` lint gate (the WP/EC may be shorter, but every section still
  resolves).
- **Two-commit topology on the branch** — `EC-NNN:` implementation +
  `SPEC:` governance close. The two-session *split* collapses; the
  two-commit *topology* does not.
- A **condensed pre-flight**: the scaffold subsumes only the
  *behavior-identity* check (the "it's additive" claim); the lane still
  explicitly confirms **dependencies complete, cited authority/contracts
  on `main`, and scope locked** — with evidence, not "implicit."
- **D-24026 live-verification** for any user-visible surface. It is
  inherently post-merge (you cannot verify the live bundle before it
  deploys), so the lane collapses it from a full PR to a **single
  STATUS-flip commit**: direct-to-`main` where permitted, else batched into
  the next WP's governance close, or a deliberately minimal PR — never its
  own draft+exec ceremony.

### Collapsed / removed

- Two-session split → one session.
- Separate `SPEC:` draft PR → folded into the single PR.
- Separate live-verification PR → the post-merge STATUS-flip commit above.
- Copilot (`01.7`) full 30-mode audit → a **targeted self-review**:
  eligibility confirmation + scaffold-result confirmation.

### Inline amendment vs self-demotion

**Inline amendment is allowed ONLY for:** a file-allowlist omission, a test
fixture correction, or a deterministic migration expansion.

**Self-demotion is MANDATORY** — STOP the lane and convert to the
two-session pattern (or split the WP) — the moment any of these occurs:

- a new contract is required,
- a layer boundary is crossed,
- a determinism / persistence / hash surface is touched,
- the file/wiring budget is exceeded,
- a `01.6` trigger fires,
- **any ambiguity arises about scope classification.**

The lane MUST NOT absorb out-of-scope work. A fixable in-scope gap is
amended inline; a scope-*category* change demotes.

### The safeguard that replaces lost independence

The two-session split exists so the draft author and the executor are not
the same agent at the same moment — pre-flight's binary verdict carries
*procedural* independence. The lane loses that. Its replacement is the
mandatory scaffold run: a lightweight WP cannot pass draft unless the
change has been **prototyped and the affected suite run, with observed
output recorded.** No reasoning or simulation substitute. This trades
procedural independence for **empirical** independence — the stronger of
the two, because an observed test result does not care who holds the pen.
(The original WP-254 pre-flight failed precisely because it was *reasoned*;
the fix is an *observed* one — see `01.4 §Empirical Scaffold` and Step 3.)

### Anti-gaming (hard disqualifiers → two-session lane)

A WP is disqualified from the lane if any of these holds:

- eligibility is **argued in prose** instead of demonstrated with evidence
  — a file count from `git diff --name-only`, an unchanged `finalStateHash`,
  the absence of a contract file, the scaffold output; assertion without the
  artifact is a FAIL;
- the scaffold run is skipped, or **not run before eligibility is
  confirmed**;
- inline EC amendments exceed **3 distinct corrections** — deliberately
  tighter than `01.0b`'s ~5-amendment heavyweight threshold (a small
  additive WP that needs 4+ corrections was never lightweight); aligns with
  the WP-007b multi-amendment-drift precedent in `01.0b`.

### Empirical precedent (2026-06-16): WP-254

WP-254 (a single-app lobby `ext_id` guard: parser + two test files + one
doc) ran the full heavyweight path — a redraft PR, a separate execution PR,
and a separate live-verification PR — for ~2h of code. Under this lane it
would have been one session: draft → scaffold (which surfaces the 16
bare-slug fixture breaks) → fold the fixture migration + the second file in
as an **inline amendment** → implement → 576/0 → govern-close → one PR →
merge → STATUS-flip after the deploy-confirmed-SHA check. The scaffold-first
gate still catches the defect — at draft time, as intended — at roughly half
the wall-clock and with none of the permission round-trips. The
ineligibility checks would NOT have tripped: single layer, four files,
additive + mechanical migration, no contract file, `finalStateHash` N/A.

## Phase 1 — Drafting (Session 1)

### Step 1: Session start

Behavior depends on harness mode (see `## Harness modes` above):

**IDE / desktop app mode:**

- A new `claude/<adjective>-<scientist>-<hash>` worktree+branch
  auto-spawns at `C:\claude-worktrees\`. There is no off-toggle.
- `SessionStart` hook fires (`01.8`). It runs
  `scripts/prune-empty-claude-branch.mjs --report` against the canonical
  clone. **Expected: silent** (count < 5 prunable branches).
- If the warning appears: stop, switch to canonical clone, run
  `node scripts/prune-empty-claude-branch.mjs --execute`, then return.
  Drafting on top of accumulated detritus inflates cognitive load for
  no benefit.

**CLI mode:**

- No auto-spawn happens. Operator picks the sub-mode:
  - **(a) Branch-in-canonical:** `git checkout -b claude/<draft-name>` from main
  - **(b) Explicit worktree:** `git worktree add C:\claude-worktrees\<name> -b claude/<draft-name>`
- SessionStart hook does NOT fire. For a manual audit:
  `node scripts/prune-empty-claude-branch.mjs --report` from canonical.
- Same accumulation discipline applies — if you'd already have
  accumulated branches in IDE mode, you still want a clean slate
  before starting Phase 1 work.

### Step 2: Pre-draft sanity (REQUIRED — hard gate)

Done **inside the new draft worktree**. The goal is to ground the draft
in the current state of `main`, not in stale assumptions.

**Pre-condition for Step 3:** canonical clone MUST be clean and
fast-forward synced to `origin/main`. If not: STOP. Do not proceed
with drafting until the working tree is clean and `git pull --ff-only
origin main` exits 0.

**Record the baseline:** capture `git rev-parse origin/main` and cite
it in the WP's `§Context` or `§Assumes` block. This makes the draft's
baseline reproducible by any future operator reading the WP.

- Confirm canonical clone is on `main`, clean, and synced:
  `git status`, `git pull --ff-only origin main`.
- Re-read the authority chain (top to bottom; takes ~2 minutes):
  - `.claude/CLAUDE.md`
  - `docs/ai/ARCHITECTURE.md` (the relevant section for the work's layer)
  - `.claude/rules/work-packets.md`
  - `.claude/rules/<layer>.md` (e.g., `server.md` if engine-server work)
- **Verify the work isn't already in flight or shipped** (supersession
  check). Before reserving WP/EC numbers, run three small checks
  against the current state of the repo — each is ~15 seconds and the
  set is small enough to run inline:

  1. `git log --oneline --all --grep="<proposed-slug>"` — has anyone
     committed under this slug or a near-match? `--all` covers branch
     tips not yet merged, so an in-flight draft on another branch
     surfaces here too.
  2. `ls docs/ai/work-packets/*<slug>*.md docs/ai/execution-checklists/*<slug>*.md`
     — does a file with similar naming already exist? Catches the
     case where a prior session drafted (or partly drafted) the same
     work and the file is still on disk.
  3. `gh pr list --state all --search "<slug>"` — is there an
     in-flight or merged PR matching this slug? Catches the case
     where the work shipped via a PR whose title diverges slightly
     from the proposed slug.

  **Programmatic alternative:** the three checks above can also be
  run from canonical as a single command:
  `node scripts/prune-empty-claude-branch.mjs --audit` (added in
  PR #131, 2026-05-23). The `stale-untracked-spec` heuristic covers
  the file-naming check (#2), and the `abandoned-no-PR` heuristic
  covers the PR-existence check (#3). The slug-grep check (#1) remains
  manual — `--audit` runs per-branch, not per-slug. Manual + tool
  together are belt-and-braces; running both costs ~30 seconds and
  catches near-collisions either misses alone.

  If any check returns a match: **STOP**. Reconcile before drafting.
  The decision is one of: (1) the work is already done — abandon the
  proposed draft and reclaim the operator's attention; (2) the work
  is in flight elsewhere — retarget (extend the existing WP, or
  coordinate with the other draft); (3) the match is a false-positive
  near-collision on slug words — note it in the WP's `§Context` and
  proceed.

  **Empirical precedent (2026-05-23):** the cleanup pass that removed
  9 stale `claude/*` branches included `romantic-mayer`, whose draft
  was a slim-`WORK_INDEX.md` refactor. The same refactor had already
  shipped directly to `main` as commit `768d42f` (`INFRA: slim
  WORK_INDEX — collapse completed WPs to one-liners`). Two drafting
  paths converged on the same work; the supersession check above
  (running `git log --grep="slim"` or `gh pr list --search
  "WORK_INDEX"` before reserving numbers) would have caught the
  collision at the start of `romantic-mayer`'s session, before any
  draft effort was spent.

- Identify the next-free numbers:
  - **WP number** — scan `docs/ai/work-packets/WORK_INDEX.md` for the
    highest landed `WP-NNN`; new draft = `WP-(N+1)`. For paired WPs,
    use sequential numbers (`WP-148` + `WP-149`) not suffixed
    (`WP-148a` + `WP-148b`) — that suffix style is reserved for the
    handful of WP-007-era precedents.
  - **EC number** — scan `docs/ai/execution-checklists/EC_INDEX.md`.
    EC numbers are independent of WP numbers (EC-149 may pair with
    WP-146, etc.).
  - **D-NNNNN decision number(s)** — only if the WP locks
    architectural decisions. Scan `docs/ai/DECISIONS.md` for the
    highest landed D-entry; reserved at draft time, **landed at
    execution time** (the entry is added to DECISIONS.md when the WP
    executes, not when it drafts).

**Number reservation ownership:** a reserved number (WP, EC, or
D-NNNNN) is **owned by the draft branch** until the draft PR merges
to `main`. If a draft is abandoned (branch discarded without merging
the WP/EC files), the reserved number returns to the free pool and
MUST NOT be left as a "burnt" gap in the indices. Abandoned drafts
that already pushed an index row to `main` must be retracted with a
follow-up SPEC commit removing the row before the number can be
re-used.

### Step 3: Frame the work

This step is exploratory but must conclude with complete answers to
all of the questions below. Discuss with Claude as long as needed; the
output is the framing that makes Step 4 mechanical file-writing rather
than invention.

- **What is the user-visible outcome?** One or two sentences. If the
  outcome is "we adjust a config knob," check `SAFE-KNOBS.md` first —
  some changes are configuration, not WPs. WPs are for code or
  contract changes; tier-1/tier-2 knob adjustments may not need one.
- **Which layer(s) does this cross?** Per the Layer Boundary in
  ARCHITECTURE.md: Registry / Game Engine / Pre-Plan / Server / App /
  Shared Tooling. Layer crossings mean either a single WP that respects
  the boundary, or a split (paired WPs, one per layer).
- **What are the dependencies?** Other WPs that must land first. If a
  hard-dep WP is still draft-state or in-flight, this WP is BLOCKED;
  don't push past that — drop it back into the queue with a clear
  reason.
- **Is this one WP or several?** A useful rule of thumb: if the
  contract surface is bigger than ~10 files OR crosses a layer
  boundary OR locks more than 3 DECISIONS, consider splitting.
  Split = paired WPs sharing a `## Assumes` chain.
- **Does this WP add or tighten validation on an existing input
  path?** A change that makes previously-accepted input newly-rejected
  — a stricter parser, a new guard, a narrowed schema, a tightened type
  check — is **scaffold-first, not contract-first**. Before it can
  reach `READY`, prototype the change on a throwaway branch and run the
  affected package's suite (`pnpm --filter <pkg> test`). This WP class
  fails one specific way: data that used to be valid is now rejected,
  and pre-existing valid-path test fixtures — often across more than
  one file, including files in other packages — carry the now-invalid
  form. That breakage is visible only by *running* the change, never by
  reasoning that it is "additive." Fold every observed failure into
  `§Scope (In)` and the `§Files Expected to Change` allowlist. A
  pre-flight `READY` reached by argument rather than an observed run is
  invalid for this class — see `01.4 §Empirical Scaffold
  (Validation-Tightening WPs)`.

  **Empirical precedent (2026-06-15):** WP-254 added a lobby
  qualified-`ext_id` guard to `parseLoadoutJson`. Its draft and
  hardening passes both reasoned "strictly additive — no existing test
  changes" and reached `READY`, but the WP-092 valid-loadout fixture
  (`VALID_COMPOSITION`) and a second file's fixture
  (`lobbyApi.test.ts SAMPLE_CONFIG`) use bare-slug ids the new guard
  rejects. Execution hit 16 broken tests across two files — one outside
  the EC allowlist — on the first `pnpm test`. A 30-second scaffold run
  at draft time would have caught all of it and folded the fixture
  migration into scope before execution opened.
- **Does this touch repos other than this one?** The marketing repo
  at `C:\www\legendary-arena-com\` is independent governance (see
  `reference_dual_repo_layout.md` in user memory). Cross-repo work
  needs one WP per repo, typically sequential.
- **Is this actually a new WP, or an audit/hardening edit to an
  existing WP?** If the work is refining
  `docs/ai/work-packets/WP-NNN-*.md` for an already-drafted WP-NNN
  (e.g., applying 9 audit-review improvements to a WP body), you're
  in `SPEC:` surgical-correction territory per `01.3-commit-hygiene
  -under-ec-mode.md`, **not** new-WP drafting per this doc. Skip the
  rest of 01.0a; treat it as a small SPEC commit. **Critical
  sequencing constraint:** if WP-NNN's execution session is open or
  imminent, the SPEC PR MUST merge BEFORE execution opens — OR the
  SPEC PR must be abandoned. Parallel edits to the same WP file
  deadlock at pre-flight; see `01.0b §Parallel WP execution` for the
  empirical precedent (WP-007b, 2026-05-11) and the resolution
  pattern.

**Step 3 outputs MUST land in the WP file in Step 4:**

- User-visible outcome → `§Goal`
- Layer crossing decision → `§Scope (In) / (Out)`
- Dependencies → `§Assumes`
- Split-vs-single decision → `§Context` (with the rationale, not just
  the verdict) if non-trivial

If you cannot answer all of the questions above, Step 4 is not yet
authorized. Loop back here.

### Step 4: Create WP + EC

#### WP file

Write to `docs/ai/work-packets/WP-NNN-{kebab-slug}.md` per the template
in `00.1-master-coordination-prompt.md`. Required sections (in this
order):

1. **Goal** — one paragraph, user-visible outcome
2. **Assumes** — every prerequisite WP, every contract this depends on,
   every external state that must be true. Each line cites the source
   that locks it.
3. **Context** — why now, what's the underlying motivation
4. **Scope (In) / (Out)** — closed enumeration of what's changing and
   explicit list of what's NOT changing
5. **Files Expected to Change** — file allowlist (the EC will mirror
   this with `01.5` runtime-wiring exceptions noted)
6. **Contract** — the API/data/UI surface this WP locks
7. **Acceptance Criteria** — testable bullets
8. **Verification Steps** — operator-runnable steps to confirm
9. **Definition of Done** — the binary gates that flip the
   `WORK_INDEX.md` checkbox

#### EC file

Write to `docs/ai/execution-checklists/EC-NNN-{kebab-slug}.checklist.md`
per `EC-TEMPLATE.md`. Target **~60** non-empty content lines; hard ceiling
**100** (see `EC-TEMPLATE.md §Rules` for the rationale and the
excluded-lines definition — header block and section titles do not count).
Required sections:

- **Before Starting** — dependencies + baseline assertions
- **Locked Values** — verbatim constants the executor must not
  re-derive
- **Guardrails** — 5–8 invariants the executor must not violate
- **Required Comments** — `// why:` locations the executor must
  produce
- **Files to Produce** — the allowlist (must match the WP's section
  5)
- **After Completing** — what to update in indices
- **Common Failure Smells** — optional; failure modes specific to
  this WP's surface

#### Index rows

- `docs/ai/work-packets/WORK_INDEX.md` — add a `[ ]` row in the correct
  phase section. Format per the file's header rules.
- `docs/ai/execution-checklists/EC_INDEX.md` — add a row mapping EC →
  WP, with Status `Pending`.

#### (Optional) DECISIONS reservations

If the WP locks design choices that need durable rationale, draft
`D-NNNNN` entries at the end of `docs/ai/DECISIONS.md`. Mark them
"Drafted YYYY-MM-DD; not yet landed." They flip to "Active
(post-execution)" in Session 2.

### Step 5: Pre-flight + copilot check + lint gate

**Gate execution order is strict and non-negotiable:**

1. Pre-flight (`01.4`) MUST run first and produce `READY TO EXECUTE`
2. Copilot check (`01.7`) MUST run only after pre-flight = READY
3. Lint gate (`00.3`) MUST be satisfied before commit (Step 7)

**Re-run rule:** if any gate fails or any artifact is edited after a
gate passed, that gate AND all downstream gates MUST be re-run. A
stale PASS verdict is not a PASS — pre-flight READY on a WP that was
subsequently edited is no longer READY.

All three gates run **inside Session 1** (the drafting session),
against the just-written WP+EC.

#### Pre-flight (`01.4`)

Read the full procedure in `01.4-pre-flight-invocation.md`. The pre-flight
produces a binary verdict:

- **READY TO EXECUTE** — dependencies verified, scope locked,
  ambiguities resolved
- **NOT READY** — listing of blocking PS-items + clarifying RS-items

If NOT READY: fix the WP/EC in-place, re-run pre-flight. Iterate until
READY.

#### Copilot check (`01.7`)

Read `01.7-copilot-check.md`. Audits the WP + EC + pre-flight report
against 30 failure modes. Verdict:

- **PASS** — proceed to session-prompt authoring
- **RISK** — minor concerns; document and proceed
- **BLOCK** — substantive issue; HOLD (fix in-place) or SUSPEND
  (re-run pre-flight after rewrites)

If RISK or BLOCK: same in-place-fix loop as pre-flight.

#### 00.3 lint gate self-review

Run the **lint gate** (`00.3-prompt-lint-checklist.md`) against the WP.
All 21 sections must PASS or have an explicit N/A justification.
Document the result inline in the WP body under a `## Lint Gate
Self-Review` heading.

#### Hard invariant before Step 6 / Step 7

Pre-condition for both session-prompt authoring (Step 6) and the
drafting commit (Step 7):

- Pre-flight verdict MUST be `READY TO EXECUTE`
- Copilot check verdict MUST be `PASS` or `RISK` (`RISK` requires the
  concern documented inline in the WP)
- Lint gate self-review MUST be complete with all 21 sections
  resolved

If any of these conditions is not met: Step 6 and Step 7 are
**unauthorized**. Loop back to Step 5 and address the failure before
proceeding.

### Step 6: Write the session prompt

Create `docs/ai/invocations/session-wpNNN-{kebab-slug}.md`. This file
is **gitignored** under the patterns we established (`session-*.md`
under `docs/ai/invocations/`), so it lives as a scratchpad. Override
with `git add -f` only if an EC normatively cites it.

Format the prompt per the examples in `docs/ai/invocations/` (e.g.,
`session-wp005a-*.md`, `session-wp010-*.md`):

1. **Header** — WP ID, title, status reference
2. **Invocation intent** — one paragraph
3. **Authority chain (read order)** — CLAUDE.md → ARCHITECTURE.md →
   .claude/rules/ → WP → EC → reference docs → source files
4. **Pre-execution checks** — baseline assertions
5. **Execution rules** — locked values, forbidden patterns,
   `01.5`-runtime-wiring authorization (if applicable; cite
   `01.5-runtime-wiring-allowance.md` and note which files are
   explicitly authorized for wiring outside the allowlist)
6. **SAFE-KNOBS scope** — if the work touches a knob surface, cite
   `SAFE-KNOBS.md` and note which tier the change falls into
7. **Session task** — the work statement
8. **Post-merge close ritual (REQUIRED)** — explicit instructions
   for the executing session to run, after the operator merges the
   PR via the GitHub UI, mirroring `01.0b §Step 11.4–11.7`:
   - `node scripts/prune-empty-claude-branch.mjs --verify-current`
     from inside the worktree (expect `VERIFY PASS`; fail STOPs the
     ritual)
   - `git branch -D claude/<name>` (local) + `git push origin
     --delete claude/<name>` (remote)
   - `node scripts/prune-empty-claude-branch.mjs --report` from
     canonical (expect silent)

Hard limit: 200 lines unless justified in the WP's `§Context`. The
session prompt is not a substitute for the WP + EC; it's the
operator's brief to the next Claude session.

**Why §8 is REQUIRED** (rationale, post-WP-148/WP-150 retrospective):
session prompts that stop at "push + PR merge" hand cleanup back to
the operator, which is fine in principle but in practice creates a
gap — the operator may forget, may be in a different worktree, or
may hit cross-worktree state issues that block local-branch deletion.
Both WP-148 and WP-150's first execution sessions had this gap; the
operator finished the ritual manually from canonical in each case.
Including §8 in the prompt means the executing session drives the
full cycle to silent-state, with the operator's only manual step
being the GitHub-UI merge click.

**Session prompt scope restriction (REQUIRED):** the prompt MUST NOT
introduce any of the following beyond what the WP + EC already
encode:

- New scope (in or out)
- New files in the allowlist
- New contract elements
- New locked values
- New forbidden patterns

The prompt may **restate** or **operationalize** what the WP + EC
encode (e.g., translating "the EC requires X" into "you MUST do X
because the EC requires it"). It may NOT extend or contradict either.
If new scope or contract surface needs adding, it goes back into the
WP + EC, gates re-run from Step 5, then the prompt regenerates.

### Step 7: Commit + close drafting session

Single commit:

```
SPEC: draft WP-NNN + EC-NNN [+ D-NNNNN..D-MMMMM]
```

Commit format per `01.3-commit-hygiene-under-ec-mode.md`. The body
explains scope, lint-gate result, pre-flight verdict, copilot-check
verdict.

Then:

1. Push the draft branch.
2. Open a small PR. Title matches the commit title.
3. Merge to `main` (squash or rebase per repo policy).
4. **Verify the branch is safe to delete** (REQUIRED post-merge gate).
   From inside the draft worktree:

   ```pwsh
   node scripts/prune-empty-claude-branch.mjs --verify-current
   ```

   Expected: `VERIFY PASS: ...`. The check handles both true-merge
   (0 commits ahead) and squash-merge (merged PR exists on origin)
   cases. If it returns `VERIFY FAIL`, **STOP** — the failure message
   explains what's wrong (uncommitted changes, unmerged work, etc.).
   Fix before proceeding.

5. **Delete the draft branch.** This is the load-bearing step that
   keeps the hook quiet next session. Both local and remote:

   ```pwsh
   git branch -D claude/<draft-branch-name>
   git push origin --delete claude/<draft-branch-name>
   ```

6. Return to canonical, run `node scripts/prune-empty-claude-branch.mjs --report`. Expected: silent.

### Abandoning a draft

Drafting sessions are interrupted for many reasons — a higher-priority
WP lands first, the scope dissolves under inspection, the slug ships
via a different commit, the operator's attention shifts. The four
abandonment states below cover the cleanup ritual for each, in
ascending order of remote visibility. Verb-forward checklist is
intentional: the prior language elsewhere in this doc ("may discard")
was too passive to drive cleanup, and the empirical evidence (cited
at the bottom of this section) is that abandoned drafts accumulate
silently when the ritual isn't explicit per-state.

This section covers the **operator-driven** abandonment case. The
**gate-blocked** case (pre-flight surfaces a dependency blocker on a
draft that's structurally OK but premature) is covered by "Blocking
drafts" in `§Phase 1 Definition of Done` below.

**Detection note:** `node scripts/prune-empty-claude-branch.mjs
--audit` (added in PR #131, 2026-05-23) surfaces candidates for
abandonment automatically. The `abandoned-no-PR` heuristic finds
state (b) and state (c) cases; the `worktree-idle` heuristic finds
state (a) cases with dirty worktrees; the `stale-untracked-spec`
heuristic finds state (a) cases where the slug has already shipped.
Run it before assuming the workspace is clean. The per-state rituals
below remain the operator-driven path for **known** abandonments;
`--audit` is the discovery path for **unknown** ones. The audit is
informational only — it surfaces candidates, never deletes.

#### (a) Pre-commit abandonment

No commits yet; the worktree contains untracked `WP-*.md` and/or
`EC-*.md` drafts (and possibly an edited `WORK_INDEX.md` /
`EC_INDEX.md`).

Ritual:

1. Inspect uncommitted state: `git status`. Confirm there are no
   commits to preserve and no edits worth retaining elsewhere.
2. Discard untracked files: `git clean -fdn` to preview, then
   `git clean -fd` to apply.
3. Reset modified-but-not-committed governance files:
   `git checkout HEAD -- docs/ai/work-packets/WORK_INDEX.md
   docs/ai/execution-checklists/EC_INDEX.md` (or whichever index
   files were edited).
4. Exit the worktree; return to canonical.
5. Remove the worktree: `git worktree remove <path>` (use `--force`
   only if step 2/3 left residual dirt and you've confirmed nothing
   was missed).
6. Delete the auto-spawned branch: `git branch -D claude/<name>`.

No number-reservation cleanup is needed: reservations are only
"burnt" once an index row lands on `main` (per Step 2's **Number
reservation ownership** rule above). The reserved WP/EC numbers
return to the free pool automatically.

#### (b) Post-commit, pre-push

Local commits exist on the draft branch but were never pushed.

Ritual:

1. Confirm the branch has no remote tracking: `git branch -vv`. If
   the branch already pushed, this is state (c), not (b).
2. Exit the worktree; return to canonical.
3. Delete the local branch: `git branch -D claude/<name>` (the `-D`
   is required because the branch isn't merged anywhere).
4. Remove the worktree: `git worktree remove <path>`.

If the abandoned commits included index-row additions to
`WORK_INDEX.md` or `EC_INDEX.md`, no main-side cleanup is needed —
the rows never landed on `main`. The reserved numbers return to the
free pool.

#### (c) Pushed, no PR

The branch was pushed to `origin` but no PR was ever opened.

Ritual:

1. Delete the remote branch: `git push origin --delete claude/<name>`.
2. Exit the worktree; return to canonical.
3. `git branch -D claude/<name>`.
4. `git worktree remove <path>`.

Same number-reservation rule as (a) and (b): no index rows on `main`,
no cleanup required.

#### (d) PR opened but never merged

A PR was opened against `main` and is now being abandoned.

Ritual:

1. Close the PR with an explanatory comment: `gh pr close <number>
   --comment "Abandoned: <reason>"` (or the GitHub UI equivalent).
   The comment is **required** — a future operator reading the PR
   history needs the reason. Common reasons: "WP shipped via
   different commit on main", "EC slot reassigned mid-flight",
   "scope dissolved under audit".
2. Apply (c)'s ritual (delete remote, delete local, remove worktree).

#### If a SPEC commit already landed an index row to `main`

Any abandonment that occurs **after** the Phase 1 SPEC PR has already
merged (e.g., the WP+EC bundle is on `main`, but the WP was abandoned
before Phase 2 executed) requires a retraction commit. Per Step 2's
**Number reservation ownership** rule above:

> Abandoned drafts that already pushed an index row to `main` must
> be retracted with a follow-up SPEC commit removing the row before
> the number can be re-used.

The retraction is a fresh small SPEC PR removing the row from
`WORK_INDEX.md` + the row from `EC_INDEX.md` (and any reserved
`D-NNNNN` entries in `DECISIONS.md`) for the abandoned WP/EC.
Without this retraction the WP/EC number is "burnt" — the index row
sits on `main` but the implementation never landed, and the number
cannot be re-used for a different WP without the indices becoming
ambiguous to future readers.

#### Empirical precedent (2026-05-23)

A cleanup pass removed 9 stale `claude/*` worktrees+branches, none
of which fit the "forgot to delete after PR merge" failure mode that
`scripts/prune-empty-claude-branch.mjs` is designed to catch. The
pattern breakdown:

- **5 branches in state (b) — post-commit, pre-push:**
  `vigorous-turing`, `angry-merkle`, `romantic-mayer`, plus 2
  detached-HEAD orphans `silly-mahavira` / `vibrant-pike`. Each had
  1 commit ahead of `main`, no PR ever opened. Pattern: drafts whose
  intended WP shipped via a different commit on `main`, leaving the
  draft branch dangling. The hook's safety contract (0 commits ahead
  + no remote + clean worktree) refuses to prune branches with
  commits ahead, so these accumulated invisibly.
- **4 dirty worktrees in state (a) — pre-commit:**
  `confident-swartz`, `musing-aryabhata`, `blissful-blackburn`,
  `serene-pike`. Each had untracked `WP-*.md` / `EC-*.md` draft
  files but no commits. Pattern: drafts where the slug later shipped
  via different artifacts — e.g., an `EC-158` slot got reassigned
  mid-flight to a different WP; `confident-swartz`'s draft `WP-105`
  body was the pre-execution version that diverged from what
  actually shipped. The hook's safety contract refuses to prune
  dirty worktrees, so these accumulated invisibly.

The explicit per-state ritual above closes the gap between the
hook's prune contract and the actual failure modes operators
produce.

### Phase 1 Definition of Done

Phase 1 is complete when ALL of the following are true:

- [ ] WP file + EC file merged to `main` via a SPEC PR
- [ ] WORK_INDEX.md and EC_INDEX.md rows added in the same SPEC commit
- [ ] Pre-flight verdict on record: `READY TO EXECUTE`
- [ ] Copilot check verdict on record: `PASS` or `RISK` (with concern documented)
- [ ] Lint gate self-review section present in the WP body
- [ ] Session prompt written at `docs/ai/invocations/session-wpNNN-*.md`
- [ ] Draft branch deleted (local + remote)
- [ ] `scripts/prune-empty-claude-branch.mjs --report` reports silent from canonical

If any item is unsatisfied, Phase 1 is NOT done. Do not proceed to
Phase 2 with partial Phase 1 completion.

**Blocking drafts:** if pre-flight surfaces a dependency blocker
(`NOT READY` because hard-dep WP is in-flight), do not merge partial
WP/EC artifacts. Choose exactly one:

- **Complete Step 4–5 fully and merge** if you want the WP to exist
  as a `[ ]` placeholder row with `BLOCKED on WP-NNN` status. This
  is appropriate when you want to reserve the WP/EC numbers and
  signal the dependency.
- **Discard the draft branch entirely** if you'd rather re-draft when
  the dep clears. This returns the WP/EC numbers to the free pool.

Half-merged states (WP file on `main` without EC file, or index row
without WP file) are forbidden.

## STOP — End of 01.0a (drafting phase)

If Phase 1 DoD is satisfied, the WP+EC bundle is on `origin/main`
and the draft branch has been cleaned up. Phase 1 is complete.

**Recommended pause point:** review the just-merged WP and EC files
on `main`. Add any operator edits as a follow-up SPEC commit (small
PR, same INFRA discipline). Do not start Phase 2 until you're
satisfied with the WP+EC content.

**When ready to execute the WP:** proceed to
`01.0b-wp-execution-phase.md` for Phase 2 (Steps 8–11), parallel-mode
rules, the consolidated quick reference card, and the deviations/
anti-patterns catalogue.

**If you decide to abandon the WP:** retract its WORK_INDEX/EC_INDEX
rows with a SPEC commit (see "Blocking drafts" above) and reclaim
the reserved numbers for future use. Do not silently leave the
unimplemented WP rows on `main`.
