# 01.2 — Bug Handling Protocol Under Execution Checklist (EC) Mode

> **REFERENCE DOCUMENT — Not an execution prompt. No output required.**
>
> **Subordination (descending authority):**
> 1. `.claude/CLAUDE.md` (Execution Checklists section)
> 2. `docs/ai/ARCHITECTURE.md`
> 3. `docs/ai/execution-checklists/EC-TEMPLATE.md` (EC structure authority)
> 4. `docs/ai/REFERENCE/01.1-how-to-use-ecs-while-coding.md` (EC workflow)
>
> If any conflict exists → higher authority wins immediately.

---

## Purpose

Once Execution Checklists are active, every observed bug is treated
as an **execution contract violation**. This protocol replaces
ad-hoc debugging and enforces:

- Deterministic behavior
- Specification traceability for every fix
- Zero drift between implementation and the WP / EC / ARCHITECTURE
  contract
- A defensible record across Work Packets, ARCHITECTURE.md
  invariants, and locked contracts

---

## Core Invariant (Non-Negotiable)

Every observed bug MUST be classified into **exactly one** of two
categories:

1. **Implementation violation** — code fails to satisfy a valid EC
   clause.
2. **Specification error** — the EC, source WP, or ARCHITECTURE.md
   contains an inconsistency.

This is a **closed classification system**. The following are
**INVALID diagnoses**:

- "Unexpected behavior"
- "Edge case"
- "Worked before"
- "Seems incorrect"
- "Just a small issue"

If a bug cannot be classified into one of the two categories →
**it is not yet understood**. Do not fix it. Improve the EC / WP
language first (Step 3 below).

---

## Scope

This protocol applies to all packages under EC-driven execution:

- `packages/game-engine`
- `packages/registry`
- `apps/server`
- Any future package that has a corresponding EC

It activates the moment a Work Packet's Execution Checklist exists
in `docs/ai/execution-checklists/EC_INDEX.md` with status `Draft`
or `Done`.

---

## Bug Intake Rule (Mandatory Gate)

When any anomaly is observed (test failure, runtime error,
incorrect game state, desync, etc.):

1. Open `EC_INDEX.md` and locate the governing Execution Checklist.
2. Open the EC file in `docs/ai/execution-checklists/`.

**Before doing anything else:**

- ❌ Do NOT open code.
- ❌ Do NOT add logs, console statements, or debug helpers.
- ❌ Do NOT run exploratory debugging.
- ❌ Do NOT attempt any code change.
- ❌ Do NOT modify the EC to fit observed behavior.

**Invariant:**

> All investigations begin from specification, not implementation.

Violation of this rule = **protocol breach**. The cost of opening
code first is that you start fitting the spec to the code instead
of the other way around.

---

## Diagnosis Sequence (Strict Order)

Follow these steps exactly. Stop at the first step that classifies
the bug.

### Step 1 — EC Clause Violation Check (Default Path)

- Identify the exact EC clause that governs the failing behavior
  (Locked Value, Guardrail, Required Comment, File list, After
  Completing).
- Evaluate the violation against the **literal text** of the EC.
- If the EC has a **Common Failure Smells** section, check
  symptoms there.

If a clause is violated → **Classification: Implementation
violation**. Proceed to "Implementation Fix" below.

### Step 2 — Specification Consistency Check

Only if Step 1 yields no violation. Re-read, in this exact order:

1. The EC itself
2. The source Work Packet (WP)
3. Relevant sections of `docs/ai/ARCHITECTURE.md`
4. Any linked decision in `docs/ai/DECISIONS.md`

Ask:

- Does the EC contradict its source WP?
- Does the WP contradict an ARCHITECTURE.md invariant?
- Is a locked value in the EC different from the WP?
- Is the EC language ambiguous enough that two readers could
  interpret it differently? (Ambiguity is a defect, not a
  feature.)

If an inconsistency is confirmed → **Classification:
Specification error**. Stop coding immediately and follow
"Specification Fix" below.

### Step 3 — Specification Insufficiency (Escalation)

If neither violation nor inconsistency can be proven after Steps
1 and 2, the bug is **not the system's fault — it's the spec's**.
The specification is insufficiently precise to classify the
behavior.

Rules:

- Do NOT fix code. The code can't be wrong against a spec that
  doesn't constrain it.
- Do NOT guess behavior.
- Improve the EC / WP language **FIRST**.
- Only after clarification → restart from Step 1.

This is a real failure mode: when the spec is insufficient, fixing
code reproduces the same insufficient-spec bug on the next
regression. The cure is to make the spec precise enough that the
bug is mechanically classifiable.

---

## Permitted Fix Types

### Implementation Fix (Default Path)

**Condition:** A valid EC clause is violated by the current
implementation.

Rules:

- Modify code only.
- NEVER weaken or reinterpret EC language to fit the current
  implementation.
- NEVER edit the EC file during an implementation fix (drift
  prevention).
- The fix MUST trace to a specific clause (Traceability
  Requirement, below).
- Re-run the full EC checklist after fixing.

### Specification Fix (Controlled Path)

**Condition:** The EC cannot be satisfied without breaking the
source WP or ARCHITECTURE.md.

Rules:

- STOP all coding immediately.
- Update the WP, EC, or ARCHITECTURE.md first.
- Record the change in `docs/ai/DECISIONS.md` with a new D-XXXX
  entry.
- Update `docs/ai/DECISIONS_INDEX.md` with the new decision
  reference.
- Re-tighten affected ECs using the auto-tighten prompt.
- Re-run the full EC checklist against the updated spec.

---

## Traceability Requirement (Mandatory)

Every fix MUST be mechanically traceable to one of:

- A specific EC clause (Implementation fix), cited in the commit
  body — e.g., "Fixes EC-010 Locked Value 'endIf returns evaluator
  result'; observed early return on first non-end card."
- A `docs/ai/DECISIONS.md` D-XXXX entry (Specification fix),
  cited by number — e.g., "Per D-14401 runtime / setup-tooling
  Layer Boundary split."

If traceability cannot be established:

→ The fix is **invalid**. Stop and re-classify per Step 1 / Step 2.

This requirement is the audit hook: `git log --grep` over EC /
SPEC prefixes, combined with the cited clause / decision, gives
any future reader a complete trace from observed bug → governance
clause → resolution. Without traceability the log degrades into
folklore.

---

## Prohibited Responses (Zero Tolerance)

- Adding logs or debug code to "discover" behavior.
- Writing code before locating the EC clause.
- Hot-patching or "quick fixes" that pass tests but violate EC
  intent.
- Editing an EC to accommodate existing code.
- Fixing symptoms instead of the violated clause.
- Relying on tribal knowledge ("we know what it should do").
- Bypassing `EC_INDEX.md` lookup ("I know which EC this is").
- Guessing at the root cause without tracing to an EC clause.
- Speculating without clause-level validation.
- Using `--no-verify` to bypass commit hooks (see
  `01.3-commit-hygiene-under-ec-mode.md` § Bypassing Hooks).

## Diagnosis Workflow

```
Bug observed
  │
  ▼
Open EC_INDEX.md → locate governing EC
  │
  ▼
Step 1: EC Clause Violation Check
  │
  ├─ Clause violated → Implementation Fix → re-run EC checklist → done
  │
  ▼
Step 2: Specification Consistency Check
  │
  ├─ Inconsistency found → Specification Fix → re-tighten EC → done
  │
  ▼
Step 3: Specification Insufficiency
  │
  └─ Tighten EC / WP language → restart from Step 1
```

---

## Regression Rule

When a previously resolved bug reappears:

→ This is **proof of insufficient specification precision** in
the governing EC.

Required action (in order):

1. Identify the EC clause that should have prevented the
   regression. If no clause covered it, the EC needs a new
   Guardrail, Common Failure Smell, or After Completing check.
2. Strengthen the EC language:
   - Add a tighter guardrail.
   - Add a new failure smell that names the regression shape.
   - Add a more precise After Completing check that mechanically
     detects it.
3. Record the EC amendment in `docs/ai/DECISIONS.md` with a new
   D-XXXX entry citing the regression that drove the tightening.
4. Update `docs/ai/DECISIONS_INDEX.md`.
5. Then apply the normal Implementation Fix procedure against the
   strengthened spec.

**Invariant:**

> Regressions improve specifications, not just code.

A regression that produces only a code fix is a missed opportunity
— the next operator will hit the same bug because the spec didn't
change.

---

## Completion Rule

A bug is resolved ONLY when **all** of the following are true:

- [ ] Every clause of the governing EC is satisfied (re-run the
      full EC checklist).
- [ ] All After Completing checks pass.
- [ ] No other EC is incidentally violated (regression check
      against adjacent surfaces).
- [ ] The change is traceable in git to the EC clause (or to the
      specification update / D-XXXX entry).
- [ ] If a regression: EC language is strengthened per the
      Regression Rule.

**Passing tests alone is insufficient.** The mechanical
re-verification against the EC contract is what closes the bug.

---

## Enforcement Summary

| Rule | Effect |
|---|---|
| No EC lookup → no coding | Guarantees spec-first debugging |
| Two-category classification | Eliminates ambiguous diagnoses |
| Strict diagnosis sequence | Prevents premature fixes |
| Traceability required | Enables `git log` audit + determinism |
| Regression strengthens EC | Hardens specification over time |
| Completion = mechanical recheck | Closes bugs against the contract |
