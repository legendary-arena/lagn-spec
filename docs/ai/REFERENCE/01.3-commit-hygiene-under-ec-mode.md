## Purpose

Under EC mode, every commit must be mechanically traceable — via its
commit subject line and enforced hooks — to exactly one of:

- an Execution Checklist (EC-###),
- a specification correction, or
- an infrastructure change.

This document defines the commit message contract, the enforcement hooks,
and the installation and recovery procedures that ensure this traceability.

---

## Installation

Run once after cloning (or after hooks are updated):

```pwsh
pwsh scripts/git/install-ec-hooks.ps1
```

This sets `git config core.hooksPath .githooks` for the local repo.

## Commit Message Format (Enforced by Hooks)

Every commit subject line must match one of these three prefixes.

> **Note:** Work Packets (WP-XXX) are not used directly in commit prefixes.
> Execution is always attributed to the governing Execution Checklist (EC-###),
> which in turn references one or more Work Packets.

### EC-scoped execution commits (code changes)

```
EC-###: <present-tense summary>
```

Examples:
- `EC-010: wire endIf to evaluateEndgame`
- `EC-018: gate fight and recruit on turnEconomy`
- `EC-005B: implement shuffleDeck with ctx.random.Shuffle`

**Required** for feature / gameplay changes under `packages/` or `apps/`.
Small infra / hygiene fixes that touch those paths but fall outside any Work
Packet may use `INFRA:` instead (per D-20801).

### Specification corrections (rare)

```
SPEC: <summary>
```

Examples:
- `SPEC: clarify registry boundary for economy stats`
- `SPEC: fix evaluation order in EC-010 locked values`

Use when updating WPs, ECs, ARCHITECTURE.md, or DECISIONS.md to fix an
inconsistency. Not for code changes.

### Infrastructure (non-gameplay)

```
INFRA: <summary>
```

Examples:
- `INFRA: add EC commit hooks`
- `INFRA: update render.yaml startCommand`
- `INFRA: rebind analytics mocks so dev-mode ESM eval resolves`

Use for tooling, scripts, CI, hooks, and repo configuration — and for small
infra / hygiene fixes under `packages/` or `apps/` that fall outside any Work
Packet (dev-mode breakage, build glue, mock wiring). Per D-20801, `INFRA:` is
accepted by the code-path gate; `SPEC:` is not. INFRA is for infra / hygiene,
**not** a route to land feature work without an EC — reviewers enforce that
boundary at PR review, since the hook cannot tell the two apart.

## Vision Trailer (Convention — Not Hook-Enforced)

When the EC's governing Work Packet contains a `## Vision Alignment`
section (per `docs/ai/REFERENCE/00.3-prompt-lint-checklist.md §17`),
the commit message body **should** include a `Vision:` trailer line
citing the same clause numbers. This makes the link from commit → vision
clauses traceable in `git log` without re-reading the WP.

The trailer is a **documentation convention**, not a hook-enforced rule.
Commits without the trailer are not rejected, but reviewers may request
it for any commit whose WP has a `## Vision Alignment` block.

### Format

```
EC-###: <present-tense summary>

<optional body paragraphs>

Vision: §<n>[, §<n>...][, NG-<n>...]
```

### Examples

```
EC-053: enforce server-side score recomputation on submission

Replay re-execution via replayGame; client-reported scores ignored.
PAR publication checked via checkParPublished; missing PAR fails closed.

Vision: §3, §22, §24, NG-1
```

```
EC-054: expose read-only public leaderboard projections

Private replays excluded. PAR-missing scenarios return empty results.
No new tables — query projections only.

Vision: §23, §25, NG-1, NG-3
```

### When to omit

- `INFRA:` commits touching only build tooling, CI, or hooks
- `SPEC:` commits that update prose unrelated to vision-bearing surfaces
- `EC-###:` commits whose governing WP has no `## Vision Alignment`
  section (e.g., structural / harness-only WPs)

### Future enforcement

If commit-msg hook enforcement of this trailer is later adopted, it must
be added behind a feature flag and announced in `DECISIONS.md` first.
Until then, this section is convention only.

## Forbidden Patterns (Rejected by Hooks)

The following words in commit subjects cause automatic rejection:

`WIP`, `wip`, `fix stuff`, `misc`, `tmp`, `updates`, `changes`, `debug`,
`quick fix`, `hotfix`, `cleanup`, `refactor`

Subject must be at least 12 characters after the prefix.

---

## Hook Architecture

| Hook | Fires when | Enforces |
|---|---|---|
| `pre-commit` | Before message is written | No secrets, no node_modules, no dist/, no .test.mjs |
| `commit-msg` | After message is written | Format, forbidden words, EC reference validation, code-path cross-check |

### What commit-msg validates for EC-### commits:
1. EC file exists: `docs/ai/execution-checklists/EC-###-*.checklist.md`
2. EC number appears in `EC_INDEX.md` (warning if missing)
3. If code files are staged, prefix must be `EC-###:` (feature work) or
   `INFRA:` (infra / hygiene, per D-20801) — `SPEC:` is rejected for code

## Using the Helper Script

```pwsh
# Interactive mode — prompts for message
pwsh scripts/git/ec-commit.ps1

# Direct mode
pwsh scripts/git/ec-commit.ps1 -Message "EC-010: wire endIf to evaluateEndgame"

# Stage all modified files + commit
pwsh scripts/git/ec-commit.ps1 -All -Message "INFRA: add EC commit hooks"

# Stage specific files + commit
pwsh scripts/git/ec-commit.ps1 -Files "src/file1.ts","src/file2.ts" -Message "EC-018: add economy types"
```

The helper verifies hooks are installed before committing.

### Dry-Run / Check Mode

To verify that a commit would pass EC hygiene without creating it:

```pwsh
pwsh scripts/git/ec-commit.ps1 -Check -Message "EC-018: <summary>"
```

This runs all EC commit rules but does not create a commit.

## Bypassing Hooks

`git commit --no-verify` bypasses all hooks. This is **prohibited under
EC mode** unless explicitly approved by a human. Claude must never use
`--no-verify` or `--no-gpg-sign` unless the user requests it.

### Emergency Override (Last Resort)

In case of repository corruption or hook malfunction, hooks may be bypassed
ONLY by temporarily renaming `.githooks/`:

```pwsh
Rename-Item .githooks .githooks-disabled
# ... perform recovery commit ...
Rename-Item .githooks-disabled .githooks
```

This must be immediately reverted after recovery.
Commits made this way must be annotated and reviewed.

## Common Errors and Fixes

| Error | Cause | Fix |
|---|---|---|
| "Subject does not match an allowed prefix" | Missing EC-###:, SPEC:, or INFRA: | Add the correct prefix |
| "Subject too short after prefix" | Description < 12 chars | Write a more specific summary |
| "Subject contains a forbidden pattern" | WIP, misc, tmp, etc. | Use a concrete description |
| "No EC file found for EC-###" | EC checklist doesn't exist | Create the EC first, or use INFRA: |
| "Code changes detected but commit is neither EC-scoped nor INFRA" | Code files staged with `SPEC:` | Use `EC-###:` (feature) or `INFRA:` (infra / hygiene) |
| "Staged .env file detected" | .env in staging | `git reset HEAD .env` |
| "Staged .test.mjs file detected" | Wrong test extension | Rename to .test.ts |

## CI Mirror

`.github/workflows/commit-hygiene.yml` runs on every PR to `main` and
re-validates all commits in the PR. This catches commits that bypassed
local hooks (missing install, `--no-verify`, or external contributors).

The CI mirror is authoritative and must be treated as a second line of
defense, not a substitute for local hooks.

Three jobs run in parallel:
- **commit-messages** — runs `.githooks/commit-msg` against every PR commit
- **file-hygiene** — checks PR diff for forbidden file patterns
- **ec-code-traceability** — verifies code-changing commits have EC-### prefix
  with an existing EC file
